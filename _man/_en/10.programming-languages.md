---

Programming languages exist to ask the computer to carry out certain operations.
For the computer to obey us, it must "understand" what kind of entities it has to manage and what kind of operations it must perform on them.
For example, if we asked the computer to double our salary, the computer would need to know at least how much our salary is and how multiplying by two works.

```
long double_it(long salary)
{
    return salary * 2;
}
```

Since the only thing the computer knows are sequences of 1s and 0s — whether on a disk, in memory, or on the communication port of the keyboard or the screen — we have to find a way to "explain" the concepts of *salary* and *multiplication* to it in binary form.
Programming languages do exactly this: they translate our requests into a language the computer can understand.

---

Programming languages can be *high-level* or *low-level*.
High-level languages, like *C* or *C++*, have constructs similar to natural language and are not tied to any specific hardware architecture.
Low-level languages, like *assembly*, have constructs similar to the machine's native code and are, for this reason, tied to a specific type of hardware.
The *double_it* function we saw earlier, translated into *assembly* language, becomes:

```
double_it(long):
        push    rbp
        mov     rbp, rsp
        mov     QWORD PTR [rbp-8], rdi
        mov     rax, QWORD PTR [rbp-8]
        add     rax, rax
        pop     rbp
        ret
```

Programs written in high-level languages, as you can see, are easier to write and to fix, but slower to run, because the code has to be "translated" into a form the CPU can understand.
Programs written in low-level languages are more complex to write or fix, but much faster to run, because they contain only the instructions strictly necessary to carry out the desired task.

A computer is like a ship.
The hardware is down below, like the engine room, and the CPU is the chief engineer: he doesn't see the sea, doesn't see the sky, doesn't even know whether the ship is heading North or South; he receives his instructions over the intercom and only cares about speed and fuel consumption.
The operating system is the crew: it's in an intermediate position, neither at the bottom nor at the top, and it looks after the ship and the cargo; it can talk to the Captain and to the chief engineer, but it never talks to the shipowners.
The user interface, on the other hand, is the bridge: it's the highest point of the ship, from which everything can be seen; it's in direct contact with the shipowners, from whom it receives directives on what it should or shouldn't do, and with the crew, who inform it of the ship's status.
Each level of the computer has its own language.
This computer's CPU "thinks" in what is called *machine language*, that is, a long series of decimal values indicating both the operations to perform and the quantities involved.
The operating system is written in C, because it's the language best suited to managing a computer.
The user interface, finally, makes use of object-oriented languages like C++ to create the graphical elements that let it interact with the user.
The languages and constructs that apply at one level don't work if applied at the other levels, because each level conceives of and manages the elements of a program differently.
The user interface knows many things about the programmer: it sees his face in the webcam, hears his voice in the microphone, sometimes reads his fingerprints on the scanner.
The operating system can't see or hear him, but it knows his personal details and knows everything he writes to his friends.
The CPU knows nothing about him: it processes his data, but doesn't know what it is.
It can process the sequence of bytes:

```

01100001
01101110
01100001
01110010
01101111
```

but it doesn't know that it's the name: *Canaro*.
In fact, it doesn't even know what the word *name* is.
To it, it's just the sequence of bytes:


---

We perceive the real world as a set of events involving one or more entities.
Each entity can be identified by its physical characteristics and has its own particular way of reacting to external stimuli.
To describe entities, programming languages have
<a href="/en/tipi-di-dato" class="xref">data types</a>;
to describe events,
<a href="/en/operatori" class="xref">instructions and operators</a>.
A programming language is more effective the better its data types and syntactic constructs manage to represent the entities the program will have to handle.
There are many programming languages.
Each language is good at explaining certain things well, and less good at others.
There are languages that describe banking transactions well but are unsuited to managing air traffic; others that work very well for creating Web pages, but with which it would be masochistic to write an operating system.
A programmer who deeply knows a certain programming language can use it effectively to write any program, but at what cost?
If the programming language is unsuited to describing the events the program will need to deal with, the programmer will have to worry about bending either the events or the language, or both, to get the expected result.
Choosing instead a language whose data types and constructs are closer to the reality being described saves time and effort and produces more reliable code.
Programming languages can be of two types: *interpreted* or *compiled*.
I can explain the difference this way: imagine you're in China and have to get back to your hotel by taxi.
If you don't speak Chinese, you have two options: either bring along an interpreter to explain to the driver where you need to go, or have the hotel give you a piece of paper with the address and show it to the taxi driver.
Both options have upsides and downsides.
If you choose to bring an interpreter you'll have extra costs, but you'll be free to go wherever you want: to the hotel, to a restaurant, or to a nightclub.
If instead you opt for the piece of paper with the address, you'll save the interpreter's fee, but once you're in the taxi, you'll only be able to go back to the hotel.
The same thing happens with programming languages.
The code of interpreted languages is read by a program called an *interpreter*, which first translates the instructions into the computer's language and then executes them.
The code of compiled languages, on the other hand, is read by a program called a *compiler*, which converts it into instructions the computer can understand, writes these instructions into a new file called an *object file*, and passes it to another program called a *linker*, which turns it into an *executable* file.
Here too, each method has upsides and downsides.
Interpreted languages are slower, because the read/convert/execute step is repeated every time the code is processed; they also require extra system resources dedicated to the interpreter.
On the other hand, every change to the source code is immediately reflected in the running program.
Compiled languages are faster and require fewer system resources, because compilation happens only once, after which the program can run on its own; but if the source code is modified, the program has to be recompiled to work correctly.

All languages have a syntax, all languages have data types; very few languages also have principles.
Principles are important, because they give a language robustness.
The C language, from which C++ is derived, has no principles: it is only a way of describing events inside a computer.
The instruction:

```
char b = 23;
```

is a way of telling the computer:

1. find a free area of memory 1 byte wide;
1. assign the address of that area to the variable *b*;
1. put the value *23* into that area of memory.

I can see you're puzzled, but think about it: multiplying by two, in binary, is the equivalent of multiplying by ten in decimal notation.
If you have to multiply a decimal number by the base of the numeral system — which is ten, precisely —, you shift it one place to the left and add a zero on the right:

```
1234 * 10 = 12340
```

If you have to multiply a binary value by the base of the system, which in this case is two, you do the same thing:

```
00010111 * 00000010 = 00101110
```

The value `00101110` equals 46, which is double 23.

C is a language that doesn't go beyond the scope it was created for.
It's a convention.
If tomorrow we stopped using computers, C would no longer exist, because all of its constructs are tied to the workings of computing hardware.
By contrast, C++ and all languages based on the object-oriented paradigm have a philosophical component that transcends the field of computing.
If tomorrow we stopped using computers, the constructs C++ inherited from C would no longer make sense, but the concepts of
<a href="/en/classi-oggetti" class="xref">class and object</a>
would still remain valid.
Many think that having principles is limiting, but that's not true.
C has no principles, and its constructs are limited to the data types it provides for.
C++, thanks to its principles, has greater flexibility and can evolve, adapting its constructs to any context.
