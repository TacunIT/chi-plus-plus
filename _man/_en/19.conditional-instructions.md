Conditional instructions are the most important element of code.

Every program has to know how to react correctly as the conditions it's used under change; to do this, we use so-called *conditional instructions*, which let you define how the system behaves depending on whether a given condition turns out to be true or false.
C++ has two kinds of conditional instruction: `if-else` sequences and the `switch` instruction.
The general form of `if-else` instructions is as follows:

```
if ( <condition> ) {
    // instructions to run if the condition is true
} else {
    // instructions to run if the condition is false
}
```

If the conditional expression at the start of the code is true, the program will run the first block of instructions; if not, it will run the second block of instructions.

```
if ( a > 8 ) {
    cout << "greater" << endl;
} else {
    cout << "less" << endl;
}
```

If the false condition requires no specific action, the second block of instructions can be omitted:

```
typedef Amount unsigned long;

Amount doubleSalary(Amount currentSalary)
{
    if ( currentSalary > 0 ) {
        currentSalary *= 2;
    }    
    return currentSalary;   
}
```

In the same way, curly braces can be omitted if the block of instructions they enclose consists of a single instruction:

```
if ( a > 8 )
    cout << "greater" << endl;
else
    cout << "less" << endl;
```

Personally, I find this form inelegant, and think it makes the code less clear, which invites mistakes.
I only use it in error-handling instructions, where the program's flow breaks off abruptly, because the ungainly look of the instruction highlights the exception, making the code more self-explanatory.

```
if ( divisor == 0 )
    throw std::runtime_error("error");
```

If there are more than two conditions to evaluate, you can chain several conditional instructions using the `else if` instruction, which lets you define an alternative condition to the first one and associate a block of code with it.
Here too, you can close the sequence with an `else` instruction, defining a block of instructions to run if none of the expected conditions is met.

```
if ( <first condition> ) {
    /*
     * instructions to run if
     * the first condition is true
    */
} else if ( <second condition> ) {
    /*
     * instructions to run if
     * the first condition is true
     */
} else {
    /*
     * instructions to run if neither
     * of the two conditions is true
     */
}
```

`if-else` instructions affect the readability of the code; it's something a good programmer must always keep in mind.
C++ is a formatting-independent language, so the same instruction can be written in many different ways:

```
if ( <condition> )
{
    ...
}
else
{
    ...
}
```

or, equally:

```
if ( <condition> ) {
    ...
} else {
    ...
}
```

or even:

```
if ( <condition> ) { ...} else { ... }
```

If the instructions are few and simple, one form is as good as another (matters of style aside, of course), but if the program's flow is, as it usually is, more complex, you need to make sure the form of the instruction makes the code easier to write, read, and, if needed, fix.
Imagine a piece of code that has to run a check at the start of processing and, depending on the outcome, either run a sequence of instructions or send an error message:

```
if ( <condition> ){
    /*
     * lines of code to
     * run if the
     * condition is true
     */
} else {
    /* error handling */
}
```

If the instructions to run when the check succeeds are few and simple, this sequence won't cause any problems, but if instead the instructions were numerous and complex, while reading the code you might reach the `else` instruction and no longer remember which condition it was tied to.
In these cases, I prefer the form:

```
if ( <error> ) {
    /* error handling */
} else {
    /*
     * lines of code to
     * run if the
     * condition is true
     */
}
```

Since error handling will never require more than a few lines of code, you'll be able to grasp the whole flow of the program at a glance, regardless of how long the second block of instructions is.
None of this, of course, is meant as either an invitation or a justification for writing complex instructions.
Unless you need to limit function calls to guarantee high execution speed, it's always better to break the program's flow down into a series of distinct, specialized functions.
You'll make your program a bit slower (or, rather: a bit less fast), but the code will be much easier to read or modify.

Now imagine a piece of code that requires many chained `if` conditions:

```
outcome = 0;

if ( <condition 1> ) {
    outcome = 1;
} else if ( <condition 2> ) {
    outcome = 2;
} else if ( <condition 3> ) {
    outcome = 3;
} else {
    outcome = 9;
}

return outcome
```

This form, however correct and formally impeccable, could turn out to be hard to manage if the conditions to consider were very complex or numerous.
The good programmer, in that case, may decide to break the (correct) rule that says not to put `return` instructions inside the code, and write the sequence like this:

```
outcome = 0;

if ( <condition 1> ) {
    return 1;
}
if ( <condition 2> ) {
    return 2;
}
if ( <condition 3> ) {
    return 3;
}

return 9
```

I'm not telling you this is the right way to write it, and you can see for yourself that the code is crude and inelegant, but there could be cases where this is the preferable form.
For example, for a system that generates code automatically, it's much simpler to handle isolated `if` instructions than chained `if-else` conditions.
Think of a *stored procedure* that has to check the referential integrity of the parameters it receives:

```

```

If you write the code this way, you can add or remove a parameter (and its related checks) without altering the rest of the code, which wouldn't be the case if you chained the `if` instructions together.
You'll lose a bit of execution speed, but the code will be much easier to write or modify.
Be careful, though: putting simple `if` instructions in sequence is quite different from building a chain of `else-if` instructions, because, if you don't halt processing with a `return` instruction in case of an error, the program will keep going, checking the following conditions, and the error in the first condition could have knock-on effects on the code that follows:

```
/** Here's where trouble begins.. **/
if ( divisor == 0 ) {
    cout << "Error: division by zero" << endl;
}

/** ..and worse follows **/
if ( (dividend / divisor) > 1 ) {
    ...
}
```

Since there's no `return` instruction, the code of the first check won't stop the function from running, and it will fail when it tries to perform a division by zero.

The `switch` instruction lets you handle multiple cases, based on the evaluation of an expression:

```
switch(<expression>)
{
    case <constant> : instructions... [break];
    case <constant> : instructions... [break];
    ...
    default: instructions...
}
```

The `case` and `default` keywords identify the values handled by the `switch` instruction.
There can (and it's useful for there to) be more than one `case`, but the constants associated with each of them must have different values.
The `default` condition, by contrast, must be unique.
Execution of the instruction begins at the `case` whose constant matches the value of the `switch` expression, and ends at the `break` keyword.
If the expression has a value not covered by any `case`, the instruction runs the code associated with the `default` label:

Compiling and running this code, you'll get:

```
% g++ src/cpp/conditional-instructions-switch.cpp -o src/out/example

Enter a value from: 1 to 9
% src/out/example 4
Mars
% src/out/example earth
Invalid value
```

In essence: with a `break` at the end of each case, the `switch` instruction is a more elegant (and efficient) form of the `if - else if` instruction:

```
if(planet == POS_ERROR) {
   cout << "Invalid value";
} else if(planet == POS_MERCURY) {
    cout << "Mercury";
} else if(planet == POS_VENUS) {
    cout << "Venus";   
} else if ..         
```

If you removed the `break`s at the end of each case, the program's output would be:

```
% src/out/example 4    
MarsJupiterSaturnUranusNeptunePlutoEnter a value from: 1 to 9
```

which makes no sense in this particular case, but which can be the right choice when two possible cases need to be handled the same way.
One last thing: always remember that, to declare variables inside a `case`, you need to add curly braces; otherwise, you'll get a compile error:

```
switch( x ) {
    case 1:
        int y = 9;      /** compile error */
        cout << x + y;
        break;
    case 2: {
        int y = 2;      /** correct */
        cout << x + y ;
        break;
    }
    default:
        cout << "default" << endl;
}

```

Answering the question about the convicted man is harder.
The purpose of your actions must always be improvement: whoever makes a mistake must understand that they've made an error, and not repeat it in other cycles of existence:

```
int repentance( action ){

    bool repeat;

    if ( action == mistake ) {
        repeat = false;
    } else {
        repeat = true;
    }

    return repeat;
}
```
If the sinner truly repents, then it's right that they be absolved, because, as Attar says:

> A hundred Worlds of sin are dissolved by the light of a single repentance.<a class="nota" href="/en/notes#fn-pentimento" id="pentimento">7</a>

But the repentance must be real: the sinner must despise their mistake, and choose to die rather than repeat it.
Cutting off the hand of a thief, forcing them to bring food to their mouth with the same hand they use to clean themselves, is a rather drastic method, but an effective one for forcing someone to meditate on the senselessness of their past actions &mdash; especially in a place like the desert, where bidets are rarer than in France.
The *Lex Talionis* can work for lesser crimes, because whoever suffers it has time to reflect on their mistakes, but in the case of a murder it's not only contrary to the duty of benevolence we owe other sentient beings, it could also be counterproductive, because if the condemned doesn't understand their mistake before dying, their criminal actions might be repeated in other cycles of the Universe.
On the other hand, we also have a duty of benevolence toward the other inmates and prison guards, so we cannot allow the condemned to kill them.
The ideal solution would be to put them in a position where they can't harm others, and then let them meditate on their mistakes, but if that isn't possible, how should we behave?
If an individual acknowledges their own mistakes and learns from them, they contribute to the improvement of the Universe; if instead they repeat the same criminal act over and over, they clearly don't understand, or don't want to understand, their mistake.
If they don't understand, they aren't *sentient*, and therefore cannot contribute to the improvement of the Universe.
If they're of no use to that improvement, we can treat them the same way as Nansen's cat:

> Nansen found the monks of the eastern and western halls arguing over a cat. He seized the cat and said: "If you can say a word of Zen, I will spare the cat; if not, I will kill it."
No one could answer, and Nansen cut the cat in two.<a class="nota" href="/en/notes#fn-nansen" id="nansen" >8</a>

For a long time, Master Canaro wondered why Nansen was able to cut the innocent cat in two.
The explanation he settled on was that the cat, not being a sentient being, and therefore incapable of contributing its own experience to the growth of the Universe, was expendable, if that could somehow lead the monks to Enlightenment.
Be careful, though: the death of the condemned, should it ever prove necessary, must not be seen as a revenge to be celebrated, but as an event as painful as it is unavoidable, to be mourned like an amputation.
Everyone, on that day, should ask themselves whether, through *thoughts, words, deeds, and omissions*, they too contributed in some way to that loss.
A Society that glorifies individualism, success, and money cannot call itself entirely innocent if someone who lacks the means or the ability to obtain them lawfully tries to get them some other way.

A Buddhist teacher, having heard that Master Canaro, in his writings, claimed there was no way to escape the cycle of rebirth, went to see him and, determined to prove him wrong, challenged him to a *Dharma Combat* to settle their respective positions.
Master Canaro replied that he didn't know what a *Dharma Combat* was; at which the Buddhist teacher explained that it was a dialectical exchange, meant to demonstrate one's knowledge of the doctrine.
Master Canaro then nodded and said:
"All right, but before we begin, tell me whether you want to win or lose this exchange, so that I can oblige you."
Hearing those words, the Buddhist teacher realized his intentions were not pure: he didn't want that challenge to arrive at the truth, only for the pleasure of winning.
So he bowed, thanked Master Canaro for making him see that weakness in himself, and from that moment on became his disciple.
