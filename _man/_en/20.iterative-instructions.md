Loop instructions are the fundamental building blocks of programming.

There are three kinds of loop instructions:

- `for`
- `while`
- `do while`.

These instructions are made up of two parts: a *loop control* instruction, which determines how long it runs, and a *loop body*, made up of the instructions that get repeated on each iteration.
The repetition can go on either for a set number of times, or until a given condition is met.

The `for` loop is used when we want to run the loop a fixed number of times.
Its general form is as follows:

```
// controllo del ciclo
for(<stato iniziale> ; <stato finale> ; <variazione>)
{
    // corpo del ciclo
    <istruzioni>
}
```

The three conditions inside the parentheses are used by the `for` instruction to control the execution of the instructions inside the loop body.
The first expression is evaluated only once, at the start of the loop, and is usually used to initialize the variables in use.
The second expression is a logical or relational condition evaluated at the start of every iteration: if it returns `0` or `false`, the loop stops running, otherwise it continues.
The third expression is evaluated at the end of every iteration, and usually consists of an expression that increments or decrements the variables used to control the loop.
To do this, we use unary operators (that is, operators that act on a single variable) called *increment operators* and *decrement operators*, which can carry out their function either before or after the variable is used, depending on whether they're placed before or after the variable's identifier:

```
x++     // valuta x e poi la incrementa di un'unità
++x     // incrementa x di un'unità e poi la valuta
x--     // valuta x e poi la riduce di un'unità
--x     // riduce x di un'unità e poi la valuta
```

If `x` is a variable of integer or floating-point type, the increment is by one arithmetic unit; if instead `x` is a pointer, the increment equals the size of the variable the pointer refers to, as I showed you when we talked about data types.

You can also use a `for` loop to run loops with an undefined number of iterations; just omit the three control expressions, leaving only *null statements*, made up of the bare `;` terminator:

```
for( ; ; )
{
    // istruzioni
}
```

The `do-while` loop is the same as the `while` loop, with the only difference that the `while` condition is evaluated at the end of the iteration, so the loop body is run at least once.
The general form of the `do-while` loop is:

```
do
{
    <istruzioni>        // corpo del ciclo
} while(espressione)    // espressione di controllo
```

I won't insult your intelligence with an example; I'd rather point out something more serious: the code of these programs is clumsy.
The problem is that the `mostraPianeta` function does too many things: it not only figures out the Planet's name, it also prints it to the screen.
As a result, our output also includes that ugly error string about position number nine.
In the first example where we used it, this wasn't a problem, but now that our program is growing, we need to make each function more specialized, separating data processing (figuring out which planet it is) from the user interface (printing the name to the screen).
To do this, we'll use a new function, which we'll add to the `pianeti.cpp` file:

```
/**
 *  nomePianeta
 *  Torna il nome di un pianeta, data la sua posizione.
 *  @param  int    posizione Posizione del pianeta.
 *  @return string nome      Nome del pianeta o null, se errore.
 */
string nomePianeta(int posizione)
{    
    string nome;

    /**
     * Questa funzione gestisce solo i casi esistenti
     * e lascia che sia la funzione chiamate a gestire
     * eventuali errori.

    switch( posizione ) {
        case POS_MERCURIO:
            nome = "Mercurio";

        case POS_VENERE:   
            nome = "Venere";            
            break;
        case POS_TERRA:    
            nome = "Terra";             
            break;
        case POS_MARTE:    
            nome = "Marte";             
            break;
        case POS_GIOVE:    
            nome = "Giove";             
            break;
        case POS_SATURNO:  
            nome = "Saturno";           
            break;
        case POS_URANO:    
            nome = "Urano";             
            break;          
        case POS_NETTUNO:  
            nome = "Nettuno";           
            break;          
        case POS_PLUTONE:  
            nome = "Plutone";           
            break;
    }

    return nome;
}
```

Writers, sculptors, and painters cannot modify their works once they've been published. They can add a touch of color here, a chisel stroke there, but these are always minor adjustments, which don't change the very structure of the art-object.
Musicians, theater artists, and, to some extent, filmmakers are a bit luckier, because they can more easily make changes to their works, but even so, these are still rare events.
Good software production, by contrast, has the advantage of being in constant evolution.
A piece of software can be *stable*, that is, have no known flaws, but it's never finished, complete, *perfect*; partly because actual use might reveal flaws that slipped through testing, partly because changes in the context of use might call for changes to the system.
Good software is therefore constantly updated, and the different versions of the same product are numbered progressively with codes made up of three numbers separated by dots, indicating, respectively, the *major* version, the *minor* version, and the *patch*; for example: *1.4.12*.
The *major* version is incremented every time drastic changes are made to the software, making it incompatible with previous versions.
The *minor* version is incremented when the code is changed in a smaller way, adding or modifying features in a way that's compatible with previous versions.
The *patch* number is incremented every time even minor changes or fixes are made to the system.
*Major version zero* (_0.y.z_) is reserved for initial development, when the software isn't stable yet and everything can change at any moment.

Not out of stubbornness, nor out of idealism, but because this *non-metaphysical metaphysics*, as I called it earlier, works: it helps me decide what the right thing to do is when it isn't easy to tell what the right thing to do is, and it helps me get through the hard moments of life, mine or other people's, without accepting anything on faith, but relying only on considerations of a logical nature.
Besides, it's a story that hasn't been written yet, and, at the same time, the story our species has always been writing.
That seems reason enough to me.
