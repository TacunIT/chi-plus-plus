Syntactic freedom and the conciseness of its constructs are C++'s main distinguishing feature.

C++, like all languages, has its own syntax, which defines the elements of the code and how they're used within programs.
In the examples we've seen so far, I've used some of these elements without explaining precisely what their role is, because I wanted to give you an overall picture of the language's features.
A bit like when you arrive at a party and get introduced to the other guests one after another, and by the end the only thing you remember is the ladies' necklines.
Now, though, we need to take a step back and examine these elements one by one, starting with what is the main element of every C++ program: the `main` function.

```
int main(int argc, char** argv)
{    
    return 0;
}
```

The one above is the smallest C++ program you can write.
It's also the most useless one, though, because it doesn't do anything.
The example below is just as useless, but a bit more complex:

```

```

Its output, as you've probably guessed, is:

```
% g++ program-structure-hello-world.cpp -o ../out/example
% ../out/example
Hello, World!
```

The first line of code:

```
#include <iostream>
```

isn't an instruction, but a directive for the preprocessor.
The preprocessor is a program that processes the code and prepares it for compilation; we'll talk about it
<a href="/en/preprocessor" class="xref">later on</a>.
For now, all you need to know is that this instruction causes the file `iostream.h` to be included in the code, which contains, among other things, the declaration of the `cout` *stream*, used in the program's only instruction.
The second line of code:

```
int main(int argc, char** argv)
```

defines the `main` function, specifying that it will have `int` as its return value and two parameters: an integer and a pointer to pointers to `char`.
These values are used to handle the parameters passed on the command line: the integer `argc` (short for *argument count*) specifies the number of parameters present in the call, including the program's own name, while the parameter `argv` (*argument vector*) is an array of pointers to all the strings present on the command line.
The body of the function is enclosed in a pair of curly braces `{}`.
The instruction:

```
std::cout << "Hello, World!" << std::endl;
```

writes the string `Hello World!` to the `std::cout` *stream* (*standard character output*), which usually corresponds to the computer's screen.
The `<<` operator, in this case, tells the system to send whatever is on its right to output, and returns a *reference* to the output stream, so that it can be chained on the same line of code, which is a more efficient and elegant form of:

```
std::cout << "Hello, World!" ;
std::cout << std::endl;
```

Finally, the instruction:

```
return 0;
```

returns the value `0` to the calling program (usually, the computer's *shell*) to indicate that no errors occurred during processing.

As I've told you, C++ is a *compiled* language, so the code, in order to be run, must be processed by the compiler with the *shell* command:

```
% g++ program-structure-argc-argv.cpp -o src/out/example
```

The `-o` parameter lets you specify the name of the output file, in this case: `src/out/example`.
If you don't set this value, the compiler generates a file named `a.out`.
The *shell* command:

```
% src/out/example; echo $?                                  
```

is made up of two instructions, separated by the `;` character.
The first instruction runs the compiled file; the second prints its return value to the screen.
If we run the program with no parameters, we get a message and error code `1`:

```

```

```

two
three
0
```

<hr id="coupling">

No program worthy of the name has only the `main` function; it splits its work across a series of functions, each carrying out a precise, well-defined task.
In a well-written program, functions display two characteristics, which English speakers and Anglophiles call: *low coupling* and *high cohesion*.
By the *coupling* of two functions, we mean the amount of information function *A* needs to have about function *B* in order to use it.
Each function expects to receive a well-defined set of parameters: the `doubleSalary` function, which we saw earlier, expects to receive a single parameter, of type `long`:

```
long doubleSalary(long salary);
```

while a hypothetical `deductVat` function might require two: the salary amount and the VAT rate:

```
float deductVat(long salary, float rate);
```

In both of these cases, all a third function needs, to be able to call `doubleSalary` or `deductVat`, is their *interface*, that is, the number, type, and order of the parameters to pass.
Now imagine that a clumsy programmer has written the function `doSomething`, which can carry out several distinct actions depending on the parameters it receives:

```
float doSomething(long salary, int action, float rate = 0)
{
    float value = 0;
    /** Differentiates the action based on the value of action */
    if(action == 1) {
        value = salary * 2;
    } else if(action == 2) {
        value = salary / ((100 + rate) / 100);
    }
    return value;
}
```

To use this function, we not only have to know its interface, we also have to know which actions correspond to the various values of the `action` parameter.
This is madness, *meshuggah*, because if one day the author modified it and decided that the value `1` of the `action` parameter now triggers VAT deduction while the value `2` triggers doubling the salary, we would also have to modify *every* function that called it, to adapt them to the new rules.
Not only would we waste time, but if we forgot to update one or more calls we would end up with a program that behaves incorrectly.
*Coupling* <!-- uso il termine inglese per evitare anfibologie con l'attività sessuale.. --> is like cholesterol: the lower, the better; so, to avoid mistakes, we need to reduce it by creating an `enum` to hold the possible values of the `action` parameter:

```
enum Action { doubleIt, deductVat };
```

As often happens, a single line of well-written code lets us save time and get more robust code, because the effect of the `action` parameter, this way, will be entirely independent of its numeric value:

<!-- @todo: verificare il codice -->
```
enum Action { doubleIt, deductVat };

float doSomething(long salary, Action action, float rate = 0)
{
    float value = 0;
    /** Acts based on the label, not the value */
    if(action == doubleIt) {
        value = salary * 2;
    } else if(action == deductVat) {
        value = salary / ((100 + rate) / 100);
    }    
    return value;
}
```

The `doSomething` function has another design flaw too, besides high coupling: it lacks internal cohesion.
In a well-written program, each function should have only one... function:

<!-- @todo: verificare il codice -->
```
inline long doubleSalary(long salary)
{
    return salary * 2;
}

inline float deductVat(long salary, float rate)
{
    return (salary / ((100 + rate) / 100);
}
```

Even in such a simple example, you can clearly see the difference between a function that can carry out several unrelated actions and a function that carries out a single, well-defined action.
By reducing the complexity of the function, moreover, we're able to declare it as <a href="/en/functions#inline" class="xref">inline</a>, increasing the program's execution speed.
