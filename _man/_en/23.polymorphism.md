Before going on, though, we need to clear up three terms related to polymorphism: *overload*, *override*, and *redefinition*.
By the term <b id="overload">overload</b> of a function, we mean a function that has the same name as another one, but different parameters.
A typical example of *function overload* is the different versions of a class's constructor:

```
Cavallo() {}
Cavallo(const char* razza, const Sesso sesso )
: Animale(razza, sesso ) {
}
```
The two functions have the same name, and the compiler will pick one or the other depending on the parameters used.
An <b id="overridden">overridden</b> function is a function with a different definition from that of a virtual function in one of its base classes:

```
const char* getSpecie() const {
    return "Asino";
}     
```

As we've seen, the compiler picks one or the other depending on the type of the object used in the call.
If the base class's function hadn't been virtual, this would have simply been a **redefinition**:

```
class Persona {
public:
    void getClass(){
        cout << "Persona" << endl;
    }
};
```

When it handles these functions, the compiler doesn't perform a dynamic type check based on the object's type at runtime, but picks the function to call based on the type of the pointer or reference used, which, as you know, can cause problems:

```
Madre   * ptrM = new Madre;
Persona * ptrP = ptrM ;
ptrM->getClass() ;
ptrP->getClass() ;  // chiama la funzione di Persona - ERRORE
```

<hr id="overload-operatori">

In C++, every operator corresponds to a function.
The one for the `+=` binary operator, for example, is:

```
<tipo>& operator +=  (<tipo>& a, <tipo>& b) ;
```

where `a` and `b` are the two objects involved in the operation, and `<tipo>` is the type of the variables involved in the operation.

```
int&    operator +=  (int&    a, int&    b) ;
float&  operator +=  (float&  a, float&  b) ;
double& operator +=  (double& a, double& b) ;
```

Since unary operators can be prefix or postfix, to let the compiler tell which is the correct function to use, the postfix operator's function gets an extra, unused, second parameter:

```
void operator ++ (<tipo> a) ;           // versione prefissa
void operator ++ (<tipo> a, <tipo>) ;   // versione postfissa
```

*Overloaded* operator functions can be called directly.
The two instructions below, once compiled, produce the same code and the same result.
If you can find any reason to use the first syntax instead of the second, go right ahead:

```
a = b.operator + (c) ;  
a = b + c ;
```

The behavior of operators is predefined for all standard types, and can be redefined to also handle aggregate data types like structs or classes.

- the selection operator `.`;
- the pointer-to-member resolution operator `.*`;
- the scope resolution operator `::`;
- the conditional operator `? :`;
- the symbols `#` and `##`, used by the preprocessor.

When you redefine an operator's behavior for a class, you need to take into account the visibility of the member data it has to use.
If the operator, as is likely, needs to handle private or protected data, there are two options: either make use of the class's interface functions, or declare the operator a `friend` of the class.
In the initial example, both options are used: the stream output operator for the `Animale` class uses the class's interface functions:

```
ostream& operator << (ostream& os, const Animale& animale) {
    os  << "Specie:" << animale.getSpecie() << "\t"
        << "Razza:"  << animale.getRazza()  << "\t"
        << "Sesso:"  << animale.getSesso()  
        << endl;
    return os;   
}
```

while the output operator for the `Monta` class is declared a `friend` of the class, and can therefore access the instance's member data directly:

```
friend ostream& operator << (ostream& os, const Monta& copula) {
    os << "DATA: "    << asctime(localtime(&copula._giorno))
       << "MASCHIO: " << *copula._maschio
       << "FEMMINA: " << *copula._femmina;
       return os;   
};
```

The choice between the two options depends on the kind of program you need to write: if you're after speed, choose the second one, which is more direct; otherwise choose the first, which will probably run slower, but won't need rewriting if the class's structure changes.
It isn't possible, though, to redefine as a class's member function an operator function whose first parameter is a class you don't control (such as, for example, the `<<` operator function, whose first parameter is a reference to `ostream`), because in a member function this parameter would be replaced by the implicit `this` parameter, which has a different data type, causing a compile error.
The operators `=`, `()`, `[]`, and `->` cannot be redefined as global functions, and must always be implemented as a class's non-static member function.
The other rules to remember, in these cases, are:

- the unary assignment operator `=` is the only case of a member function that isn't inherited by any child classes; if it isn't redefined, it performs a member-by-member assignment of the attributes, and has the syntax:

    ```
    C& C::operator = (const C& origine) ;
    ```

- the binary operator `[]` lets you implement vectors of a particular kind while keeping a standard syntax, and has the form:

    ```
    c.operator [] (n) ;
    ```

    where `c` is an object of class `C` and the index `n` can be of any data type;

- to redefine the binary function-call operator `()`, you use the syntax:

    ```
    c.operator()(p) ;
    ```

    where `c` is again an object of class `C`, and `p` is a list, possibly empty, of parameters;

- the unary class-member access operator `->` is interpreted as:

    ```
    (C.operator -> ())->m ;
    ```

    and returns either an object or a pointer to an object of class `C`.


Redefining the `new` and `delete` operators, whose behavior is closely tied to the hardware, might not be a clever choice from the point of view of code portability; that said, if a class needs to manage memory in a particular way, it can do so, but must follow two rules:

- the `new` operator must have its first argument of type `size_t`, and return a pointer to `void`;
- the `delete` operator must be a `void` function with a first argument of type `void*`, and a second, optional, argument of type `size_t`.

In C, to convert an `int` into a `double`, you use the cast operators:

```
long int i = 5 ;
double d = (double) i ;
```

C++ accepts this syntax, just as it accepts using `malloc` instead of `new`, but its standard syntax, which vaguely resembles class constructors, requires the data to be converted to be passed as a parameter to a function with the same name as the type you want it converted into:

```
long int i = 5 ;
double d = double(i) ;
```

The C++ compiler is able to convert any primitive data type into another, but it has no way of knowing how to handle user-defined data types; we therefore have to instruct it, just as we did with constructors and operators, by defining coercion paths from and to primitive data types.
The first case, that is, converting from a primitive type to a user-defined one, is the simplest: it essentially means defining, where one doesn't already exist, a constructor for the new class that takes parameters of primitive type.
When instead there's no constructor to extend — that is, when the coercion goes from a user-defined type to a primitive data type, or one provided by a library whose source code you don't have — you need to redefine the `()` conversion operator.
Imagine you've created a new data type, `Frazione`, to handle rational numbers.

<hr id="template">

The last thing I need to tell you about, regarding polymorphism, is *templates*.
In C++, templates are blueprints used to define multi-purpose functions or classes.
If the same task can be carried out in a similar way on parameters of different types, instead of writing identical functions or classes for each parameter type, you can write a single template function or class and call it each time with the appropriate parameter type:

```
int    somma(int    a, int    b) { return a + b; }
float  somma(float  a, float  b) { return a + b; }
double somma(double a, double b) { return a + b; }

template <class T>
T somma(T a, T b) { return a + b; }
```

When the compiler finds a template in the code, whether it's a class declaration or a function call, it replaces it with the corresponding code, the same way it does for <a href="/en/preprocessor#macro" class="xref">precompiler macros</a>, but, unlike what happens with macros, the type of the template's parameters is subject to strict checking, just like the rest of the code.
The format for declaring a <i id="funzioni-template">template function</i> is:

<p class="code">
<b>template <class</b> <i>identificatore</i><b>></b> <i>dichiarazione</i>;
<br />  
<b>template <typename</b> <i>identificatore</i><b>></b> <i>dichiarazione</i>;
</p>

There's no difference between the first and second form: both `class` and `typename` produce the same effect.
*identificatore* is a symbol identifying a given data type, or a user-defined class.
For example, the syntax for a template function that returns the greater of two parameters would look something like this:

```
template<class T>
T maggiore (T x, T y) {
    return (x > y) ? x : y;
}
```

In this case, the type identifier is the letter `T`, which appears both between the angle brackets on the first line and in the parentheses on the second, but it can be any string.
There can be more than one parameter:

```
template<class C1, class C2>
T funz (C1 x, C2 y) {
...
}
```

and they can have a default value:

```
template<class N = int>
T funz (N n) {
...
}
```

Calling template functions is similar to calling ordinary functions, with the addition of the type of the parameters to be handled:

```
cout << maggiore<int>   (  9,  12) << endl;    
cout << maggiore<double>(0.4, 1.2) << endl;    
cout << maggiore<char>  ('a', 'z') << endl;    
```

<hr id="stl">

The code I showed you at the start of this lesson uses a template class:

```
list<Monta> monte;
```

The `list` class is one of the classes of C++'s *Standard Template Library*, a library of classes and functions that let you solve common programming problems, such as storing, sorting, or searching a set of data.
The components of the STL are:

- a library of **containers**, which let you store objects and data;
- **iterators**, which let you move through a container's contents;
- a collection of **algorithms**, which let you perform sorting and searching operations on sets of data;
- function-objects, or **functors**, which encapsulate a specific function.

The `list` class is an example of a container, and represents a list of elements stored in non-contiguous areas of memory.
By contrast, the `vector` class implements a list of elements stored in a single area of memory, the same way C arrays work.
All STL vectors have member functions that let you manage their elements; the `push_back` function, for example, adds an element to the end of the list:

```
monte.push_back(Monta(cavallo, giumenta));
monte.push_back(Monta (asino, asina));       
monte.push_back(Monta (asino, giumenta));     
monte.push_back(Monta (cavallo, asina));
```

<i id="iteratori-stl">Iterators</i> are constructs that let you move through a container's contents, picking out its elements.
We used one in the instruction:

```
list<Monta>::iterator it;
for (it=monte.begin(); it!=monte.end(); it++) {
    cout << *it << endl;
}
```

The first instruction in the `for` loop assigns to the iterator `it` the first element of the list, returned by the member function `monte.begin`.
The second instruction checks that the iterator is different from `monte.end`, which points to the end of the list.
The third instruction increments the iterator by one position, and shows how redefining an operator for a class makes the code easier to read: even if you've never seen a template class before, you can immediately tell that instruction increments the value of `it` by one.

Enthusiasm aside, maps and satellite images have several things in common with metaphysical disciplines.
Maps and images too, like metaphysics, are forced to represent their subject at a reduced scale, and with only two dimensions instead of three (or four, if, besides depth, you also want to factor in time).
For this reason, maps and images too have to represent their subject by means of analogies: topographic maps use contour lines and symbols; satellite images use pixels or tiny colored dots.
In neither case is what we see actually what it represents; it's our brain that decides to believe it is: in the case of the topographic map, because the legend lets us establish a correlation between meaning and symbol; in the case of the image, because our eye recognizes trees, the sea, or houses in those combinations of pixels or colored dots.
Another analogy, a consequence of the two points above, is that it's a mistake to confuse symbols with what they represent: the little dark squares on maps are *not* houses; the colored dots in the images are *not* a forest.
Maps and images only make sense at a certain level of reading; if we go beyond it, if we try to get more information or more realism by looking closer, we get the opposite effect, because the symbols reveal themselves for what they are: colored dots or lines on a sheet of paper.
This doesn't mean that what they represent is false, though, only that we aren't looking with *the right pair of eyes*, as Hunter Thompson would say.
Master Canaro thought this was the reason some religions oppose the direct depiction of the Divinity: because it's easy to end up confusing the symbol with what it represents.

Going back to our original comparison, traditional religions are satellite images, while C'hi++ is a topographic map.
While religious Creeds manage to reproduce &mdash; within the limits imposed by our condition &mdash; all the beauty of Creation, C'hi++ limits itself to giving a schematic description of it, poorer in content and in poetry, but easier to accept for those who don't have the blessing of Faith.
A satellite image has contemplative value: it's beautiful to look at on your computer screen, or even to hang on a wall, like a painting, but if you've gotten lost in a forest or in the middle of the mountains, a topographic map, precisely because of its schematic nature, will more easily let you find your way home.
C'hi++ doesn't try to steal followers away from canonical religions.
That would make no sense: it would be like trying to convince someone already married to the love of their life to enter a marriage of convenience instead — if you have Faith, you don't need rational confirmations; they might please you, but you don't need them.
What C'hi++ can do, though, is give strength to those (many) people who *still believe in everything no one believes in anymore*, as Longanesi described them; that *Band of Honest Men*<a class="nota" href="/en/notes#fn-banda-onesti" id="banda-onesti">3</a> who, every day, do their duty as best they can even when it doesn't pay off, even when everything and everyone around them seems to push them toward selfishness and indifference.
It can help them not to give up, and can teach them that winning isn't what matters — playing ever better is.
Recognizing the mistakes you've made, learning from them, and trying not to repeat them, game after game, in a continuous search for the best you can do.
If they behave this way, whatever their job is, even if it's cleaning toilets, it will still be Art.
