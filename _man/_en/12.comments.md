Comments are the single most important thing when programming.

A comment is a short piece of text that describes, in plain language, the workings or purpose of the code it refers to:

```
/**
 * Function to double a salary.
 * Takes as parameter the value of the
 * current salary and returns the
 * doubled value.
 */
long double_it(long salary)
{
    return salary * 2;   //  doubles the value
}
```

Even though it sits inside the code, a comment is ignored at compile time, because it's an addition useful only to the programmer.
The processor doesn't care to know what the code it's executing is for: it just executes it, without getting distracted by functional — or, worse, ethical — implications that would slow down its response time.
It's important to comment your code well.
Good code produces programs that last over time, and, over time, one tends to forget.
Even if you're lucky enough to have a good memory, you have to consider the possibility that your code will be used by a colleague.
Not only is it rude to force them to analyze every single line of code to understand how it works, it's also unproductive.
Just like a shortage, an excess of comments is also a mistake to avoid.
The inexperienced programmer doesn't comment their code because they think it's a waste of time. This is truly an irresponsible attitude.
The overzealous novice fills their code with comments, either because they assume whoever reads it will be less experienced than they are, or because they want to show off their limited knowledge.
The experienced programmer only comments what isn't self-evident, and points out the possible weak spots of their program so that whoever ends up reusing or fixing it pays due attention to them.
The perfect programmer doesn't comment their code at all, because their code is self-explanatory:

```
typedef Amount unsigned long;

Amount doubleSalary(Amount currentSalary)
{
    return currentSalary * 2;   
}
```

This is the same piece of code we saw before, but this time comments aren't needed, because the function's intent is expressed directly in the code, by making explicit the name of the function, of its parameter, and of its return value.
The `typedef` instruction, which you've also seen in previous examples, is used to define new data types as aliases for existing data types.
The instruction:

```
typedef time_t Date;
```

defines the data type `Date` as an alias for the standard data type `time_t`, while the instruction:

```
typedef Amount unsigned long;
```

defines the data type `Amount` as an alias for the standard data type `unsigned long`.
Defining your own data types not only makes the code more readable, it also makes it easier to modify.
The original function uses values of type `unsigned long`, that is, 4-byte sequences that allow storing integers from 0 to 4,294,967,295 (2<sup>32</sup>).
This data type can be used for a salary with no decimal digits, like Master Canaro's back in the last century, but if we had to double a salary with decimal digits, we'd have to use data types like `float` or `double` and, without a `typedef`, we'd be forced to change both the return value and the parameter type of the function:

```
float double_it(float salary)
{
    return salary * 2;   //  doubles the value
}
```  
Whereas if we've defined our own data type, we only need to modify the `typedef` instruction, leaving everything else unchanged:


```
typedef Amount float;

Amount doubleSalary(Amount currentSalary)
{
    return currentSalary * 2;   
}
```

It's acceptable for comments to have a touch of humor — it can help release some of the tension for the reader, the same way cracking a joke while working does, but even here you shouldn't overdo it.

C++ provides two distinct ways to comment code: two adjacent `//` characters, or the `/*  */` sequence, which the Language borrowed from its predecessor, C.
In the first case, the compiler ignores everything from that symbol to the end of the line; in the second case, it ignores everything between the two character sequences.

```
/**
 * This syntax is more convenient for comments
 * that span more than one line
 */
for (int i = 0; i < 10; i++) {
    cout << i << endl; // this one, for single-line comments
}
```

Both solutions have pros and cons that become clear with experience. Choosing one over the other is often a matter of convenience, other times a matter of style, but if you understand the true essence of the Language, it's just a matter of rhythm.

Comments are a bit like literature: writing too little is bad; writing too much, dwelling on needless descriptions, is just as wrong.
The comparison also holds in reverse: books, works of literature, are the comments on the code of life.
You write down what you wish to remember.
