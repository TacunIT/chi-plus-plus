The purpose of functions is to make code easier to manage.
It's possible to write a program that only has the `main` function, but this only makes sense for very simple programs, like some of the examples we've seen so far.

Functions are one of C++'s <a href="/en/data-types#funzioni">data types</a>, and like all data types, they can be *declared* and *defined*.
The <a id="dichiarazione">declaration</a> of a function establishes its return type and the parameters it requires as input:

```
float scorporaIVA(long stipendio, float aliquota);
```

The <a id="definizione">definition</a> of a function establishes its behavior:

```
float scorporaIVA(long stipendio, float aliquota)
{
    float valore;
    valore = stipendio / ((100 + aliquota) / 100);
    return valore;
}
```

As I told you when we talked about <a href="/en/programming-languages">programming languages</a>, generating an executable file happens in two stages: first the compiler converts the C++ code into an *object file*, then the *linker* turns the object files (there could be more than one) into a single executable.
For this process to work, a function's *declaration* must be present in every piece of code that uses it, so the compiler can check that it's being used correctly; the *definition*, by contrast, must appear only once.

Earlier I told you to *consider* the possibility of isolating repeated instructions into a function, because creating a function isn't always the right choice.
Software, like many human endeavors, is the product of a series of trade-offs, and you'll have to make different architectural choices depending on the kind of program you need to build.
Good software, besides working correctly, should be fast, easy to modify, and require few system resources.
Sometimes, though, you have to sacrifice one of these positive traits to boost another.
For example, if you had to write extremely fast software, it might be better to have duplicated instructions rather than function calls, because calling a function inevitably causes some slowdown.

The existence of an `inline` function is somewhat reminiscent of a passage from the <i>Samyutta Nikaya</i>:

 > There is no distinction between a being, its function, and the time of its appearance. Beings appear out of non-existence, exist for an instant, and then cease to exist. Their existence, activity, and action are one and the same thing. Past and future are mere names.<a class="nota" href="/en/notes#fn-samyutta" id="samyutta">2</a>

In general, though, the lifecycle of a function involves three distinct phases: *declaration*, *definition*, and its subsequent use:

```
/** Dichiarazione */
float raddoppiaStipendio(float stipendio);

/** Definizione */
float raddoppiaStipendio(float stipendio)
{
    return stipendio * 2;
}

/** Chiamata */
float importo = raddoppiaStipendio(1500);

```

Only the `main` function is exempt from this rule, because it requires no declaration and is never called by other functions.
<!-- Qui si potrebbe fare un parallelo con gli elementi costitutivi dell'Universo, secondo lo Spazionismo, ma preferisco separare la dottrina dal manuale C++ vero e proprio -->
The variable `float stipendio`, which appears in parentheses both in the declaration and in the definition of the function, is called the function's *parameter*.
The value `1500`, assigned to the parameter in the function call, is called the *argument*.
These variables are also called *formal parameters*, in the case of the declaration and the definition, and *actual parameters*, in the case of the call.
When you call a function, you can pass it arguments in three ways: by *value*, by *reference*, by *pointer*:

```
{% include_relative src/funzioni-argomenti.cpp %}
```

The output of this program is:


Passing a *reference* to a variable as a function argument is equivalent to passing the variable itself.
This can be useful for functions that require large objects as input, or that need to directly modify the value of the variables passed to them as parameters, but it should be avoided in any other case, because it lets the program modify the value of a variable in a sneaky way, which can be very hard to spot when debugging.
Another thing worth knowing, about function parameters, is that they can have default values:

```

```
The output of this program will be:

```

Valore: 0
```

A parameter's default value must be specified in the function's declaration.
Remember, though, that when you assign a default value to one of a function's parameters, you must do the same for every parameter that follows it, if there are any:

```
void funz1(float f, void * ptr = NULL);        // OK
void funz2(double d, int b = 2, char c = 'c'); // OK
void funz3(int i = 3, int n );                 // ERRORE!
```

Default values are used when one or more of a function's parameters is often assigned the same value.
For example, if the `log` function we saw in the previous example were mostly called with the same value for the `livello` parameter, we could assign it as a default:

```
inline void log(const char* messaggio, int livello = LOG_AVVISO);
```

making the code faster and easier to write:

```
log("ho aperto il file");

doc << "Testo del documento.\n";
log("ho scritto sul file", LOG_DEBUG);

doc.close();
log("ho chiuso il file");
```

<hr id="parametri-variabili">

In one of our
<a href="/en/programmer" class="xref">earliest conversations</a>,
I told you that the Good Programmer, while writing code, always asks whether there's a more efficient way of doing what he's doing.
Today you've seen proof of that: we improved the first example in the section on `inline` functions by unifying message handling into a single function, and then improved it again by making that function `inline`.
There's a problem, though: the program's output:

```
% ./src/out/esempio                                   
ho aperto il file
ho scritto sul file
ho chiuso il file
```

only works if there's a single file to handle, as in our case.
But if there were two (or more) input or output files, it would be useful to know *which* file the message refers to.
To solve the problem, we could increase the number of formal parameters of the `log` function:

```
void log(int livello, const char* messaggio, const char* file);
```

but the new parameter would be useless for calls like:

```
log(LOG_ERRORE, "specificare il path del file");
```

Alternatively, we could build the message inside the calling function:

```
string s1 = "ho chiuso il file: ";
string s2 = filename;
string s3 = s1 + s2;
log(LOG_AVVISO, s3.c_str());
```

but this would make the code heavier, more complicated, and slower; it would also shift back into the calling functions part of the output functionality we had happily isolated into the `log` function.
The correct solution to this kind of problem is *functions with variable parameters*.
A typical example of this kind of function is the C language's `printf` function, whose first argument is used to determine the type and number of the arguments that follow it:

```
/** Dichiarazione, nel file stdio.h */
int printf(const char *format, ...) ;

/** Utilizzo */
printf("stringa: %s; intero: %d", "codice errore", -1);

```

To indicate variable arguments, in a function's declaration, three dots are used after the fixed parameters:

```
void log(int livello, int n_parametri, ...);
```

In the function's definition, to handle the parameters, you need to create a reading loop using three macros defined in the `stdarg.h` file:

```
void va_start(va_list ap, lastfix);
type va_arg(va_list ap, type);
void va_end(va_list ap);
```

`va_start` initializes the `ap` variable list so it can receive the variable arguments; the `lastfix` parameter specifies which is the function's last fixed parameter.

`va_arg` returns the next parameter in the `ap` list; the `type` parameter indicates the parameter's data type.

`va_end` ends the processing of the parameters, and must always be called before the function returns.

Be careful, though: a function that calls itself, if not written well, can generate an in[de]finite recursion, like two mirrors facing each other.
The perspective space inside the mirrors is inexhaustible, but a computer's memory, however large, has its limits, and too many recursions can cause a *stack overflow*.<a href="/en/notes#fn-stack" class="nota" id="stack">3</a>
For this reason, you should resort to recursion only when it's truly necessary, preferring, whenever possible, [iterative instructions](https://chiplusplus.org/en/istruzioni-iterative) and always applying mechanisms to control recursion depth, to prevent a stack overload.
Another kind of recursion to be wary of is the kind sometimes used to lend credibility to a belief with no real foundation, by citing another, similar belief.

In a novel about the life of the Japanese swordsman Miyamoto Musashi, there's a line that follows logically from the Roman proverb I told you about yesterday:

>Non bisogna interferire nel funzionamento dell’Universo, ma prima è necessario capire quale sia, il funzionamento dell’Universo<a href="/en/notes#fn-musashi" class="nota" id="musashi">4</a>

The problem with novels is that they often tell you what to do, but almost never explain how, or why.
Be wary of teachers who only know how to teach, because everything they tell you they learned from books.
A good teacher must excel at something, whether it's fencing, archery, carpentry, or motorcycle maintenance.
He may not be *the* best, but he must be *among* the best; only then will you know his ideas are sound.
By contrast, teachers who have never left a school have never had to put their ideas to the test of reality.
They teach you what they think is right, but what's right or true within a school isn't always equally true or right in the real world.
It's easy to be a saint inside a monastery, among people who share your culture<!-- qui, in senso letterale --> and your own principles; the hard part is remaining a saint outside of it.
The Buddha Shakyamuni preached Universal Loving-kindness because he lived in India twenty-six hundred years ago, but if he'd been stuck in city traffic after a day's work, his ideas might have looked a bit more like Nietzsche's.
Fortunately, Musashi &mdash; the real one, the Samurai &mdash; was always very clear both about what's right to do and about how to achieve it.

<hr id="verita">

I told you about Musashi's rules not because they're the only ones available, but because they're intensely personal.
They draw on the principles of other disciplines, like Buddhism, Bushido, or Tao, but they're *something else*, as Eddie Cochran would say.
You'll have to do the same: learn everything you can, from good and bad teachers alike, and then define your own rules for life, which may end up the same as, similar to, or entirely different from the ones you were taught.
The Way, like Truth, is a model, whom each of us portrays from our own point of view, trying to guess at the body hidden behind the folds of the drapery.
Portraying her the same way someone else has would be wrong, because your point of view isn't the same as theirs, but looking at her from several points of view can help you understand her shape better.
