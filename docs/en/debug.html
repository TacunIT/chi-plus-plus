Debugging

A hundred Worlds of sin are dissolved by the light of a single ticket

Debugging may not be the meaning of life, as your book’s title claims, but it’s without question the most important aspect of programming. In your career as a programmer you may never use a functor class or redefine the -> operator, but you will certainly make mistakes, and you’ll have to fix them. And yet, programming manuals never talk about debugging. Thousands of pages have been written about the various programming languages; not a spring goes by without some new programming paradigm blossoming — structured, object-oriented, agile, fuzzy — and yet, no one has ever bothered to formalize the process of fixing code. It’s no coincidence that the decline of software began when laser printers replaced the old dot-matrix ones. Code can’t be read on an A4 sheet: unless it’s a trivial program, it won’t fit, either in height or width. The continuous-feed paper of a 136-column dot-matrix printer, by contrast, lets you print your entire code and read through it calmly; fix it, if needed, and improve it if possible. That’s how code reviews used to be done, back when there was the time and the money to do code reviews.

Neglecting to debug is like claiming you’ll never make a mistake. It doesn’t matter how efficient the programming language is; it doesn’t matter how sharp and vigilant the compiler is; it doesn’t even matter how good the programmer is: sooner or later, distraction, tiredness, or some external event will let a bug slip into the code. A pointer used improperly, one loop too many or too few in a for instruction, a forgotten comma between the parameters of a printf: whatever it is, if the compiler can’t catch it, it will end up in the program and stay there, waiting to produce its harmful effects. The program might work correctly for years, but then, one day, something won’t go the way it should, and at that point you’ll have to comb through the code to find the bug. Making mistakes is unavoidable, and it’s important to know how to fix them. Even more important, though, is being able to notice them. Just as Ikebarba begins in the shop, debugging begins the moment you write the code. The best way to keep your code free of bugs is to write good code. Writing good code means always doing the best you possibly can. Never save a file unless you’re sure it will work the way it should, and always try to think about what else your code might do, besides what you want it to. As I told you in one of our first conversations, the Good Programmer isn’t satisfied with the quickest path, but always looks for the most efficient and reliable one, because he knows writing good code costs less than fixing badly written code. Master Canaro once said:

The Good Programmer is like a sailor who, before a long voyage, checks all of his deck equipment, taking apart every single item and replacing anything that might break, because he doesn’t know what will happen once he’s out at sea.

The Good Programmer cannot help it, because it’s in his nature; that’s the difference between someone who works as a programmer and someone who is a programmer.

Software bugs can be of three kinds:

  • bugs that show up during compilation;
  • bugs that show up while the program is running;
  • analysis bugs.

Compile errors are caused by malformed constructs, which block the process of building the program. A mistake you’ll make often is forgetting the semicolon at the end of a class declaration:

class C {
private:
    float _raggio;
    float _area;
public:
    C (int m) : _raggio(m) {}
    float getRaggio() { return _raggio; }
    float area() {
       Quadrato q;
       return 3.14159 * q(_raggio);
    }
}

Many programmers ignore warnings, thinking that if the program compiles, there’s nothing left to worry about. Don’t make this mistake: no warning should ever be ignored.

Only a formally flawless program can produce runtime errors. If it weren’t formally flawless, in fact, it wouldn’t have compiled, and couldn’t have run at all. Runtime errors are more dangerous the milder their effects are. A bug that crashes the system will certainly be noticed and fixed; a small calculation error, by contrast, might go unnoticed and therefore cause serious problems. If you forget the increment operator inside a while loop, you’ll produce an infinite loop, which will certainly catch your attention:

Runtime errors can be of two kinds: those that show up deterministically, and those that show up randomly. The bug in the list of planets will show up every time the program runs, and will therefore be (relatively) easy to spot. By contrast, this code will only produce a bug under certain conditions:

/** 
 * @file debug-errore-stocastico.cpp
 * Errore che si verifica solo in deterinate circostanze.
 */
 
#include <iostream>
#include <fstream>

using namespace std;

#define ERR_FILE_NONE       -10
#define ERR_FILE_OPEN       -20
#define S_ERR_FILE_NONE     "Definire un file di input"
#define S_ERR_FILE_OPEN     "Impossibile aprire il file di input"
#define N_CHAR_MIN          300
#define N_BANNER_MAX        3
#define PUNTO               '.'

/**
*   Aggiunge un banner dopo ogni punto, 
*   ogni N_CHAR_MIN caratteri.
*/
int banner_testo(ifstream& testo)
{
    int  n_banner = 1;
    int  letti = 0;
    char c = 0;
 
    /** 
    *  Legge tutto il contenuto del file di input
    */
    while(testo.good()) {
        
        /** 
        *   Legge il cotenuto del file, 
        *   carattere per carattere 
        */
        if((c = testo.get()) != EOF) {            

            /** Incrementa il numero di caratteri letti */
            letti++;

            /** Scrive il carattere letto */
            cout << c;            

            /** 
            *   Se il carattere corrente è un punto
            *   e ha letto almeno N_CHAR_MIN caratteri
            *   e ha ancora banner da aggiungere, 
            *   inserisce il codice del banner nel testo. 
            */
            if(c == PUNTO 
            && letti >= N_CHAR_MIN 
            && n_banner <= N_BANNER_MAX) {                
                cout << endl 
                     << "<div id=\"banner-" 
                     << n_banner 
                     << "\"></div>" 
                     << endl;
                n_banner++;
                letti = 0;
            }
        }       
    } 
    
    cout << endl;
    
    return n_banner;
} 
 
int main(int argc, char** argv)
{    
    ifstream testo;
    
    /** Verifica che ci sia il nome del file di input */
    if(argc < 2) {
        cerr << S_ERR_FILE_NONE << endl;
        exit(ERR_FILE_NONE);
    }
    
    /** Apre il file in lettura */
    testo.open(argv[1]);
    if(!testo.good()) {
        cerr << S_ERR_FILE_OPEN << endl;
        exit(ERR_FILE_OPEN);
    }

    /** Elabora il testo */
    banner_testo(testo);
    
    /** Chiude il file */
    testo.close();
    
    return 0;
}

On top of the cost of these tasks, you obviously have to add the potential damage from the system not working, which could easily amount to a month, if not a year, of the programmer’s salary. You need to think about all this when you write code, because you have a responsibility both to your employer and to the users of the system, who might well be your friends or your relatives. If you do a poor job for Coca-Cola, you can always think: “Who cares, I drink Pepsi”; not ethical, but at least not self-destructive. But if you do a poor job for the State, you’re making life worse for yourself and for everyone you know, and that, besides being unethical, is also stupid.

Analysis bugs are a consequence of the Age-Old Dichotomy — and of the analyst’s inexperience, of course. Like all sentient beings, clients too can fall victim to the Age-Old Dichotomy between what they want and what they actually need. A client stating their requirements is like a child telling their mother what they want to eat. The impulse is real — appetite or hunger, whatever it may be —, but the way they’d like to satisfy it isn’t necessarily the most correct one; in fact, it’s usually a desire induced by Entropy’s cunning servant: Marketing. The client has every right to behave this way: each of us does the same thing when we walk into a shop; as an analyst, though, you must not let your judgment be swayed by your client’s desires. In this, Analysis is comparable to a mystical discipline: just as the Buddha gives each person he meets what they need, you must set yourself aside and become the other person, think as they think, feel their needs. From what they say, and how they say it, you must be able to tell their needs apart from their desires, making sure the pursuit of the latter doesn’t get in the way of the project’s success. Obviously, you’ll also have to make sure your own desires don’t get in the way of, or slow down, the project. Just as it influences your client’s opinions, Marketing can influence you too, pushing you to pick a fashionable technology unsuited to the goal you need to reach. Be careful that this doesn’t happen. Constantly keep an eye on your choices; like the Good Programmer, always think on multiple levels, and check in with other colleagues or your supervisor to make sure you’ve made the right choices. Don’t be ashamed of your mistakes: they’re the bricks you’ll build your experience on. Also be prepared to have to make mistakes. Sooner or later, a client will ask you to make a choice you believe is wrong. When that happens, you must first try to convince them of the mistake, explaining why their request isn’t correct. If you can’t, repeat your objection, so it’s clear that yours isn’t a guess, but a certainty — but don’t push further than that: both because it could create friction with the person you’re dealing with, and because, in the long run, you might discover it was actually them who was right. If you fail to convince the client of their mistake, you’ll have to carry out what they asked, but at the same time you’ll need to make sure the consequences of that choice can’t be attributed to you down the road. Since the harmful consequences of an architectural mistake might not surface until months or even years after the analysis phase, don’t just tell the client they’re wrong — write it down, so there’s a record of your objection. Your worst enemy, in these cases, will be your own pride. Once, Master Canaro told me:

When I was a young project manager and argued with my clients, I always tried to prove to them that I was right; now I just try to do what needs to be done.

Paraphrasing Iacopone da Todi, a debugger can help you pinpoint the part of your code generating a bug, but you first have to figure out which function to examine, because debugging an entire program’s code, in the cases where that’s even possible, would be long and extremely frustrating. The way the code was written will make it more or less easy to check. Imagine the problem is the variable x: if your entire codebase has the ability to change its value, you might have to examine every single function to check it isn’t misusing it. By contrast, if the variable x can only be modified in a few places in the code, your search will be more targeted and quicker. This is why in the lesson on iterative functions we separated data processing from user-interface handling: because this way, depending on the kind of bug that shows up — a calculation bug or an output bug —, we’ll know which function to go look at. Some features of C++, like strong typing and encapsulation, can help you in this regard, but they won’t always be enough to pinpoint exactly where your code is doing something wrong. In these cases, you’ll have to proceed by trial and error, breaking your program down into smaller and smaller pieces, so as to reduce the number of lines of code you need to check. A quick way to do this is to comment out all the calls in the main function, then restore them one at a time, until you find the one that contains the bug:

int main(int argc, char** argv)
{    
    ifstream testo;        

    verifica_parametri(argc, argv);
    apri_file(testo, argv[2]);            
/*
    elabora_file(testo);
    chiudi_file(testo);
*/    
    return 0;
}

If the function that isn’t working is too complex to debug directly, you’ll repeat the process, commenting out its own calls until the amount of code left to examine is reasonably small. Another way you can make it easier to track down bugs in your code is to add messages that let you know what operation the program is performing:

int apri_file(ifstream& testo, const char* path)
{
#ifdef __LOG__    
    log(LOG_DEBUG, 2, "Apro il file: ",  path);
#endif

    testo.open(path);
    return ERR_NONE;
}

The log function is the one we saw in the lesson on functions with variable parameters, and it lets us know the name of the file being opened while the program runs. These tracing functions are useful during debugging, but they slow down the program’s execution, because they require access to an external device, whether that’s the computer screen or a file on the hard drive. For this reason, it’s good practice to be able to disable them in the final version of the program. In this case, we did so by wrapping the call in a precompiler ifdef directive, so that it’s only included in the code if the __LOG__ constant is defined. Since this is a constant that’s never used by the code itself, but only serves to change how the program is compiled, we can define __LOG__ directly on the compiler’s command line:

> g++ sorgente.cpp -D __LOG__ -o eseguibile

When the bug shows up — usually a few minutes before you have to stop working to go out or do something else —, and you have to identify its cause, the first problem you’ll face is being able to reproduce the conditions under which it occurs. As we saw a moment ago, if the bug depends on the input data, to identify the problem, you’ll need to figure out which data triggers it; sometimes it’ll be easy, but in other cases it can turn out to be extremely complex. Several years ago, Master Canaro had to register on a website that also asked for his date of birth — which, as you know, was February 29, 1964. The new-user registration form gave him no trouble, but the edit-profile form, evidently written by a less experienced programmer, wouldn’t let him update his details, because, according to it, his date of birth was invalid. This means the system used two separate functions to validate the date of birth, one in the registration function and another in the edit function, and that at least the one used in the edit function wasn’t a standard function, but ad-hoc code. Both of these choices are mistakes: a single given action on the data should always correspond to a single function. Performing the same operation with separate pieces of code is wrong; both because it increases the odds of making a mistake, and because it slows down how quickly a bug can be identified during debugging. It’s also wrong to rewrite functions that already exist: D. J. Bernstein did it, with the I/O functions of qmail, but that was a deliberate architectural choice, because he wanted functions that were better and safer than the standard library’s. On top of these two programming mistakes — unacceptable, on a site that handles financial transactions —, there’s a deep carelessness in the code’s debugging phase, because the correct handling of edge cases, like leap years, must always be checked. When you test how a program works, you can’t just check that it does what it’s supposed to do — you also have to make sure it doesn’t do what it’s not supposed to do. In particular, you need to check that it behaves correctly if:

  • you give it correct input data;
  • you give it no data at all;
  • you give it incorrect data;
  • you give it excess data.

So, if the input is a date, you’ll need to check that your system correctly handles both the value 29-02-1964 and the value 29-02-1965; if the input is a text string, you’ll need to make sure the system correctly handles the case where it receives more characters than expected, and that it strips any leading or trailing whitespace, unless that’s an actual functional requirement.

using namespace std;

/** Codici e stringhe di errrore */
#define LOG_DEBUG         1
#define LOG_AVVISO        2
#define LOG_ERRORE        3
#define ERR_NONE          0
#define ERR_FILE_NONE   -10
#define ERR_FILE_OPEN   -20
#define ERR_FILE_READ   -30
#define S_DEBUG         "DEBUG"
#define S_AVVISO        "AVVISO"
#define S_ERRORE        "ERRORE"
#define S_ERR_FILE_NONE "Definire un file di input"
#define S_ERR_FILE_OPEN "Impossibile aprire il file di input"
#define S_ERR_FILE_READ "Impossibile leggere il file di input"

/**
*   Definisce una classe derivata da exception
*   per la gestione degli errori.
*/
class Eccezione: public exception
{
private:
    int         _codice;
    const char* _errore;
public:

    /** Costruttore */
    Eccezione(int codice, const char* errore) 
    : _codice(codice), _errore(errore) {        
    }
    
    /** Funzione virtuale pura: va ridefinita */
    virtual const char* what() const throw() {
        return _errore;
    }

    /** Funzioni di interfaccia */
    int getCodice() { return _codice; }
    const char* getErrore() { return _errore; }

    /** Ridefinizione dell'operatore di output */
    friend ostream& operator<< (ostream& os, Eccezione e){
        os << e._codice << ": " << e._errore << endl;
        return os;
    }
};

/** Funzione di gestione degli errori */
void errore(int codice, bool exit = true)
{
    const char* errore = NULL;

    /** 
    *   Identifica la stringa di errore corrispondente 
    *   al codice di errore ricevuto.
    */
    switch(codice) {
        case ERR_FILE_NONE: errore = S_ERR_FILE_NONE; break;
        case ERR_FILE_OPEN: errore = S_ERR_FILE_OPEN; break;
        case ERR_FILE_READ: errore = S_ERR_FILE_READ; break;
    }

    /** Se ne trova una, lancia un'eccezione */
    if(errore != NULL){
        Eccezione e(codice, errore);
        throw e;
    }
}

/** 
*   Questa è la funzione di log che abbiamo visto 
*   nella lezione sulle funzioni.
*/
void log(int livello, int n_parametri, ...)
{        
    /** Definisce il livello del messaggio */
    const char* s_livello; 
    switch(livello) {
        case LOG_DEBUG:  s_livello = S_DEBUG ; break;
        case LOG_AVVISO: s_livello = S_AVVISO; break;
        default:         s_livello = S_ERRORE; break;
    }
    
    /** Scrive il testo del messaggio */
    cerr << '[' << s_livello << "] ";

    /** Scrive i parametri della lista */
    va_list lista_parametri;
    va_start(lista_parametri, n_parametri);
    for(int p = 1; p <= n_parametri; p++) {        
        cerr << va_arg(lista_parametri, char*) ;        
    }    
    va_end(lista_parametri);

    cerr << endl;
}

/** Verifica la presenza dei parametri di avvio */
int verifica_parametri(int argc, char** argv)
{
    return (argc < 2) ? ERR_FILE_NONE : ERR_NONE;
} 

/** Apre il file in lettura */
int apri_file(ifstream& testo, const char* path)
{
/**
*   Questo codice viene compilato solo se
*   è definita la macro __LOG__
*/    
#ifdef __LOG__    
    log(LOG_DEBUG, 2, "Apro il file: ",  path);
#endif

    testo.open(path);
    return ERR_NONE;
} 

/** Legge il file di input e lo scrive a video */
int elabora_file(ifstream& testo)
{
    int  letti = 0;
    char c = 0;
    while ((c = testo.get()) != EOF) {            
        letti++;
        cout << c;            
    } 
    return letti;
} 

/** Chiude il file di input */
void chiudi_file(ifstream& testo)
{
    testo.close();
} 
 
int main(int argc, char** argv)
{    
    ifstream testo;
    
    try {
        
        int esito = ERR_NONE;
        
        /** 
        *   Verifica che ci sia il nome del file di input 
        *   e gestisce eventuali errori.
        */
        esito = verifica_parametri(argc, argv);
        errore(esito);
        
        /** 
        *   Imposta la exception mask dello stream per fare
        *   sì che un errore di I/O generi un'eccezione,
        *   poi apre il file in lettura.
        *   Usa un blocco try/catch per intercettare una
        *   eventuale eccezione e gestirla in maniera
        *   omogenea al resto del codice.
        *   Non può aggiungere una catch in fondo perché
        *   l'eccezione non dà informazioni sulla causa 
        *   dell'errore che l'ha generata.
        */
        try {
            testo.exceptions ( std::ifstream::badbit 
                             | std::ifstream::failbit );
            esito = apri_file(testo, argv[2]);            
        } catch(ifstream::failure e) {
            errore(ERR_FILE_OPEN);            
        }
        
        /** 
        *   Elabora il testo e gestisce eventuali errori.
        *   Dato che la funzione torna il numero di 
        *   caratteri letti, deve chiamare errore solo
        *   se non ce ne sono.
        */
        testo.exceptions ( std::ifstream::goodbit);
        if(elabora_file(testo) == 0) {
            errore(ERR_FILE_READ);            
        }
        
        
        /** Chiude il file di input */
        chiudi_file(testo);
    
    } catch (Eccezione e) {
        cerr << e;
        exit(e.getCodice());
    }
    
    return 0;
}

In keeping with what we said a moment ago, to check this program’s behavior we’ll need to run at least four tests:

# dati corretti
src/out/esempio src/cpp/debug-testo-1.txt

# nessun dato
src/out/esempio

# dati errati
src/out/esempio src/cpp/file-inesistente

# dati in eccesso
src/out/esempio src/cpp/debug-testo-1.txt abcdefghilmnopqrstuvz

We then need to check that every error condition is handled correctly. In our case, the expected errors are:

#define ERR_FILE_NONE   -10
#define ERR_FILE_OPEN   -20
#define ERR_FILE_READ   -30

The first two errors are checked by the standard tests; the third case we can check by passing the program an empty file:

src/out/esempio src/cpp/debug-vuoto.txt

Unfortunately, though, if you compile and run this code with correct data, you get an error, even though the file exists:

> g++ src/cpp/debug-gestione-errori.cpp -o src/out/esempio
> src/out/esempio src/cpp/debug-testo-1.txt               
-20: Impossibile aprire il file di input
> ls src/cpp/debug-testo-1.txt
src/cpp/debug-testo-1.txt

If you recompile the program, defining the __LOG__ macro to check which file the program is opening:

#ifdef __LOG__    
    log(LOG_DEBUG, 2, "Apro il file: ",  path);
#endif

when you run the program, you get a new error:

> g++ src/cpp/debug-gestione-errori.cpp -D __LOG__ -o src/out/esempio
> src/out/esempio src/cpp/debug-testo-1.txt                           
[DEBUG] Apro il file: zsh: segmentation fault  src/out/esempio src/cpp/debug-testo-1.txt

This isn’t the behavior the function was meant to have, but it still lets us understand what the problem might be. The segmentation fault error means the program is trying to access an area of memory that doesn’t belong to it. The memory area in question is the one associated with the path parameter, which was in turn initialized with the value of the variable argv[2]:

esito = apri_file(testo, argv[2]);

The bug is the index 2 in the argv array. As you’ve probably noticed, the code of this program is a reworking of the code from the lesson on streams, which had to handle three command-line parameters. This time, though, the program’s call string only has two values: the program’s own path, and the input file’s name:

> src/out/esempio src/cpp/debug-testo-1.txt

Reusing code is a good thing; forgetting to update the value of the argv array’s index is a bad thing, because the memory address pointed to by argv[2] no longer belongs to the program: we can’t use it as a path for an open function, and we can’t print it to the screen. If we fix the index, the program handles every usage condition correctly:

> g++ src/cpp/debug-gestione-errori.cpp \
    -D __LOG__ \
    -o src/out/esempio
> src/out/esempio
-10: Definire un file di input

> src/out/esempio src/cpp/file-inesistente
[DEBUG] Apro il file: src/cpp/file-inesistente
-20: Impossibile aprire il file di input

> src/out/esempio src/cpp/debug-vuoto.txt
[DEBUG] Apro il file: src/cpp/debug-vuoto.txt
-30: Impossibile leggere il file di input

> src/out/esempio src/cpp/debug-testo-1.txt
[DEBUG] Apro il file: src/cpp/debug-testo-1.txt
Essere un ossessivo-compulsivo con una leggera tendenza ...

This careless mistake was made easier by using a hardcoded number to define the array’s index. Writing a number or a string directly into the code is certainly quicker and more tempting than defining constants for the precompiler:

#define PARAM_PATH 1
...
esito = apri_file(testo, argv[PARAM_PATH]);

but, in the long run, it’s counterproductive, because it makes the code harder to read and to modify. Harder to read, because constants help you understand what the code does. If you read the instruction:

return (argc < 2) ? ERR_FILE_NONE : ERR_NONE;

you can understand what it does even without knowing the code. If instead you read the same instruction, but without the constants:

return (argc < 2) ? -10 : 0;

to understand what it does you’ll have to go read the program’s documentation, assuming one exists. Harder to modify, because using constants instead of hardcoded values lets you change a constant’s value by editing a single spot:

#define ERR_NONE  1

If we hadn’t used a constant, to get the same result we would have had to change three separate instructions:

return (argc < 2) ? ERR_FILE_NONE : ERR_NONE;

...

return ERR_NONE;

...

int esito = ERR_NONE;

In a program more complex than our example, there would certainly have been more changes, and they’d have been harder to identify; moreover, if we forgot even one of them (likely), we would have introduced a bug into the system. Hardcoded constants can only be used in the first phase of a program’s development, when you’re not yet sure the path you’ve chosen is the right one. At this stage it’s acceptable to run tests by putting values directly into the code, but once the algorithm is reasonably stable, you’ll need to convert every value into a constant.

A system is a house that, as soon as it is built and furnished, needs (subject as it is to the corroding action of the elements) a more or less vigorous but constant work of maintenance, and that at a certain point it is no longer worth restoring and propping up, and it must be torn down and rebuilt from the foundations. But with this crucial difference: that, in the work of thought, the house is perpetually new and perpetually sustained by the old one, which, as if by magic, endures within it.1

Sooner or later, your life will crash, just like software. It doesn’t matter how careful you’ve been, or what your Karma is: at some point the ground under your feet will start to give way, and you’ll fall down the mountain you were climbing, finding yourself right back at the starting point. What you do in that moment will decide the rest of your life, and will show you what kind of man you are — or woman, since these things don’t only happen to men. Start preparing for that moment now, because, when it comes (when, not if), you’ll probably be alone, and making decisions will be very hard, because you’ll have lost all faith in yourself. If you break a leg, or an arm, your brain can tell you whether it’s healing or getting worse, but if you hit your head, it isn’t easy to tell how you’re doing, because the organ being examined and the organ doing the examining are one and the same. In the same way, if you don’t trust yourself, it’s hard to tell whether the choices you’re making are the right ones. Fear or caution might push you away from the right choice, so, first of all, you’ll need to debug your own life, to understand whether, and to what extent, you should blame yourself for what happened; once you’ve done that, you’ll need to identify your mistakes and find a way not to repeat them. The most common approach is to look back and try to figure out what your own faults have been, treating them as isolated events within an existence made up mostly of correct choices. But since each of us tends — more or less unconsciously — to look for causes outside ourselves for our own misfortunes, inventing conspiracies or shifting our own responsibilities onto others, the best thing to do, in these cases, is to adopt the opposite attitude, and start from the assumption that everything bad that has happened in your life is your own fault, and only then identify the cases where what happened, in fact, didn’t depend on you. This bottom-down2 approach has two merits: the first is that, by analyzing past events, you might discover that some of the blame you’d assigned yourself wasn’t real; the second is that it will be harder to lie to yourself. Each of us bears some share of responsibility for what happens to us, even for events we didn’t directly cause. Assuming, from the outset, the blame for all the bad that has happened to us makes it harder to lie to ourselves, and lets us identify all our faults, however small. Be careful, though: you shouldn’t think of your faults the way a thirteenth-century inquisitor would, but as bugs in the program of your life. Some will generate errors, others only warnings, and your duty is to identify as many of them as possible, and then try to fix them, to improve how the system runs.

Change begins when you set out on a new path, even if that path is nothing more than a track left by a thirsty goat that found a spring.3

Unlike with software, it isn’t always possible to fix the bugs of our existence. You can change a flawed aspect of your character, but that doesn’t mean it’s possible to undo the damage it caused to yourself or to others. To draw a parallel with software debugging, character flaws are compile errors, while the effects of our mistakes are runtime errors; the former we can fix, the latter, no: what’s done is done. What we can and must do, though, is sincerely repent them, that is, acknowledge them as mistakes, so as to avoid repeating them in future cycles of existence. Even if we can’t undo the effects of a mistake, we can still ask forgiveness from those we’ve harmed. Saying sorry won’t be enough: you say sorry when what you did wasn’t up to you, like when someone bumps into you and you spill your wine on your neighbor’s clothes; but if you threw the wine at them on purpose, you’ll need to ask forgiveness, try as much as possible to make up for what you did, and never do it again, either to them or to anyone else. Don’t underestimate the healing power of forgiveness, even though there will be cases where you won’t get it. While it’s true that you can’t change the past for someone you’ve hurt, it’s just as true that you can try to make up for the harm done by making their present or future better. The people we hurt are often the ones we’re closest to; letting them know we haven’t forgotten them, and that what happened pains us, can help mend wounds — on both sides.

When you look back on what’s happened to you in the past, like a good analyst, you’ll need to try to put yourself in the shoes of the people you dealt with, and understand their reasons, beyond any grudges or grievances. To do this, you’ll need to keep in mind a few factors that shape all of our behavior. The first, obviously, is the influence of the Age-Old Dichotomy, which, through its lackey Marketing, pushes people to satisfy their desires instead of their needs, creating flawed frameworks of values and leading them to forget that old age and death await everyone. The second factor to consider is human nature; remember:

Never attribute to malice what can be explained by stupidity.

Or, as De Santillana put it:

There is a good rule for historians that I have always kept in mind while working on my Galileo: never underestimate the power of foolishness and senselessness in human affairs.4

To settle matters concerning relationships between couples, though, the safest approach is the anthropological one. As Master Canaro used to say:

A hundred years of feminism cannot beat a hundred thousand years of evolution.

According to him, any anomalous behavior within a couple can be explained by keeping three principles in mind:

  1. women are talking incubators;
  2. men are sperm dispensers;
  3. every excess conceals an excess of the opposite kind, of equal size.

Paraphrasing that godless Dawkins, we are the hard drive of our genes, the mass storage that guarantees their persistence. Our software can vary, but the firmware5 that defines our low-level behavior has stayed unchanged for thousands of years, and drives us to do what we were created for, that is, to reproduce. Neither the cultural superstructures we’ve invented, nor the idols we prostrate ourselves before, nor even the Age-Old Dichotomy, can modify our ROM. Ignoring, or worse, rebelling against this state of affairs is the first step toward ruin, unhappiness, or both.

We live mostly on dry land, and so we consider that to be normal. But if you spend a long stretch of time on a ship, or on an island, you come to understand that normality is water, and that land is only an exception. In the same way, we place great importance on the cycle of birth, reproduction, and death that we call “life”, and see in it the ultimate purpose of the Universe, forgetting that life is just a particular case of existence, and that a universe made of rocks would still be a marvel. Every form of communal life, in order to survive, requires the legitimization of certain biological follies. Over time, many of these conventions, born out of contingent needs, inevitably end up clashing with the common sense of justice, and have to be either abolished or amended — only every amendment or adjustment, instead of making them more “just”, pushes them further away from their original purpose, and leaves them open to mistaken interpretations, or even ones opposite to what their original aim was.
