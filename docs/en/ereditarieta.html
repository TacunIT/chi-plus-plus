Inheritance

A parent class can disown a child class, if it turns out bad, but a child class cannot disown its parent class, because it owes it more than it could ever repay — and, in particular, owes it its own existence.

Inheritance, that is, the ability to create genealogies of classes, is C++’s main distinguishing feature.

As we saw in the previous lesson, a hypothetical Pesce (Fish) class would need different attributes depending on what it’s being used for. In a language like C, which doesn’t allow inheritance, you’d therefore have to set up two different data structures for each case:

enum Acqua  { dolce, salata };
enum Sesso  { maschio, femmina };
enum Colore { rosso, blu, verde, argento };

struct PesceAlimentare {
    Sesso  _sesso;
    char*  _specie;
    float  _prezzo;
    time_t _data_cattura;  
    int    _peso;
    bool   _commestibile;
    char*  _area_pesca;
    ...
};

struct  PesceAcquario {
    Sesso  _sesso;
    char*  _specie;
    float  _prezzo;
    time_t _data_acquisto;  
    Colore _colore;
    Acqua  _acqua;
    ...
};

and different functions to manage the data:

void gestisciPesceAlimentare(struct PesceAlimentare& pesce);
void gestisciPesceAcquario(struct PesceAcquario& pesce);

This means that if you’ve already written (and verified, fixed, and tested) a management program for a fishmonger, to turn it into a management program for selling aquarium fish, you’d have to rewrite (and re-verify, re-fix, and re-test) all the code, even though part of the data involved is the same. Inheritance, by contrast, lets you isolate the characteristics common to both contexts into a single class, and derive from this base class two specialized classes:

/** 
 * @file ereditarieta-singola.cpp
 * Esempio di ereditarietà singola.
 */
 
#include <iostream>
#include <string>
#include <ctime> 

using namespace std;

enum Acqua  { dolce, salata };
enum Sesso  { maschio, femmina };
enum Colore { rosso, blu, verde, argento };

class Pesce {
private:
            
    static int _n_pesci;
                        
protected:
    
    /** Dati membro privati */
    Sesso  _sesso;
    float  _prezzo;
    string _specie;
    bool   _commestibile;

public:

    /** Costruttore della classe inline */
    Pesce(Sesso sesso, float prezzo, const char* specie) 
    : _sesso(sesso), _prezzo(prezzo), _specie(specie)
    {
        _n_pesci++;
    }
    
    /** Distruttore inline */
    ~Pesce() { 
        _n_pesci--;
    }

    /** Funzioni di interfaccia */
    void setSesso(Sesso sesso);
    void setPrezzo(float prezzo);
    void setSpecie(const char* specie);

    /** Funzioni di interfaccia inline */
    string getSpecie() { return _specie.c_str(); }
    float  getPrezzo() { return _prezzo;  }
    Sesso  getSesso()  { return _sesso; }
    bool   getCommestibile() { return _commestibile; }
    void   setCommestibile(bool commestibile) { 
        _commestibile = commestibile; 
    }

};

int Pesce::_n_pesci = 0 ;

class PesceAlimentare : public Pesce {
private:
    
    /** Dati privati della classe */
    time_t _data_cattura;  
    string _area_pesca; 

public:

    /** Costruttore della classe inline */
    PesceAlimentare(Sesso sesso, float prezzo, const char* specie) 
    : Pesce(sesso, prezzo, specie)
    {
        /** Auspicabilmente.. */
        _commestibile = true;
    }

    //...
};

class PesceAcqario : public Pesce {
private:
    
    /** Dati privati della classe */
    time_t _data_acquisto;  
    Colore _colore;
    Acqua  _acqua;

public:
    
    /** Costruttore della classe inline */
    PesceAcqario(Sesso sesso, float prezzo, const char* specie) 
    : Pesce(sesso, prezzo, specie)
    {
    } 
    
    //...
};

This approach has two upsides: the first is that you won’t need to repeat the testing, debugging, and validation phases for the functions common to both systems, since they’ll already have been verified during the development of the first system; the second is that, by reusing part of the code, you’ll be able to spot and fix any bugs that slipped through the first round of testing, or improve the behavior of the shared functions, benefiting both systems.

In C++, inheritance can be either single or multiple, depending on whether the new class inherits its characteristics from one or more pre-existing classes:

class Figlio : public Mamma
{
    /** Ereditarietà singola */
};

class Figlio : public Mamma, private Papa
{
    /** Ereditarietà singola */
};

By contrast, it’s not allowed (nor would it make sense) for a class to inherit twice from the same base class:

class Errore : public Mamma, public Mamma
{
	...
};

The Mamma and Papa classes can be called base classes, subclasses, fundamental classes, or subtypes; the Figlio class can be called either the derived class, the superclass, or the supertype. The instruction:

class PesceAlimentare : public Pesce {
    ...
};

declares the PesceAlimentare class as derived from the Pesce class. The access specifier between the two class names defines the visibility of the base class’s data inside the child class:

class B : public A
{
    // tutti i membri di A mantengono
    // in B la loro visibilità originale
};

class B : protected A
{
    // tutti i membri public di A diventano membri
    // protected di B; i membri protected e private
    // mantengono la loro visibilità originale
};

class B : private A
{
    // tutti i membri di A, quale che sia la loro
    // visibilità, diventano membri private di B;
};

In the absence of an access qualifier, the compiler treats all the data of a class declared with the class keyword as private, and all the data of a class declared with the struct keyword as public:

/** 
 * @file ereditarieta-accesso.cpp
 * Accesso ai membri della classe base dalle classi derivate.
 */
 
/** Classe base, dati membro pubblici */ 
struct Mamma { 
    int a;
};

/** Classe derivata come struct: dati per default pubblici */
struct Figlia : Mamma {
    int b;
};

/** Classe derivata come class: dati per default privati */
class Figlio : Mamma {
    int c;
};

/** Funzione generica, non friend delle classi */
void funz(Figlia& figlia, Figlio& figlio)
{
    figlia.a++;
    figlio.a++;    
}

This is the declaration of the derived class PesceAlimentare’s constructor:

PesceAlimentare(Sesso sesso, float prezzo, const char* specie)
: Pesce(sesso, prezzo, specie) {
    _commestibile = true;
}

The second line is the class’s initialization list, and contains the constructors of the base classes. When you instantiate an object of a derived class, the system first calls the base classes’ constructors, then the child class’s own constructor. This way, the derived class’s constructor is guaranteed to be working on correctly initialized member data. Using the base class’s constructor to initialize shared data is necessary for two reasons: the first is that some of the base class’s data might be private, and therefore inaccessible to the derived class; the second is that this achieves low coupling between the base class and the derived class, so that if the internal implementation of the base class’s constructor ever needed to change (while keeping the interface unchanged), the code of its derived classes wouldn’t need to change too. The order in which the base classes’ constructors are called during the object’s initialization depends on the order in which they appear in the child class’s constructor. cout << "classe base: " << pesce1.Articolo::getTipo() << endl;

You can avoid this kind of problem by declaring the Persona class as a virtual base class of both Madre and Padre:

class Padre : public virtual Persona {
};

class Madre : virtual public Persona {
};

This way, the Figlio class will inherit all the members specific to the Madre and Padre classes, but only a single copy of the methods and attributes of the virtual Persona class that both of them contain.

As I’ve told you, defining a new class is equivalent to defining a new data type, which the compiler will treat the same way as the language’s primitive data types. This means, for example, that if we want, we can create an array of objects of class Pesce the same way we’d create an array of int or char:

/** Array di oggetti di classe Pesce */
Pesce acquario[10];

There’s only one limitation: since every element of an array must be initialized when it’s created, the class must have a default constructor. If we decided to create an array of objects of the Punto class we saw in the last lesson, the array’s elements would have to be initialized explicitly:

Punto spline[3] = { Punto(3,5), Punto(0,0), Punto(7,7) } ;

We can also add objects belonging to a base class’s derived classes into an array of objects of that base class:

Persona famiglia[3] = { Padre(), Madre(), Figlio() } ;

Conversely, base class objects cannot appear in an array of objects of a derived class, and an instruction like the one below will produce an error:

Figlio classe[4] = { Figlio(), Figlio(), Persona(), Figlio() } ;

Just as, in chess, the queen can move like a rook, but a rook cannot move like a queen, an object of type Persona doesn’t contain all the information of an object of type Figlio, and so cannot be used in its place. The same reasoning that applies to arrays also applies to pointers. A pointer to objects of type Persona can be assigned an object of type Figlio, while the reverse operation will cause a compile error:

Persona *ptrP = new Figlio() ;  // OK
Figlio  *ptrF = new Persona();  // ERRORE!

The compiler is able to understand the relationship between a derived class and its base class, and can therefore establish a coercion path from the object’s type to the pointer’s type, but it has no way of accessing the members or functions of a derived class from a base class object. We said earlier on that pointers are like masks that isolate certain sequences of bits, whose size varies depending on the pointer’s type. The same reasoning applies to classes too: a base-class pointer associated with a derived-class object will only “see” the data and functions of its own class:

/** 
 * @file ereditarieta-puntatori.cpp
 * Gestione dei puntatori a classi derivate.
 */
 
#include <iostream>
#include <string>

using namespace std;

class Persona {
public:
    void getClass(){
        cout << "Persona" << endl;
    }
};

class Madre : virtual public Persona {
public:
    void getClass(){
        cout << "Madre" << endl;
    }
};
 
int main(int argc, char** argv) 
{
    Madre   * ptrM = new Madre;
    Persona * ptrP = ptrM ;
    ptrM->getClass() ;
    ptrP->getClass() ;
    return 0;
}

If you compile and run this code, you get:

> g++ src/cpp/ereditarieta-puntatori.cpp -o src/out/esempio
> src/out/esempio

There’s no error: the getClass() function invoked in the second output instruction isn’t, as you might expect, that of the Madre class, to which the object belongs, but that of the base class, which is the only one the program can access through a pointer to objects of type Persona. This (correct) behavior of the program becomes particularly risky if the class has a destructor:

/** 
 * @file ereditarieta-distruttori.cpp
 * Gestione dei distruttori nelle classi derivate.
 */
 
#include <iostream>
#include <string>

using namespace std;

/** Definiamo una classe base */
class Padre {
public:
    ~Padre() ;
} ;

/** Definiamo delle classi derivate. */
class Figlio : public Padre {
public:
    ~Figlio() ;
};

class Nipote : public Figlio {
public:
    ~Nipote() ;
};

int main(int argc, char** argv) 
{
    /** Creiamo un array di puntatori di classe base. */
    Padre * dinastia[3] ;     

    /** Assegnamo delle istanze delle tre classi ai puntatori */
    dinastia[0]= new Padre  ; 
    dinastia[1]= new Figlio ;
    dinastia[2]= new Nipote ;

    /** Facciamo qualcosa con i nostri oggetti.. */

    /** ..e poi li eliminiamo */
    delete dinastia[0] ; 
    delete dinastia[1] ;  
    delete dinastia[2] ;  
}

No compiler will ever give you an error for this code, but the destructor called, in all three cases, will be that of the base class Padre, with consequences ranging from problematic to disastrous.

Virtual functions are functions that get called based on the class of the object they belong to, regardless of the type of the reference or pointer being used. This is made possible by a mechanism called dynamic binding or late binding, which consists of postponing the linking of functions until the program actually runs, unlike what happens with normal member functions, which are linked to the code at compile time — so-called early binding. In practice, it works like this: the addresses of all functions declared virtual are stored in an internal table, and only when one of these functions is called by the program does the system look up its address, performing the linking in real time. You can see for yourself that using virtual functions, besides causing a slight delay in the program’s execution time, since the function’s address has to be looked up, also uses up some of the system’s resources to store the address table — so, as with everything, it’s best not to overdo it. The rules governing the use of virtual functions are:

  • the versions of the functions in derived classes must have the same return type and the same parameters as the base class’s version: if not, the compiler treats the two functions as different, and the “virtual” effect is lost;

  • a virtual function cannot also be static: the very concept of a virtual function requires a link between an object and a function; static functions are independent of the objects of their class, so the two are incompatible;

  • a function can only be declared virtual in the base class: it’s not possible to make that declaration in a derived class;

  • you can repeat the virtual specifier in derived classes too, but it isn’t necessary: you can see this in the example, where the second version of the getClass() function doesn’t have the virtual keyword in front of it;

  • using the scope resolution operator inevitably cancels out the effect of virtual functions.

It’s possible to declare a virtual function in the base class without defining its behavior, using the syntax:

virtual <tipo> nomefunzione([argomenti]) = 0 ;

This kind of function is called a pure virtual function, and it makes the class it belongs to an abstract class. Abstract classes are generic classes that can be used as ancestors for a lineage of specialized classes, but that cannot be used directly. The rules that apply to abstract classes are:

  • any class with at least one pure virtual function is considered abstract;
  • pure virtual functions are inherited by derived classes as pure virtual functions, so, if a derived class doesn’t redefine a pure virtual function of its base class, the compiler will treat it as an abstract class too;
  • abstract classes cannot be used as function arguments or return types;
  • abstract classes cannot be the type of an object or of an explicit conversion.

Given an abstract class Mammifero, the following instructions would cause compile errors:

void funz(Mammifero m);  // non possono essere argomenti..
Mammifero funz();        // né valori di ritorno..
punt = (Mammifero*)ptr;  // né il tipo di una conversione.
Mammifero pollo;         // né il tipo di un oggetto

It’s possible, though, to declare a pointer or a reference to an abstract class, and use them to create arrays or queues that can be used with instances of different classes:

/** 
 * @file ereditarieta-classi-astratte.cpp
 * Gestione delle classi astratte.
 */
 
#include <iostream>
#include <string>

using namespace std;

enum Sesso { maschio, femmina, boh};
const char* Sessi[] = {"maschio", "femmina", "boh" };

class Mammifero
{
private:

    Sesso _sesso;

public:

    Mammifero(Sesso sesso = boh) 
    : _sesso(sesso) {
    }

    virtual void getSesso() {
        cout << Sessi[_sesso] << endl ;
    }    

    /** Funzione virtuale pura */
    virtual void getSpecie() = 0;
} ;

class Cane : public Mammifero
{ 
private:

    string _nome;

public:

    Cane(const char * nome, Sesso sesso) 
    : Mammifero(sesso), _nome(nome){
    }

    void getNome() {
        cout << _nome << endl ;
    }

    /** Definizione della funzione virtuale */
    virtual void getSpecie() {
        cout << "cane" << endl ;
    }    
};

int main()
{
    Cane mioCane("Scylla", femmina) ;
    Mammifero& cane = mioCane; 

    /** 
     * Accede alle funzioni della classe Cane
     * dall'istanza della classe.
     */
    mioCane.getSpecie();
    mioCane.getSesso();
    mioCane.getNome();
    
    /** 
     * Accede alle funzioni della classe Cane
     * dal puntatore alla classe Mammifero.
     */
    cane.getSpecie();
    cane.getSesso();
    
    return 0;
}

The Mammifero base class defines only an abstraction, leaving it to its derived classes to define attributes and methods specific to each particular species. In the same way, the getSpecie function defines only a concept, not an algorithm; it’s up to the individual derived classes to redefine the function’s behavior, adapting it to their own needs. It is possible, however, to also define a behavior for pure virtual functions; for the Mammifero class it might look something like this:

inline void Mammifero::getSpecie()
{
    cout << "nessuna" << endl ;
}

Since objects of class Mammifero cannot exist, though, the base version of the getSpecie function could only be called by making use of the :: operator:

mioCane.Mammifero::getSpecie();
cane.Mammifero::getSpecie();

The teaching value of this code, though, is unchanged. The Karma of the two objects is determined by their classes’ declaration, which leaves them no choice but to do what they were created to do. The output of the functions, the style of the comments, or the fact that in one case the classes are called A and B, while in the other they’re called Persona and Madre, are just formal differences that don’t affect the example’s Dharma, which is to illustrate the behavior of virtual functions. If, instead of the new code, I had used Master Canaro’s original, you would have understood it just as well; maybe even better, because the new code seems better to me, but that doesn’t mean it is to you too. The same principle also applies to the Universe. Just as objects within the same program occupy different positions in memory depending on when the program is run, sentient beings can manifest in different places and times across different cycles of existence. Each of us is an orchestra player who has been handed a score. We can play it more or less well, or not play it at all, hiding within the swell of the orchestra, but our worth is only — forgive the pun — instrumental, because what matters isn’t us: it’s the music; and this isn’t the first, nor will it be the last, time we play it. I, here, now, with my nose, my mouth, and my hair, am teaching you what I know about C’hi++, and you, with your own nose, your own mouth, and your own hair, are writing it down in your book, but the very same information we’re passing on and perpetuating could be passed on and perpetuated by entirely different people, at another time, or on another Planet, in another cycle of the Universe. In all likelihood we’ve already had this exact conversation in the past, and we’ll have it again in the future. Perhaps we won’t use the same words; perhaps we’ll have different names, perhaps you’ll be the master and I the disciple, but our friendship will be the same, because that’s part of our class’s declaration; it cannot, and must not, change.

Mythical figures, it is true, are born and pass away, but not quite as we mortals do. They need characteristic names, like that of “The Once and Future King”. Did they exist in the past? Then they existed even earlier, or will exist again, under other names, in other guises, just as the sky brings back its configurations to us eternally. If one tried to define them precisely as persons and things, they would surely vanish before our eyes, like the fruits of a sick fancy. But if their true nature is respected, they will reveal that nature as functions. 1

Every age has its heroes and its demons; the memory of their battles gives rise to myth. Science is fleeting: we know little about the scientific knowledge of past peoples, while we know their myths well, because myth is immortal; Science isn’t, unless it’s absorbed by myth and turned into legend or superstition. Master Canaro, for instance, was convinced that the superstition around the numbers 13 and 17 arose from observing the behavior of locusts, which, depending on the species, spend either thirteen or seventeen years underground in a larval stage, then all emerge together and spend their short lives devastating crops. The fear of the numbers 13 and 17, in his view, was scientific knowledge handed down across time and space as superstition, by farming cultures that periodically watched their harvests get destroyed.

Art is the engine of myth. A self-fueling engine, like the Sun, because it feeds on epic events and in turn generates them, inspiring heroes to deeds worth remembering. The word art, like love for that matter, is one of those words people use often but never precisely define, because they’re considered self-explanatory concepts that need no defining. That’s wrong: as we’ve seen, all words, even the most common ones, can be interpreted in different ways. In an essay he wrote in his youth, on Love and Art, Master Canaro wrote that:

Art is the trace of Man’s journey toward God

later specifying that, by the term “God” (another word each of us interprets differently), he meant the meaning of Existence. Some years later, though, looking at photos of tarts made by chef Gianluca Fusto,2 he realized his definition was imperfect, because it didn’t include, or at least left on the margins, those art-objects that didn’t fall within the canonical artistic categories. He therefore changed his definition of Art to:

Art is the trace of Man’s journey toward Perfection

This wasn’t a contradiction, but a clarification, since for him — as for us, for that matter — the pursuit of perfection was, in fact, the meaning of Existence, and therefore, God. This episode from Master Canaro’s life has the peculiarity of demonstrating the very principles it states: the first is that we shouldn’t be afraid to question our own ideas, if we realize they’re wrong or incomplete; the second is that, if we approach life the right way, everything we do will be Art, not just flower arranging or the Tea ceremony. In life there are no first-class moments, when we do the things we enjoy, and second-class moments, when we do what needs to be done: every instant matters. To underscore this precept, Master Canaro defined an aesthetic for hanging laundry on a drying rack, and, so as not to risk being taken too seriously, called it Ikebarba. The principles of Ikebarba, as Master Canaro formulated them, are:

Ikebarba is made for man, not man for Ikebarba.3 Ikebarba should not be a burden for whoever practices it, but a joyful duty. The laundry has to be hung out to dry regardless; the time it takes to do so sloppily or mindfully is more or less the same, but a well-made Ikebarba will produce dry laundry in less time, and will make the sight of the drying rack less annoying.

Ikebarba begins in the shop.
Garments of different colors, or in loud colors, are hard to match chromatically; it's therefore best to buy clothes in muted tones, ideally matching one another, to make composing them on the drying rack easier and more elegant. Be careful, though: a military- or Maoist-style wardrobe, with identical garments all the same color, is an excess to avoid, because it would make the composition (and not only that) monotonous.

Ikebarba shuns clothespins. Clothespins are as useful as psychoanalysis: it's balance that should keep the laundry on the line, not an external coercive force. The various garments should be arranged on the line so that the weight on one side balances the weight on the other.

There are three kinds of Ikebarba:

  • chromatic: when the laundry is arranged on the rack by color;
  • functional: when the garments are arranged according to their respective drying times, placing heavy items on the outside, where they get more air, and lighter or synthetic ones on the inside;
  • perfect: when the aesthetic and functional aspects merge into a harmonious whole.

As you can probably guess, Master Canaro’s intentions were largely ironic (he confessed to me that the discipline’s first formulation was born as an attempt to disguise his obsession with tidiness), and for a long time he was unsure whether to include it in the body of the Doctrine. He decided to do so once he realized his own reservations stemmed from exactly the kind of prejudice Ikebarba was meant to counter. As Banzan and Paul Simon teach us, Truth is everywhere, if we know how to look for it, even on subway walls or in market stalls. In any case, Ikebarba can genuinely benefit those who practice it. First of all, it tempers the harmful effect of the Age-Old Dichotomy and its servants Marketing and Fashion, which push us to buy clothes we don’t need and will only use for a short time. By reducing the chromatic variation of the laundry, it also reduces the number of weekly washes, and with it the need for electricity, water, detergents, and plastic. Even the rejection of clothespins has a functional value of its own: if garments are hung draped over the lines, the water in the fabric will tend to run down both sides, reducing drying time. Admittedly, the fact that our Order’s Rule requires us to wear white shirts doesn’t solve the pollution problem, but, as Master Canaro used to say:

Fur is made of hair

and until our species learns to make more responsible use of its gonads, all we can do is compensate as best we can for the problems tied to overpopulation.
