Streams

You can never dip your bytes in the same stream twice

Today I’ll tell you about streams, which, as is well known, are C++’s most important component.

C++ inherits from C the lack of dedicated keywords for I/O management. Instead of instructions like BASIC’s print, it uses libraries of classes and functions that let you convert the objects a program manages into printable text, or convert text elements into objects. It couldn’t be otherwise: C++ doesn’t just have to handle strings and numbers, like BASIC, but also floating-point numbers, pointers, and above all user-defined data types, for which it wouldn’t be possible to define a standard behavior, and which would therefore need to be handled differently from primitive data — so much for the consistency of the language. Besides being able to make use of C’s function libraries, C++ has its own I/O library, based on the stream class hierarchy, which lets you handle user-defined data types too. We’ve already seen examples of this feature when we talked about polymorphism and operator overloading:

ostream& operator << (ostream& os, const Animale& animale) {
    os  << "Specie:" << animale.getSpecie() << "\t"
        << "Razza:"  << animale.getRazza()  << "\t"
        << "Sesso:"  << animale.getSesso()  
        << endl;
    return os;   
}

This code “teaches” the << operator how to behave when displaying an object of class Animale. The same can be done (and we did it) for any other user-defined type. It’s the language’s syntax that adapts to the programmer’s needs, not the other way around. A few key concepts for understanding streams are:

  • a stream is an abstraction representing the source or destination of a set of data of variable length: keyboard input, screen output, memory buffers, strings, files;
  • output to a stream, toward any destination, is called writing or insertion, and is performed by means of the << operator;
  • the terms reading or extraction, instead, refer to the operation of acquiring data from a source, performed by the >> operator.

C++’s iostream library lets you handle stream I/O operations by means of classes derived from two base classes: streambuf and iosbase. The library has two different “family lines”: one meant for handling single-byte characters, and one meant for multi-byte characters. The classes of the multi-byte library have the same name as the ordinary classes, with the prefix “w” added. This is the inheritance layout of the iostream library’s classes:

                              ios_base
                                 |
                             basic_ios

               __________________|_________________       
              |                                    |
              |                                    |
        basic_istream                         basic_ostream
             | |                                  | |   
             | |                                  | |  
             | |__________________________________| |  
             |                   |                  |

                            basic_iostream
                         ________|________
                        |                 |
               basic_stringstream    basic_fstream

Except for ios_base, these are all template classes, later instantiated with different parameters to handle the character types char and wchar_t. For example, the ostream class is a specialization of the basic_ostream class:

typedef basic_ostream<char> ostream;

Its multi-byte counterpart is the wostream class:

typedef basic_ostream<wchar_t> wostream;

The basic_ostream template class, in turn, derives from basic_ios:

template<
    class CharT,
    class Traits = std::char_traits<CharT>
> class basic_ostream
: virtual public std::basic_ios<CharT, Traits>

which, in turn, derives from ios_base:

template<
    class CharT,
    class Traits = std::char_traits<CharT>
> class basic_ios
: public std::ios_base

In short: if the name is prefixed with basic_, it’s the template class; if it has a “w” in it, it’s the multi-byte version; otherwise it’s the ordinary class. Besides the classes derived from iosbase, the library also includes classes for managing data buffers:

                       basic_streambuf
                      ________|________
                     |                 |
               basic_stringbuf     basic_filebuf

The virtual template class basic_streambuf, which is part of the library but not of the ios_base lineage, holds the data and functions needed to manage a character buffer. Its derived classes basic_stringbuf and basic_filebuf are instead specialized in handling in-memory and on-file buffers, respectively. Here too, the library includes two versions of each class, specialized for handling char and wchar_t.

typedef streambuf  basic_streambuf<char>
typedef wstreambuf basic_streambuf<wchar_t>
typedef filebuf    basic_filebuf<char>
typedef wfilebuf   basic_filebuf<wchar_t>

As you’ve probably guessed, going through the individual classes of the iostream library is an activity that rivals classical epigraphy for sheer tedium, but it’ll let us see a whole series of principles we discussed in previous lessons put into practice, so, let’s brace ourselves and get started.

The ios_base class and its first descendant, basic_ios, are generic classes that provide the basic functions for managing streams, regardless of whether they’re input or output streams. A peculiarity of ios_base is that it has no public constructor, so it cannot be used to create objects — only as a base for derived classes. The specialized instances of basic_ios are:

typedef basic_ios<char>    ios;
typedef basic_ios<wchar_t> wios;

Through the methods of these classes, you can check or change a stream’s internal state, its formatting, or define callback functions for handling data. The openmode member data, for example, defines how the stream should be opened:

app Makes every output operation happen at the end of the stream.
ate On opening the stream, moves the insertion point to the end (at end) of the I/O buffer.
binary Treats the stream’s content as a binary data stream.
in Allows input operations.
out Allows output operations.
trunc Clears the stream’s content when opened.

The iostate member data, which we’ll use in one of the next examples, holds information about the stream’s current state:

goodbit No error
eofbit The end of the stream has been reached.
failbit The last I/O operation failed.
badbit The last I/O operation was invalid.
hardfail An unrecoverable error occurred.

Both of these member data items are bitmasks,1 so they can hold more than one value at once. The following instruction, for example, opens a file stream by OR-combining three possible openmode values:

fstream file_io("io.txt"
               , ios_base::in | ios_base::out | ios_base::app);

After basic_ios, the library’s classes specialize into input or output: on one side, basic_istream, from which the two standard input streams cin and wcin derive; on the other, basic_ostream, from which the standard output streams cout, cerr, clog, and their “wide” counterparts wcout, wcerr, wclog, derive. From these two generic classes derive template classes specialized in input or output to file or memory:

template
<class Elem, class Tr = char_traits<Elem>>
class basic_ifstream
: public basic_istream<Elem, Tr>

template
<class Elem, class Tr = char_traits<Elem>, class Alloc = allocator<Elem>>
class basic_istringstream
: public basic_istream<Elem, Tr>

template
<class Elem, class Tr = char_traits<Elem>>
class basic_ofstream
: public basic_ostream<Elem, Tr>

template
<class Elem, class Tr = char_traits<Elem>, class Alloc = allocator<Elem>>
class basic_ostringstream
: public basic_ostream<Elem, Tr>

and a class able to handle both operations:

template
<class Elem, class Tr = char_traits<Elem>>
class basic_iostream
: public basic_istream<Elem, Tr>
, public basic_ostream<Elem, Tr>

this one too, with two specializations for handling files and memory:

template
<class Elem, class Tr = char_traits<Elem>, class Alloc = allocator<Elem>>
class basic_stringstream
: public basic_iostream<Elem, Tr>

template
<class Elem, class Tr = char_traits<Elem>>
class basic_fstream
: public basic_iostream<Elem, Tr>

cout << "Hello World!"  // stringhe
     << 12              // interi
     << 0.35            // float
     << argv[1]         // puntatori
     << endl;

We haven’t yet talked about stream input, though it works in a fairly similar way:

/**

 * Programma di esempio per la gestione dell'input da stream.
 */
 
#include <iostream>

using namespace std;

int main(int argc, char** argv)
{    
    string stringa;
    
    /** Scrive un messaggio sullo schermo */
    cout << "Inserire una stringa: ";
    
    /** Legge una stringa da tastiera */
    cin >> stringa;

    /** La scrive sullo schermo */
    cout << stringa << endl;

    return 0;

As you can see, read operations with the >> operator stop at the first whitespace character; so, if we want to read the whole string, we need to change the code:

/**

#include <iostream>

using namespace std;

int main(int argc, char** argv)
{

    string stringa;
    
    /** 
    *   Legge una stringa da tastiera tramite
    *   la funzione membro della classe istream.
    */
    cout << "Inserire una stringa: ";
    cin.getline(buffer, 256);
    cout << buffer << endl;


    /** 
    *   Legge una stringa da tastiera tramite
    *   la funzione globale getline.
    */
    cout << "Inserire una stringa: ";
    getline(cin, stringa);
    cout << stringa << endl;

    return 0;
}

Another feature of the << and >> operators is that their precedence is lower than almost all other operators, which lets you write instructions like this one:

cout << "Due più due fa: " << 2 + 2 << '\n' ;

The logical AND operator |, the inclusive OR operator &, and the exclusive XOR operator ^ have lower precedence than the << and >> operators, and, if not enclosed in parentheses, operations involving them can cause bugs. For example, in an instruction like the following, the & operator would be interpreted as a reference to an object, with consequences different from those expected:

cout << "Il valore è: " << 2 & 2 << '\n' ;	// ERRORE!

the correct syntax, instead, is:

cout << "Il valore è: " << (2 & 2) << '\n' ;	// OK

The default behavior of stream input operators also includes some formatting conventions:

  • the base conversion format is decimal;
  • the fill character is a space;
  • the precision of floating-point digits is the same used by printf(), rounding at the sixth decimal digit;
  • the default field width is 0, which means the output stream will use as many characters as needed to display the entire value or string.

The first three changes are permanent: once set, the new values remain in effect until another instruction changes them again; changes to the input field width, on the other hand, only apply to the instruction that requests them. In some earlier examples we saw that it’s possible to change a stream’s default output format by means of manipulators:

cout << setfill('0') << setw(2) << ora._h << ":"
     << setfill('0') << setw(2) << ora._m << ":"
     << setfill('0') << setw(2) << ora._s << endl;

The same result can be achieved by means of dedicated functions of the ios_base and basic_ios classes, which let you change the fill character, the precision of decimal digits, and the field width:

/** 
 * @file src/stream-format.cpp
 * Formattazione dell'I/O con gli stream.
 */
 
#include <iostream>
#include <iomanip>

using namespace std;


int main(int argc, char** argv)
{    
    double d = 123.456789 ;

    /** Mostra i valori di default */
    cout << endl;
    cout << "Valori di default" << endl;
    cout << "  width:     "  << cout.width()        << endl;
    cout << "  precision: "  << cout.precision()    << endl; 	
    cout << "  fill:      '" << cout.fill()  << "'" << endl;	
    cout << "  output:    "  << d                   << endl;

    /** Modifica il formato e mostra il nuovo output */
    cout << endl;
    cout << "Modifica formato" << endl;
    cout << "  output:    "  ;
    cout.precision(4) ;
    cout.fill('#') ;
    cout.width(10) ;
    cout << d << endl;
    
    /** Mostra la persistenza dei valori */
    cout << endl;
    cout << "Valori correnti" << endl;
    cout << "  width:     "  << cout.width()        << endl;
    cout << "  precision: "  << cout.precision()    << endl; 	
    cout << "  fill:      '" << cout.fill()  << "'" << endl;	
    cout << "  output:    "  << d                   << endl;

    return 0;
}

When a stream read or write operation fails, the value of the iostate member data becomes non-zero. The basic_ios class has boolean member functions that return true or false depending on whether the iostate value indicates a given event, and the rdstate function, which returns iostate’s absolute value.

good no error: the value of iostate is 0
eof the end of the file has been reached
fail a non-blocking I/O error occurred
bad a blocking I/O error occurred
rdstate returns the current value of iostate

These functions let you stop reading or writing a stream when an error occurs, or when the end of the file has been reached. One thing you should never do, though, is use the eof function inside a while loop to read a file:

/** 
 * @file src/stream-eof.cpp
 * Gestione dell'I/O su file con gli stream.
 */
 
#include <iostream>
#include <fstream>

using namespace std;


int main(int argc, char** argv)
{    
    ifstream testo;
    int      n;
    
    /** apre il file in lettura */
    testo.open(argv[1]);
    
    /** Verifica che il file sia aperto */
    if(testo.is_open()) {
        
        /** Utilizza eof per gestire il ciclo */
        while(!testo.eof()) {

            /** Legge un numero dal file */
            testo >> n;

            /** Lo scrive a video */
            cout << n << endl;
        }
        
    }
        
    /** Chiude il file di input */
    testo.close();
    
    return 0;
}

If you have this program read a file containing the numbers 10, 20, and 30, you’ll get this output:

> g++ src/cpp/stream-eof.cpp -o src/out/esempio
> src/out/esempio src/cpp/stream-eof.txt       
10
20
30
30

The bug happens because the eof check runs before the fourth read operation, when the stream is still in the good state. A better way to handle these cases is to use the good function, which also lets you check that the file was opened correctly:

/** 
 * @file src/stream-good.cpp
 * Gestione dell'I/O su file con gli stream.
 */
 
#include <iostream>
#include <fstream>

using namespace std;


int main(int argc, char** argv)
{    
    ifstream testo;
    int      n;
    
    /** apre il file in lettura */
    testo.open(argv[1]);
    
    /** 
    *   Il ciclo si ripete fino a che non 
    *   si verifica un errore
    */
    while(testo.good()) {

        /** Legge un numero dal file */
        testo >> n;

        /** Si interrompe se il file è finito */
        if(testo.eof()) break;
        
        /** Altrimenti, scrive il numero */
        cout << n << endl;
    }
                
    /** Chiude il file di input */
    testo.close();
    
    return 0;
}

If you compile and run this program, you get the correct result:

> g++ src/cpp/stream-good.cpp -o src/out/esempio
> src/out/esempio src/cpp/stream-eof.txt        
10
20
30

Exceptions let you handle errors that occur while the program is running. When something abnormal happens, the system throws an exception, that is, it transfers control of the process from the current function to specific blocks of instructions called exception handlers. For this to happen, the code that generates the error must be enclosed in a try/catch block:

try {

    // codice che potrebbe dare errore

} catch (...) {

    //  istruzioni per la gestione dell'errore
}

Exceptions can be thrown and handled either by specific code within the program, or by C++’s automatic mechanisms:

/** 
 * @file src/stream-eccezioni-1.cpp
 * Programma di esempio per la gestione delle eccezioni.
 */
 
#include <iostream>
#include <fstream>

using namespace std;

int main(int argc, char** argv)
{    
    ifstream testo;

    /** 
    *   Fa sì che, se sia generata un'eccezione 
    *   in caso di errore nella gestione del file.
    */
    testo.exceptions ( ios_base::failbit );

    /** Questa istruzione genererà un'eccezione */
    testo.open("fileinesistente.txt");
    
    testo.close();
    
    return 0;
}

If you compile and run this code, you’ll trigger an error which, not being handled by the program, is handled by C++’s standard function:

> g++ src/cpp/stream-eccezioni-1.cpp -o src/out/esempio
> src/out/esempio
libc++abi: terminating with uncaught exception of type
std::__1::ios_base::failure: ios_base::clear
: unspecified iostream_category error
zsh: abort      src/out/esempio

If instead we put the code that opens the file inside a try/catch block, and define a handler for file-opening errors, the result will be more controlled:

/** 
 * @file src/stream-eccezioni-2.cpp
 * Programma di esempio per la gestione delle eccezioni.
 */
 
#include <iostream>
#include <fstream>

using namespace std;


int main(int argc, char** argv)
{    
    ifstream testo;

    try {
        
        testo.exceptions ( ios_base::failbit );
        testo.open("fileinesistente.txt");        
        testo.close();
        
    } catch(ios_base::failure e) {
     
        cerr << "Errore in esecuzione" << endl;      
   
    }    
   
    return 0;
}


> g++ src/cpp/stream-eccezioni-2.cpp -o src/out/esempio
> src/out/esempio                                      
Errore in esecuzione

We can even prevent file-opening errors altogether, by having the program itself throw an exception if it notices the filename is missing from the startup parameters:

/** 
 * @file src/stream-eccezioni-3.cpp
 * Programma di esempio per la gestione delle eccezioni.
 */
 
#include <iostream>
#include <fstream>

using namespace std;


int main(int argc, char** argv)
{    
    ifstream testo;

    try {
        
        /** Verifica che esista il nome del file da aprire */
        if(argc < 2)
            throw "Nome file mancante";
            
        testo.exceptions ( ios_base::failbit );
        testo.open(argv[1]);        
        testo.close();
        
    } catch(ios_base::failure e) {
     
        cerr << "Errore in esecuzione" << endl;      
   
    } catch(char const* msg) {
        
        cerr << msg << endl;
    }
   
    return 0;
}

This way, the program is able to handle every possible runtime error:

> src/out/esempio

> src/out/esempio nomefile.txt

The C++ standard library has a dedicated class for handling exceptions:

class exception {
public:
    exception () throw();
    exception (const exception&) throw();
    exception& operator= (const exception&) throw();
    virtual ~exception() throw();
    virtual const char* what() const throw();
};

By defining a class derived from exception, with additional member data and a specialized what function, you can handle error reporting in a more structured way. That’s what we’ll do in the next example.

/** 
 * @file src/stream-eccezioni-4.cpp
 * Programma di esempio per la gestione delle eccezioni.
 * Richiede, in input, il numero di caratteri da leggere 
 * e il path del file di input:
 *
 *    src/out/esempio <n caratteri da leggere> <file di input>
 */
 
#include <iostream>
#include <fstream>
#include <exception>

using namespace std;

/** Codici e stringhe di errore */
#define ERR_NONE          0
#define ERR_PARAMETRI   -10
#define ERR_FILE_OPEN   -20
#define S_SINTASSI      "USO: esempio <n caratteri> <path file>"
#define S_ERR_PARAMETRI "Errore nei parametri di input."
#define S_ERR_FILE_OPEN "Impossibile aprire il file di input"

/**
*   Definisce una classe derivata da exception
*   per la gestione degli errori.
*/
class Eccezione: public exception
{
private:
    int         _codice;
    const char* _errore;
public:

    /** Costruttore */
    Eccezione(int codice, const char* errore) 
    : _codice(codice), _errore(errore) {        
    }
    
    /** Funzione virtuale pura: va ridefinita */
    virtual const char* what() const throw() {
        return _errore;
    }

    /** Funzioni di interfaccia */
    int getCodice() { return _codice; }
    const char* getErrore() { return _errore; }

    /** Ridefinizione dell'operatore di output */
    friend ostream& operator<< (ostream& os, Eccezione e){
        os << e._codice << ": " << e._errore << endl;
        return os;
    }
};

int main(int argc, char** argv)
{    
    ifstream testo;
    testo.exceptions ( ios_base::badbit );

    try {

        char c     = 0;
        int  letti = 0;
                    
        /** 
        *   Verifica che ci siano sia il nome del file di input 
        *   che il numero di caratteri da leggere.
        */
        if (argc < 3) 
            throw Eccezione(ERR_PARAMETRI, S_ERR_PARAMETRI);
            
        /** Definisce il numero di caratteri da leggere */
        int da_leggere = atoi(argv[1]);
        
        /** 
        *   Imposta la exception mask dello stream per fare
        *   sì che un errore di I/O generi un'eccezione,
        *   poi apre il file in lettura.
        *   Usa un blocco try/catch per intercettare una
        *   eventuale eccezione e gestirla in maniera
        *   omogenea al resto del codice.
        */
        try {
            testo.exceptions ( ios_base::badbit 
                             | ios_base::failbit );
            testo.open(argv[2]);            
        } catch(ifstream::failure e) {
            throw Eccezione(ERR_FILE_OPEN, S_ERR_FILE_OPEN);      
        }

        /**
        *   Re-imposta la exception mask per evitare
        *   eccezioni a fine file.
        */
        testo.exceptions ( ios_base::goodbit);

        /** 
        *   Legge il testo e lo stampa a video 
        *   Se è stato definito un numero massimo di 
        *   caratteri, si ferma lì.
        */
        while(testo.good()) {
            if((c = testo.get()) != EOF) {     
                letti++;
                cout << c;
            } 
            if((da_leggere != 0) && (letti >= da_leggere)) {
                cout << endl;
                break;
            }
        } 
                
        /** Chiude il file di input */
        testo.close();
            
    } catch (Eccezione e) {

        /** Stampa a video l'eccezione */
        cerr << e << endl;

        /** Mostra la sintassi di chiamata **/
        cerr << S_SINTASSI << endl;
        
        /** Esce con un codice di errore */
        exit(e.getCodice());
    }
    
    return 0;
}

This alone, though, won’t be enough to give meaning to our existences, because, without persistence, our actions will be like objects of class streambuf: once the computer is turned off, they will no longer exist, and everything we’ve done, right or wrong, will have no influence on what happens next. The only thing that can save us from oblivion and meaninglessness is the Cookies, persistence. To give meaning to our existence, we need a hard drive on which to save the streams of our lives, so that each cycle of existence can draw on the treasure of past experiences. Without it, Hitler would be worth as much as Gandhi, and Albert Schweitzer as much as Ted Bundy. It has to be a metaphysical hard drive, though, to escape the Big Crunch, and this brings us back to the importance of transcendence. Looking back, over the half-century I have direct visibility into, I see a long series of ideological failures. The ‘68 movement swept away parts of our Society that were certainly in need of revision, but fundamental all the same, without giving us anything back in return, except the miniskirt. The armed struggles of the seventies shot the wrong people, while Reaganomics collapsed at the end of the ’80s along with the Berlin Wall. The hopes of the ’90s crashed into the Twin Towers on September 11, 2001, and even the Internet, which was originally meant to be a means of giving everyone a chance to express their ideas, gradually turned into a system of mass control and disinformation. What little was left of our values and our ideas was annihilated by smartphones and social media. There’s a common factor among the ideologies of the last fifty years that accelerated their obsolescence, and made them unable to survive their first defeat: their more or less fierce rejection of any non-instrumental form of spirituality. If you define a framework of values while at the same time denying any form of transcendence, you’re forced to look for the values and motivations of your own ethics within that same framework. You can do it, but it’s wrong, and limiting. It’s wrong, because the rules you define are always a consequence of contingent needs (wars, oppression, social inequality), and once those are gone, the system’s logical framework loses its meaning and falls apart. It’s limiting, because it narrows down the number of possible goals to pursue to a finite set of actions or milestones, beyond which there’s no longer any possibility of improvement. Think of the game of Chess: there’s no reason, within the chessboard itself, that forces each piece to move only in one specific way. Rooks move horizontally, bishops move diagonally, and the knight jumps in an “L”-shaped path, in accordance with rules defined outside the chessboard,2 but it’s precisely these limitations that give the game its charm. Modern Society, by contrast, is a chessboard where every piece moves however it likes, because, in the name of a misguided sense of freedom, every rule has been done away with. Successful people who take their own lives are pawns who, having reached the far end of the board through their own skill, discovered there’s no such thing as promotion anymore, because along with the rules, the players themselves have been abolished too.
