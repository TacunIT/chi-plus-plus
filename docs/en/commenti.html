Comments

The Compiler and the Linker show no mercy; they treat the ten thousand objects as straw dogs

Comments are the single most important thing when programming.

A comment is a short piece of text that describes, in plain language, the workings or purpose of the code it refers to:

/**
 * Funzione per il raddoppio dello stipendio.
 * Richiede come parametro il valore dello
 * stipendio corrente e ne torna il valore
 * duplicato.
 */
long raddoppia(long stipendio)
{
    return stipendio * 2;   //  raddoppia il valore
}

Even though it sits inside the code, a comment is ignored at compile time, because it’s an addition useful only to the programmer. The processor doesn’t care to know what the code it’s executing is for: it just executes it, without getting distracted by functional — or, worse, ethical — implications that would slow down its response time. It’s important to comment your code well. Good code produces programs that last over time, and, over time, one tends to forget. Even if you’re lucky enough to have a good memory, you have to consider the possibility that your code will be used by a colleague. Not only is it rude to force them to analyze every single line of code to understand how it works, it’s also unproductive. Just like a shortage, an excess of comments is also a mistake to avoid. The inexperienced programmer doesn’t comment their code because they think it’s a waste of time. This is truly an irresponsible attitude. The overzealous novice fills their code with comments, either because they assume whoever reads it will be less experienced than they are, or because they want to show off their limited knowledge. The experienced programmer only comments what isn’t self-evident, and points out the possible weak spots of their program so that whoever ends up reusing or fixing it pays due attention to them. The perfect programmer doesn’t comment their code at all, because their code is self-explanatory:

typedef Importo unsigned long;

Importo raddoppiaStipendio(Importo stipendioCorrente)
{
    return stipendioCorrente * 2;   
}

This is the same piece of code we saw before, but this time comments aren’t needed, because the function’s intent is expressed directly in the code, by making explicit the name of the function, of its parameter, and of its return value. The typedef instruction, which you’ve also seen in previous examples, is used to define new data types as aliases for existing data types. The instruction:

typedef time_t Data;

defines the data type Data as an alias for the standard data type time_t, while the instruction:

typedef Importo unsigned long;

defines the data type Importo as an alias for the standard data type unsigned long. Defining your own data types not only makes the code more readable, it also makes it easier to modify. The original function uses values of type unsigned long, that is, 4-byte sequences that allow storing integers from 0 to 4,294,967,295 (232). This data type can be used for a salary with no decimal digits, like Master Canaro’s back in the last century, but if we had to double a salary with decimal digits, we’d have to use data types like float or double and, without a typedef, we’d be forced to change both the return value and the parameter type of the function:

float raddoppia(float stipendio)
{
    return stipendio * 2;   //  raddoppia il valore
}

Whereas if we’ve defined our own data type, we only need to modify the typedef instruction, leaving everything else unchanged:

typedef Importo float;

Importo raddoppiaStipendio(Importo stipendioCorrente)
{
    return stipendioCorrente * 2;   
}

It’s acceptable for comments to have a touch of humor — it can help release some of the tension for the reader, the same way cracking a joke while working does, but even here you shouldn’t overdo it.

C++ provides two distinct ways to comment code: two adjacent // characters, or the /* */ sequence, which the Language borrowed from its predecessor, C. In the first case, the compiler ignores everything from that symbol to the end of the line; in the second case, it ignores everything between the two character sequences.

/**
 * Questa sintassi è più comoda per i commenti
 * che si estendono su più di una riga
 */
for (int i = 0; i < 10; i++) {
    cout << i << endl; // questa, per i commenti su una riga
}

Both solutions have pros and cons that become clear with experience. Choosing one over the other is often a matter of convenience, other times a matter of style, but if you understand the true essence of the Language, it’s just a matter of rhythm.

Comments are a bit like literature: writing too little is bad; writing too much, dwelling on needless descriptions, is just as wrong. The comparison also holds in reverse: books, works of literature, are the comments on the code of life. You write down what you wish to remember. Master Canaro used to say that the hardest thing about writing a manual is that, sometimes, to explain a certain concept, you first need to explain another concept, which in turn can’t be understood unless you understand the first. This is one of those moments. The equivalent of comments, in C’hi++, is what Master Canaro called Cookies: a kind of mass storage of the Universe, in which is saved the memory of the choices made by sentient beings in each cycle of existence. I’ll therefore put off discussing the Cookies until a later point, and instead tell you an anecdote about Master Canaro that has to do precisely with code comments. Once, Master Canaro showed his students a piece of code and asked them to explain what it did:

/**********************************************************
 
#include <iostream.h>

#define ERR_NOFILE 1

int errore( int errore );
void leggi_file( char * nomefile );
void crea_matrice(void);
int inizializza(void);
void imposta_probabilita(void);
void attrattore(void);
int chiudi(void);

int main(int argc, char * argv[])
{
    int esito = 0 ;
    if(argc < 2) {
        esito = errore(ERR_NOFILE);
    } else {
        leggi_file(argv[1]);
        crea_matrice();
        if(inizializza()) {
            const char * valore = imposta_probabilita();
            attrattore(valore);
            esito = chiudi();
        }
    }
    return esito ;
}

***********************************************************/

The students studied the code and gave their answers, but they were all wrong, because every instruction was enclosed between a /* and a */ and was never compiled.
