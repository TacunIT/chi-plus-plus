Functions

Call me: IsNull

Functions are C++’s main distinguishing feature.

The purpose of functions is to make code easier to manage. It’s possible to write a program that only has the main function, but this only makes sense for very simple programs, like some of the examples we’ve seen so far. The flow of a non-trivial program, as Stroustrup would say, will always be split across several functions, because this makes the code easier to read, to fix, and to modify.

/** 
 * @file tipi-di-dato-dimensione.cpp
 * Mostra la dimensione dei principali tipi di dato del C++.
 */
 
#include <iostream>
#include <iomanip> 

using namespace std;

int main()
{    
    int w = 12;
    
    cout << setw(w) << "bool: "   << sizeof(bool)   << endl;
    cout << setw(w) << "char: "   << sizeof(char)   << endl;
    cout << setw(w) << "int: "    << sizeof(int)    << endl;
    cout << setw(w) << "float: "  << sizeof(float)  << endl;
    cout << setw(w) << "double: " << sizeof(double) << endl;
    
    return 0;
}

This is the code of an example we saw in the lesson on data types. If we decided to change how the program displays data to the user, we’d have to touch every single line of the program, which would be tedious and could introduce bugs. By contrast, if we isolate the display function, any changes or fixes only need to be applied in one place. The code of the following example is a perfectible illustration of this approach:

/** 
 * @file tipi-di-dato-limiti.cpp
 * Mostra i valori possibili per i principali tipi di dato del C++.
 */

#include <iostream>
#include <iomanip> 
#include <limits>

using namespace std;

/**
 * Funzione template che mostra la dimensione e i 
 * valori minimi e massimi per un tipo di dato.
 */
template<typename T>
void dimensione()
{
    cout << sizeof(T) << " bytes,"
         << setw(4)   << "da:"   
         << setw(21)  << numeric_limits<T>::min()
         << setw(4)   << "a:"    
         << setw(21)  << numeric_limits<T>::max() 
         << endl;
}

int main()
{    
    /** Variazione della dimensione e dei valori del tipo int */
    cout << setw(16) << "int: "; 
    dimensione<int>();
    cout << setw(16) << "unsigned int: "; 
    dimensione<unsigned int>();
    cout << setw(16) << "short int: "; 
    dimensione<short int>();
    cout << setw(16) << "unsigned short: "; 
    dimensione<unsigned short>();
    cout << setw(16) << "long int: "; 
    dimensione<long int>();
    cout << setw(16) << "unsigned long: "; 
    dimensione<unsigned long>();
    
    return 0;
}

Whenever there are instructions in your code that repeat, consider the possibility of isolating them into a function. For example, in the main function of the program above there are repeated output functions, which makes the code harder to read, more laborious to modify, and increases the chance of mistakes, because you’re the one defining the string with the data type’s name, and you have to be careful to write the right value every time:

cout << setw(16) << "int: ";
...
cout << setw(16) << "unsigned int: ";
...
cout << setw(16) << "short int: ";
...
cout << setw(16) << "unsigned short: ";
...
cout << setw(16) << "long int: ";
...
cout << setw(16) << "unsigned long: ";

In this particular case, it was an unavoidable choice, because, being one of the first examples I gave you, I didn’t want to complicate things too much for you, but this isn’t the right way to write code. It’s wrong (or, at the very least, risky) to mix data processing with interface functions, especially in a case like this, where part of the output is already delegated to a specific function. Better to let the dimensione function handle all the output, while the main function limits itself to defining the flow of processing, calling the output function for the data types we’re interested in:

/** 
 * @file funzioni-limiti-modificato.cpp
 * Esempio sui limiti del C++ con output in un'unica funzione.
 */

#include <iostream>
#include <iomanip> 
#include <limits>
#include <typeinfo>
#include <cxxabi.h>

using namespace std;

/** 
 * Mostra il nome, la dimensione e i valori minimi 
 * e massimi per il tipo di dato corrente.
 */
template<typename T>
void dimensione()
{
    cout << setw(14)  
         << abi::__cxa_demangle(typeid(T).name(), NULL, NULL, NULL) 
         << ": "
         << sizeof(T) << " bytes,"
         << setw(4)   << "da:"   
         << setw(21)  << numeric_limits<T>::min()
         << setw(4)   << "a:"    
         << setw(21)  << numeric_limits<T>::max() 
         << endl;
}

/** 
 * Il nuovo codice della funzione main:  più compatto 
 * e più facile da leggere o modificare.
 */
int main(int argc, char** argv)
{    
    dimensione<int>();
    dimensione<short int>();
    dimensione<unsigned short>();
    dimensione<long int>();
    dimensione<unsigned long>();
    return 0;
}

The instruction:

abi::__cxa_demangle(typeid(T).name(), NULL, NULL, NULL);

displays the data type of the current variable. For now, just accept it on faith; I’ll explain how the :: operator behaves when we talk about classes. The output of this code is identical to that of the previous example; if anything, it’s a bit better, because I reduced the width of the first column from 16 to 14 characters, and did so by modifying a single setw(14) instruction, instead of the six in the previous code:

Functions are one of C++’s data types, and like all data types, they can be declared and defined. The declaration of a function establishes its return type and the parameters it requires as input:

float scorporaIVA(long stipendio, float aliquota);

The definition of a function establishes its behavior:

float scorporaIVA(long stipendio, float aliquota)
{
    float valore;
    valore = stipendio / ((100 + aliquota) / 100);
    return valore;
}

As I told you when we talked about programming languages, generating an executable file happens in two stages: first the compiler converts the C++ code into an object file, then the linker turns the object files (there could be more than one) into a single executable. For this process to work, a function’s declaration must be present in every piece of code that uses it, so the compiler can check that it’s being used correctly; the definition, by contrast, must appear only once.

/** 
 * @file funzioni-stipendio-main.cpp
 * Funzione main del programma con file oggetto separati.
 */
#include <iostream> 

/** 
 * Dichiarazione della funzione raddoppiaStipendio. 
 * L'ho messa qui per semplicità: di solito, le dichiarazioni
 * si trovano in un file di include separato.
 */
float raddoppiaStipendio(float stipendio);

/**
 * La funzione main richiama la funzione raddoppiaStipendio,
 * che è definita in un altro file sorgente.
 */
int main(int argc, char** argv)
{        
    std::cout << raddoppiaStipendio(1500) << std::endl;    
    return 0;
}


/** 
 * @file funzioni-stipendio-funz.cpp
 * Funzione del programma con file oggetto separati.
 */

/** 
 * Definizione della funzione raddoppiaStipendio. 
 */
float raddoppiaStipendio(float stipendio)
{

Earlier I told you to consider the possibility of isolating repeated instructions into a function, because creating a function isn’t always the right choice. Software, like many human endeavors, is the product of a series of trade-offs, and you’ll have to make different architectural choices depending on the kind of program you need to build. Good software, besides working correctly, should be fast, easy to modify, and require few system resources. Sometimes, though, you have to sacrifice one of these positive traits to boost another. For example, if you had to write extremely fast software, it might be better to have duplicated instructions rather than function calls, because calling a function inevitably causes some slowdown. But if you duplicate instructions, you not only increase the size of the program, you also make it harder to read and modify, which is bad. In these cases, some increase in code size is unavoidable, but inline functions let you keep the code readable and modifiable:

/** 
 * @file funzioni-inline-1.cpp
 * Esempio di codice con istruzioni ripetute.
 */

#include <iostream>
#include <iomanip> 
#include <fstream>

#define S_FILENAME  "src/out/esempio.txt"

using namespace std;

int main(int argc, char** argv)
{        
    /** 
     * Crea una variabile di classe ofstraeam 
     * per gestire il file di output  */
    ofstream doc;

    /** Apre il file di output */
    doc.open (S_FILENAME);
    cout << "ho aperto il file " << endl;

    /** Scrive sul file di output */
    doc << "Testo del documento.\n";
    cout << "ho scritto sul file" << endl;

    /** Chiude il file di output */
    doc.close();
    cout << "ho chiuso il file" << endl;

    return 0;
}

 * Esempio di codice con funzione di output unica.

#include <iostream>
#include <iomanip> 
#include <fstream>

using namespace std;

/** Funzione di output unica */
void log(const char* messaggio)
{
    /** Mostra il messaggio a video */

}

The output of this program is identical to that of the previous example, but if you add the inline keyword before the return type of the log function:

inline void log(const char* messaggio)
{
    /** Mostra il messaggio a video */
    cout << messaggio << endl;
}

and recompile the program, you’ll see that the size of the executable file has grown, because the compiler replaced every function call with a copy of the function’s own code. Even though the executable’s size has grown, the code is still easy to read and modify:

/** 
 * @file funzioni-inline-3.cpp
 * Esempio di funzione inline.
 */

#include <iostream>
#include <iomanip> 
#include <fstream> 

#define LOG_DEBUG   1
#define LOG_AVVISO  2
#define LOG_ERRORE  3

#define S_DEBUG     "debug"
#define S_AVVISO    "avviso"
#define S_ERRORE    "errore"

#define ERR_NO_FILE_NAME  -1
#define ERR_NO_FILE_OPEN  -2

using namespace std;

/** Funzione di output inline */
inline void log(const char* messaggio, int livello)
{
    /** Definisce la spaziatura del primo campo */
    cerr << setw(7);  
    
    /** Scrive il livello del messaggio */
    switch(livello) {
        case LOG_DEBUG:  cerr << S_DEBUG ; break;
        case LOG_AVVISO: cerr << S_AVVISO; break;
        default:         cerr << S_ERRORE; break;
    }
    
    /** Scrive il testo del messaggio */
    cerr << ": " << messaggio << endl;
}

int main(int argc, char** argv)
{        
    /** Crea una variabile per gestire il file di output */
    ofstream doc;

    /** Puntatore per il nome del file di output */
    const char* filename = NULL;

    /** Se manca il nome del file di output, errore */
    if(argc < 2) {
        log("specificare il path del file", LOG_ERRORE);
        return ERR_NO_FILE_NAME;
    }
    
    /** Legge il nome del file di output */
    filename = argv[1];
    
    /** Prova ad aprire il file di output */    
    doc.open (argv[1]);
    
    /** Se c'è stato un errore, lo segnala ed esce */
    if(!doc.is_open()) {
        log("impossibile aprire il file", LOG_ERRORE);    
        return ERR_NO_FILE_OPEN;    
    }
    
    /** OK, il file è pronto */
    log("ho aperto il file", LOG_AVVISO);

    /** Scrive sul file di output */
    doc << "Testo del documento.\n";
    log("ho scritto sul file", LOG_DEBUG);

    /** Chiude il file di output */
    doc.close();
    log("ho chiuso il file", LOG_AVVISO);

    return 0;
}

The existence of an inline function is somewhat reminiscent of a passage from the Samyutta Nikaya:

There is no distinction between a being, its function, and the time of its appearance. Beings appear out of non-existence, exist for an instant, and then cease to exist. Their existence, activity, and action are one and the same thing. Past and future are mere names.2

In general, though, the lifecycle of a function involves three distinct phases: declaration, definition, and its subsequent use:

/** Dichiarazione */
float raddoppiaStipendio(float stipendio);

/** Definizione */
float raddoppiaStipendio(float stipendio)
{
    return stipendio * 2;
}

/** Chiamata */
float importo = raddoppiaStipendio(1500);

Only the main function is exempt from this rule, because it requires no declaration and is never called by other functions. The variable float stipendio, which appears in parentheses both in the declaration and in the definition of the function, is called the function’s parameter. The value 1500, assigned to the parameter in the function call, is called the argument. These variables are also called formal parameters, in the case of the declaration and the definition, and actual parameters, in the case of the call. When you call a function, you can pass it arguments in three ways: by value, by reference, by pointer:

/** 
 * @file funzioni-argomenti.cpp
 * Esempio di gestione degli argomenti di una funzione.
 */

#include <iostream>
#include <iomanip> 
#include <fstream> 

using namespace std;

/**
*   Chiamata per valore: tutto quello che succede qui
*   alle variabili a e b non influenza le variabili 
*   x e y della funzione chiamante.
*/
void perValore(int a, int b)
{
    int tmp = a;
    a = b;
    b = tmp;
    return;
}

/**
*   Chiamata per reference: tutto quello che succede qui
*   alle variabili a e b è come se fosse fatto alle variabili
*   x e y della funzione chiamante.
*/
void perRiferimento(int &a, int &b)
{
    int tmp = a;
    a = b;
    b = tmp;
    return;
}

/**
*   Chiamata per puntatore: anche qui, quello che succede 
*   alle variabili a e b avviene anche alle variabili x e y 
*   della funzione chiamante, ma il codice è meno chiaro.
*/
void perPuntatore(int *a, int *b)
{
    int tmp = *a;
    *a = *b;
    *b = tmp;
    return;
}

int main(int argc, char** argv)
{        
    int  x = 11; int  y = 22;    
    cout << setw(20) << "Valore iniziale: x=" << x << ", y=" << y << endl;
    
    int* p = &x;
    int& r = x;
    
    cout << "p:" << p << ",r=" << r << endl;

    /**
    *   Passaggio per valore: l'argomento della chiamata
    *   a funzione è il valore delle variabili x e y:
    */
    perValore(x,y);
    cout << setw(20) << "perValore: x=" << x << ", y=" << y << endl;

    /**
    *   Passaggio per reference: l'argomento della chiamata
    *   a funzione sono le variabili x e y, ma la funzione 
    *   utilizzerà dei loro alias, non il loro valore:
    */
    perRiferimento(x,y);
    cout << setw(20) << "perRiferimento: x=" << x << ", y=" << y << endl;

    /**
    *   Passaggio per puntatore: l'argomento della chiamata
    *   è esplicitamente l'indirizzo delle variabili x e y:
    */
    perPuntatore(&x,&y);
    cout << setw(20) << "perPuntatore: x=" << x << ", y=" << y << endl;
    
    return 0;
}

Passing a reference to a variable as a function argument is equivalent to passing the variable itself. This can be useful for functions that require large objects as input, or that need to directly modify the value of the variables passed to them as parameters, but it should be avoided in any other case, because it lets the program modify the value of a variable in a sneaky way, which can be very hard to spot when debugging. Another thing worth knowing, about function parameters, is that they can have default values:

/** 
 * @file _man/src/funzioni-parametri-default.cpp
 * Esempio di funzione con parametri di default.
 */

#include <iostream>

using namespace std;

/** Funzione con parametri di default  */
void stampaValore(int valore = 0)
{
    /** Stampa a video il valore del parametro */
    cout << "Valore: " << valore << endl;
}

int main(int argc, char** argv)
{        
    /** Richiama la funzione con un argomento */
    stampaValore(123);

    /** Richiama la funzione senza argomenti */
    stampaValore();

    return 0;
}

The output of this program will be:

% g++ src/cpp/funzioni-parametri-default.cpp -o src/out/esempio
% src/out/esempio
Valore: 123
Valore: 0

A parameter’s default value must be specified in the function’s declaration. Remember, though, that when you assign a default value to one of a function’s parameters, you must do the same for every parameter that follows it, if there are any:

void funz1(float f, void * ptr = NULL);        // OK
void funz2(double d, int b = 2, char c = 'c'); // OK
void funz3(int i = 3, int n );                 // ERRORE!

Default values are used when one or more of a function’s parameters is often assigned the same value. For example, if the log function we saw in the previous example were mostly called with the same value for the livello parameter, we could assign it as a default:

inline void log(const char* messaggio, int livello = LOG_AVVISO);

making the code faster and easier to write:

log("ho aperto il file");

doc << "Testo del documento.\n";
log("ho scritto sul file", LOG_DEBUG);

doc.close();
log("ho chiuso il file");

In one of our earliest conversations, I told you that the Good Programmer, while writing code, always asks whether there’s a more efficient way of doing what he’s doing. Today you’ve seen proof of that: we improved the first example in the section on inline functions by unifying message handling into a single function, and then improved it again by making that function inline. There’s a problem, though: the program’s output:

% ./src/out/esempio                                   
ho aperto il file
ho scritto sul file
ho chiuso il file

only works if there’s a single file to handle, as in our case. But if there were two (or more) input or output files, it would be useful to know which file the message refers to. To solve the problem, we could increase the number of formal parameters of the log function:

void log(int livello, const char* messaggio, const char* file);

but the new parameter would be useless for calls like:

log(LOG_ERRORE, "specificare il path del file");

Alternatively, we could build the message inside the calling function:

string s1 = "ho chiuso il file: ";
string s2 = filename;
string s3 = s1 + s2;
log(LOG_AVVISO, s3.c_str());

but this would make the code heavier, more complicated, and slower; it would also shift back into the calling functions part of the output functionality we had happily isolated into the log function. The correct solution to this kind of problem is functions with variable parameters. A typical example of this kind of function is the C language’s printf function, whose first argument is used to determine the type and number of the arguments that follow it:

/** Dichiarazione, nel file stdio.h */
int printf(const char *format, ...) ;

/** Utilizzo */
printf("stringa: %s; intero: %d", "codice errore", -1);

To indicate variable arguments, in a function’s declaration, three dots are used after the fixed parameters:

void log(int livello, int n_parametri, ...);

In the function’s definition, to handle the parameters, you need to create a reading loop using three macros defined in the stdarg.h file:

void va_start(va_list ap, lastfix);
type va_arg(va_list ap, type);
void va_end(va_list ap);

va_start initializes the ap variable list so it can receive the variable arguments; the lastfix parameter specifies which is the function’s last fixed parameter.

va_arg returns the next parameter in the ap list; the type parameter indicates the parameter’s data type.

va_end ends the processing of the parameters, and must always be called before the function returns.

/** 
 * @file funzioni-variabili.cpp
 * Esempio di funzione con parametri variabili.
 */

#include <iostream>
#include <iomanip> 
#include <fstream> 

#define LOG_DEBUG   1
#define LOG_AVVISO  2
#define LOG_ERRORE  3

#define S_DEBUG     "debug"
#define S_AVVISO    "avviso"
#define S_ERRORE    "errore"

#define ERR_NO_FILE_NAME  -1
#define ERR_NO_FILE_OPEN  -2

#define S_ERR_NO_FILE_NAME  "specificare il path del file"
#define S_ERR_NO_FILE_OPEN  "impossibile aprire il file:"

using namespace std;

/** Funzione di output con parametri variabili */
void log(int livello, int n_parametri, ...)
{

    /** Definisce la spaziatura del primo campo */
    cerr << setw(8);  
    
    /** Scrive il livello del messaggio */
    switch(livello) {
        case LOG_DEBUG:  cerr << S_DEBUG ; break;
        case LOG_AVVISO: cerr << S_AVVISO; break;
        default:         cerr << S_ERRORE; break;
    }

    /** Scrive il carattere di separazione */
    cerr << " | ";

    /** Dichiara la variabile per la lista dei parametri */    
    va_list lista_parametri;
    
    /** 
     * Inizializza la lista dei parametri e considera
     * tutti gli argomenti dopo n_parametri come variabili.
     */
    va_start(lista_parametri, n_parametri);

    /** Legge tutti i parametri nella lista e li scrive a video */
    for(int p = 1; p <= n_parametri; p++) {        
        cerr << va_arg(lista_parametri, char*) ;        
    }    

    /** Chiude la lista dei parametri */        
    va_end(lista_parametri);
    
    /** Scrive il testo del messaggio */
    cerr << endl;

}

int main(int argc, char** argv)
{        
    /** Crea una variabile per gestire il file di output */
    ofstream doc;

    /** Puntatore per il nome del file di output */
    const char* filename = NULL;

    /** Se manca il nome del file di output, errore */
    if(argc < 2) {
        log(LOG_ERRORE, 1, S_ERR_NO_FILE_NAME);
        return ERR_NO_FILE_NAME;
    }
    
    /** Legge il nome del file di output */
    filename = argv[1];
    
    /** Prova ad aprire il file di output */    
    doc.open (argv[1]);
    
    /** Se c'è stato un errore, lo segnala ed esce */
    if(!doc.is_open()) {
        log(LOG_ERRORE, 2, S_ERR_NO_FILE_OPEN, filename);    
        return ERR_NO_FILE_OPEN;    
    }
    
    /** OK, il file è pronto */
    log(LOG_AVVISO, 2, "ho aperto il file: ", filename);

    /** Scrive sul file di output */
    doc << "Testo del documento.\n";
    log(LOG_DEBUG, 2, "ho scritto sul file: ", filename);

    /** Chiude il file di output */
    doc.close();
    log(LOG_AVVISO, 2,  "ho chiuso il file: ", filename);

    return 0;
}

/** 
 * @file /src/funzioni-ricorsiva.cpp
 * Esempio di funzione ricorsiva.
 */

#include <iostream>
#include <iomanip> 

using namespace std;

/** Dichiarazione della funzione ricorsiva */
long fattoriale(int n);

/** Funzione main prima della definizione */
int main(int argc, char** argv)
{   
    int n = 9;
    cout << n << "! = " << fattoriale(n) << endl;
    return 0;
}

/** Definizione della funzione ricorsiva */
long fattoriale(int n)
{
    long fatt = 1;
    if(n > 1) {
        fatt = n * fattoriale(n-1);
    }
    return fatt;
}

Be careful, though: a function that calls itself, if not written well, can generate an in[de]finite recursion, like two mirrors facing each other. The perspective space inside the mirrors is inexhaustible, but a computer’s memory, however large, has its limits, and too many recursions can cause a stack overflow.3 For this reason, you should resort to recursion only when it’s truly necessary, preferring, whenever possible, iterative instructions and always applying mechanisms to control recursion depth, to prevent a stack overload. Another kind of recursion to be wary of is the kind sometimes used to lend credibility to a belief with no real foundation, by citing another, similar belief.

In a novel about the life of the Japanese swordsman Miyamoto Musashi, there’s a line that follows logically from the Roman proverb I told you about yesterday:

Non bisogna interferire nel funzionamento dell’Universo, ma prima è necessario capire quale sia, il funzionamento dell’Universo4

The problem with novels is that they often tell you what to do, but almost never explain how, or why. Be wary of teachers who only know how to teach, because everything they tell you they learned from books. A good teacher must excel at something, whether it’s fencing, archery, carpentry, or motorcycle maintenance. He may not be the best, but he must be among the best; only then will you know his ideas are sound. By contrast, teachers who have never left a school have never had to put their ideas to the test of reality. They teach you what they think is right, but what’s right or true within a school isn’t always equally true or right in the real world. It’s easy to be a saint inside a monastery, among people who share your culture and your own principles; the hard part is remaining a saint outside of it. The Buddha Shakyamuni preached Universal Loving-kindness because he lived in India twenty-six hundred years ago, but if he’d been stuck in city traffic after a day’s work, his ideas might have looked a bit more like Nietzsche’s. Fortunately, Musashi — the real one, the Samurai — was always very clear both about what’s right to do and about how to achieve it. These rules, which let Musashi reach the venerable age of sixty,5 after emerging victorious from just as many duels against all the finest swordsmen of his time, can help you understand how the Universe works, and what to do so as not to disturb it. The practice of the arts, whether intellectual or martial, and the knowledge of trades, combined with attention to everything around you, will broaden your store of experience and let you tell the pros and cons of any situation apart. The next step will be to free yourself from desire and attachment to things: just as the log function needs to know which are its fixed parameters and which are variable, you’ll have to learn to tell your desires apart from your needs, in order to escape the influence of the Age-Old Dichotomy. The Way, with a capital “W”, is a lot like a way with a lowercase “w”. Along a way, you come across road signs, which show you which direction to go or what speed to keep, and billboards, which point out nearby attractions and invite you to stray from your path to go visit them. In the same way, along the Way, you’ll find needs, which will guide you toward your destination, and desires, which will dis-tract you from your path and at-tract you toward alternative destinations. If you act according to your needs, you’ll always know you’re moving in the right direction, even when you’re forced to slow down or take roads you don’t enjoy. If instead you act according to your desires, you’ll wander here and there, like “a donkey without reins”,6 and by the time you finally get back on the right road, you might no longer have time to reach your destination.

I told you about Musashi’s rules not because they’re the only ones available, but because they’re intensely personal. They draw on the principles of other disciplines, like Buddhism, Bushido, or Tao, but they’re something else, as Eddie Cochran would say. You’ll have to do the same: learn everything you can, from good and bad teachers alike, and then define your own rules for life, which may end up the same as, similar to, or entirely different from the ones you were taught. The Way, like Truth, is a model, whom each of us portrays from our own point of view, trying to guess at the body hidden behind the folds of the drapery. Portraying her the same way someone else has would be wrong, because your point of view isn’t the same as theirs, but looking at her from several points of view can help you understand her shape better. Musashi says to pay attention even to insignificant things, Nan-in and Tenno develop their Every-Minute Zen, Wittgenstein, in his Diaries, writes:

Only one thing is necessary: to be able to observe everything that happens to you. Concentrate! God help me!

It’s clear they’re all painting the same picture, even if each does so in his own style. This is why Musashi prescribes knowing the Way of other trades: because there’s always something to learn from someone who does their job well. What’s true for the carpenter, with a shift in perspective, is also true for the strategist, and could be true for you too, if you ever find yourself managing a project or a team. Even as a programmer, though, you’ll have to keep constantly up to date on techniques and programming languages, and you’ll need to understand how those who work on databases or systems do their job. You’ll need to be able to spot malfunctions even when they have no visible effect on the graphical interface, and you’ll only manage that if you pay attention to even the smallest detail. The names change, because four centuries have passed, but the problems remain the same, and so do the solutions. After all, art, science, philosophy, and religion are all attempts to answer the same questions about our existence. Since the object of study is the same, it’s only natural that, sometimes, the answers resemble each other, in the same way that the algorithm for calculating a number’s factorial is more or less the same, regardless of the programming language.
