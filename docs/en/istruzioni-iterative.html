Iterative instructions

Indeed it is a rather strange thing to hear, but the name we at least give it — a name no one, for lack of experience in the matter, would ever think of — is: Spazionism

Loop instructions are the fundamental building blocks of programming.

There are three kinds of loop instructions:

  • for
  • while
  • do while.

These instructions are made up of two parts: a loop control instruction, which determines how long it runs, and a loop body, made up of the instructions that get repeated on each iteration. The repetition can go on either for a set number of times, or until a given condition is met.

The for loop is used when we want to run the loop a fixed number of times. Its general form is as follows:

// controllo del ciclo
for(<stato iniziale> ; <stato finale> ; <variazione>)
{
    // corpo del ciclo
    <istruzioni>
}

The three conditions inside the parentheses are used by the for instruction to control the execution of the instructions inside the loop body. The first expression is evaluated only once, at the start of the loop, and is usually used to initialize the variables in use. The second expression is a logical or relational condition evaluated at the start of every iteration: if it returns 0 or false, the loop stops running, otherwise it continues. The third expression is evaluated at the end of every iteration, and usually consists of an expression that increments or decrements the variables used to control the loop. To do this, we use unary operators (that is, operators that act on a single variable) called increment operators and decrement operators, which can carry out their function either before or after the variable is used, depending on whether they’re placed before or after the variable’s identifier:

x++     // valuta x e poi la incrementa di un'unità
++x     // incrementa x di un'unità e poi la valuta
x--     // valuta x e poi la riduce di un'unità
--x     // riduce x di un'unità e poi la valuta

If x is a variable of integer or floating-point type, the increment is by one arithmetic unit; if instead x is a pointer, the increment equals the size of the variable the pointer refers to, as I showed you when we talked about data types. In this case, the variable p is an integer, so the instruction:

for ( p = POS_MERCURIO; p <= POS_PLUTONE; p++ ) {

will increment the variable p by 1.

You can also use a for loop to run loops with an undefined number of iterations; just omit the three control expressions, leaving only null statements, made up of the bare ; terminator:

for( ; ; )
{
    // istruzioni
}

A loop like this will keep repeating indefinitely, and, if it isn’t stopped somehow, will inevitably cause problems for the computer running it. You therefore need to set a limit on the number of repetitions, using the same break keyword we used with switch instructions. This time, though, we’ll start doing things the right way, and split the three components of the previous program into three separate files: pianeti.h, which will hold the declarations of the constants and of the mostraPianeta function; pianeti.cpp, containing the definition of the mostraPianeta function; and pianeti-main.cpp for the main function:

The increment of the variable p, in this case, happens inside the instruction:

if(!mostraPianeta( p++ )) break;

The break keyword is one of the three interruption instructions C++ inherited from C; the other two are the continue instruction, which sends processing back to the start of the loop, and the return instruction, which ends the function, returning any return value to the calling function.

The do-while loop is the same as the while loop, with the only difference that the while condition is evaluated at the end of the iteration, so the loop body is run at least once. The general form of the do-while loop is:

do
{
    <istruzioni>        // corpo del ciclo
} while(espressione)    // espressione di controllo

I won’t insult your intelligence with an example; I’d rather point out something more serious: the code of these programs is clumsy. The problem is that the mostraPianeta function does too many things: it not only figures out the Planet’s name, it also prints it to the screen. As a result, our output also includes that ugly error string about position number nine. In the first example where we used it, this wasn’t a problem, but now that our program is growing, we need to make each function more specialized, separating data processing (figuring out which planet it is) from the user interface (printing the name to the screen). To do this, we’ll use a new function, which we’ll add to the pianeti.cpp file:

The new main function will be:

/** 
 * @file pianeti-while-string.cpp
 * Gestione dei Pianeti con valori stringa.
 */
 
#include <iostream>
#include "pianeti.h"

using namespace std;

int main(int argc, char** argv)
{    
    int    p  = POS_MERCURIO;
    string nome;

    /** 
     * Scinde l'elaborazione del dato dalla sua
     * eventuale visualizzazione
     */
    while (!(nome = nomePianeta(p)).empty()) {
        cout << p++ << ": " << nome << endl;
    }
        
    return 0;
}

The code is syntactically more complex, but once you understand that the instruction:

while(!(nome = nomePianeta(p)).empty())

means:

run the nomePianeta function, passing it the variable p as a parameter, until it returns an empty string to you

the flow of the program becomes clearer than it was in the previous cases. The loop body has gone from three instructions to one, and the two operations of processing and display are now clearly separated in time. Besides getting code that’s easier to read, fix, run, and modify, we’ve also gotten rid of that annoying error message. I’d say it was worth it, wouldn’t you?

Writers, sculptors, and painters cannot modify their works once they’ve been published. They can add a touch of color here, a chisel stroke there, but these are always minor adjustments, which don’t change the very structure of the art-object. Musicians, theater artists, and, to some extent, filmmakers are a bit luckier, because they can more easily make changes to their works, but even so, these are still rare events. Good software production, by contrast, has the advantage of being in constant evolution. A piece of software can be stable, that is, have no known flaws, but it’s never finished, complete, perfect; partly because actual use might reveal flaws that slipped through testing, partly because changes in the context of use might call for changes to the system. Good software is therefore constantly updated, and the different versions of the same product are numbered progressively with codes made up of three numbers separated by dots, indicating, respectively, the major version, the minor version, and the patch; for example: 1.4.12. The major version is incremented every time drastic changes are made to the software, making it incompatible with previous versions. The minor version is incremented when the code is changed in a smaller way, adding or modifying features in a way that’s compatible with previous versions. The patch number is incremented every time even minor changes or fixes are made to the system. Major version zero (0.y.z) is reserved for initial development, when the software isn’t stable yet and everything can change at any moment.
