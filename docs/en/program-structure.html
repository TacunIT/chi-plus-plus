Syntactic freedom and the conciseness of its constructs are C++’s main distinguishing feature.

C++, like all languages, has its own syntax, which defines the elements of the code and how they’re used within programs. In the examples we’ve seen so far, I’ve used some of these elements without explaining precisely what their role is, because I wanted to give you an overall picture of the language’s features. A bit like when you arrive at a party and get introduced to the other guests one after another, and by the end the only thing you remember is the ladies’ necklines. Now, though, we need to take a step back and examine these elements one by one, starting with what is the main element of every C++ program: the main function.

int main(int argc, char** argv)
{    
    return 0;
}

The one above is the smallest C++ program you can write. It’s also the most useless one, though, because it doesn’t do anything. The example below is just as useless, but a bit more complex:

/** 
 * @file src/struttura-hello-world.cpp
 * Esempio minimo di programma C++.
 */
 
#include <iostream>

int main(int argc, char** argv)
{    
    std::cout << "Hello, World!" << std::endl;
    return 0;
}

Its output, as you’ve probably guessed, is:

% g++ struttura-hello-world.cpp -o ../out/esempio
% ../out/esempio
Hello, World!

The first line of code:

#include <iostream>

isn’t an instruction, but a directive for the preprocessor. The preprocessor is a program that processes the code and prepares it for compilation; we’ll talk about it later on. For now, all you need to know is that this instruction causes the file iostream.h to be included in the code, which contains, among other things, the declaration of the cout stream, used in the program’s only instruction. The second line of code:

int main(int argc, char** argv)

defines the main function, specifying that it will have int as its return value and two parameters: an integer and a pointer to pointers to char. These values are used to handle the parameters passed on the command line: the integer argc (short for argument count) specifies the number of parameters present in the call, including the program’s own name, while the parameter argv (argument vector) is an array of pointers to all the strings present on the command line. The body of the function is enclosed in a pair of curly braces {}. The instruction:

std::cout << "Hello, World!" << std::endl;

writes the string Hello World! to the std::cout stream (standard character output), which usually corresponds to the computer’s screen. The << operator, in this case, tells the system to send whatever is on its right to output, and returns a reference to the output stream, so that it can be chained on the same line of code, which is a more efficient and elegant form of:

std::cout << "Hello, World!" ;
std::cout << std::endl;

Finally, the instruction:

return 0;

returns the value 0 to the calling program (usually, the computer’s shell) to indicate that no errors occurred during processing. The following program puts into practice the concepts we’ve seen so far:

#include <iostream>

#define NO_ERRORI     0
#define NO_PARAMETRI  1

int main(int argc, char** argv)
{    
    int errore = NO_ERRORI;
    
    /** Se c'è solo il nome del programma, errore */
    if(argc < 2) {
        std::cerr << "Specificare un parametro!" << std::endl;
        return NO_PARAMETRI;
    }

    /** Stampa tutti i parametri ricevuti */
    for(int p=0; p < argc; p++) {
        std::cout << *argv++ << std::endl;        
    }

    return 0;
}

As I’ve told you, C++ is a compiled language, so the code, in order to be run, must be processed by the compiler with the shell command:

% g++ struttura-argc-argv.cpp -o src/out/esempio

The -o parameter lets you specify the name of the output file, in this case: src/out/esempio. If you don’t set this value, the compiler generates a file named a.out. The shell command:

% src/out/esempio; echo $?

is made up of two instructions, separated by the ; character. The first instruction runs the compiled file; the second prints its return value to the screen. If we run the program with no parameters, we get a message and error code 1:

Specificare un parametro!
1

If instead we run the program passing it some parameters, we get this:

% src/out/esempio un due tre; echo $?
src/out/esempio
un
due
tre
0

No program worthy of the name has only the main function; it splits its work across a series of functions, each carrying out a precise, well-defined task. In a well-written program, functions display two characteristics, which English speakers and Anglophiles call: low coupling and high cohesion. By the coupling of two functions, we mean the amount of information function A needs to have about function B in order to use it. Each function expects to receive a well-defined set of parameters: the raddoppiaStipendio function, which we saw earlier, expects to receive a single parameter, of type long:

long raddoppiaStipendio(long stipendio);

while a hypothetical scorporaIVA function might require two: the salary amount and the VAT rate:

float scorporaIVA(long stipendio, float aliquota);

In both of these cases, all a third function needs, to be able to call raddoppiaStipendio or scorporaIVA, is their interface, that is, the number, type, and order of the parameters to pass. Now imagine that a clumsy programmer has written the function facciQualcosa, which can carry out several distinct actions depending on the parameters it receives:

float facciQualcosa(long stipendio, int azione, float aliquota = 0)
{
    float valore = 0;
    /** Differenzia l'azione in base al valore di azione */
    if(azione == 1) {
        valore = stipendio * 2;
    } else if(azione == 2) {
        valore = stipendio / ((100 + aliquota) / 100);
    }
    return valore;
}

To use this function, we not only have to know its interface, we also have to know which actions correspond to the various values of the azione parameter. This is madness, meshuggah, because if one day the author modified it and decided that the value 1 of the azione parameter now triggers VAT deduction while the value 2 triggers doubling the salary, we would also have to modify every function that called it, to adapt them to the new rules. Not only would we waste time, but if we forgot to update one or more calls we would end up with a program that behaves incorrectly. Coupling is like cholesterol: the lower, the better; so, to avoid mistakes, we need to reduce it by creating an enum to hold the possible values of the azione parameter:

enum Azione { raddoppia, scorpora };

As often happens, a single line of well-written code lets us save time and get more robust code, because the effect of the azione parameter, this way, will be entirely independent of its numeric value:

enum Azione { raddoppia, scorpora };

float facciQualcosa(long stipendio, Azione azione, float aliquota = 0)
{
    float valore = 0;
    /** Agisce in base all'etichetta, non al valore */
    if(azione == raddoppia) {
        valore = stipendio * 2;
    } else if(azione == scorpora) {
        valore = stipendio / ((100 + aliquota) / 100);
    }    
    return valore;
}

The facciQualcosa function has another design flaw too, besides high coupling: it lacks internal cohesion. In a well-written program, each function should have only one… function:

inline long raddoppiaStipendio(long stipendio)
{
    return stipendio * 2;
}

inline float scorporaIVA(long stipendio, float aliquota)
{
    return (stipendio / ((100 + aliquota) / 100);
}

Even in such a simple example, you can clearly see the difference between a function that can carry out several unrelated actions and a function that carries out a single, well-defined action. By reducing the complexity of the function, moreover, we’re able to declare it as inline, increasing the program’s execution speed.

Once, a disciple asked Master Canaro how it was possible for the Energy of the universe to have feelings or aspirations, and whether this self-awareness didn’t clash with the idea that C’hi++ is a metaphysics free of metaphysical elements. Master Canaro answered that the disciple was right (an elliptical way of saying he was an idiot). He himself liked to believe it was so, because he was a sentimental old man, but the disciple was free to think it was just a rhetorical device, used to make the story more engaging. He then explained that the only thing one needed to believe, even in the absence of proof, was that the Universe, at the end of this phase of expansion, would go back to collapsing into the One. At that point, there would be two possibilities: either it would remain One for the rest of Time (a legitimate hypothesis, but a boring one), or it would explode again, giving rise to a new Universe. Here too, there would be two hypotheses. The first is that a Big Bang can only happen under certain conditions, and that those conditions necessarily lead to a Universe identical to the one we know now; so, if the One explodes again, everything starts over from scratch. The second hypothesis is that every Big Bang happens under its own specific circumstances and in its own specific way, and that therefore, if the One explodes again, a new Universe will be born, one that may have few or no points of contact with the current one. Master Canaro said the first hypothesis was possible, but unlikely, and that we should therefore assume the second one was correct: “It doesn’t change anything, anyway: if the time available to us for the Universe to explode and implode is infinite, then however low the probability of two identical explosions occurring might be, it’s impossible for it not to happen, sooner or later. As that godless Dawkins puts it:

Given infinite time, or an infinite number of opportunities, anything is possible. 3

By that same principle, taking for granted that sooner or later this Universe will come to exist again, it’s entirely reasonable to think — not out of faith, but on the basis of a simple probabilistic calculation — that whatever is in it, too, may sooner or later come to exist again. Ourselves included.”
