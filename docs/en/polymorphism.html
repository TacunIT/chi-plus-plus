Polymorphism

When Me they fly, I am the wings
I am the double and the int

As you’ve probably already guessed from everything we’ve said so far, C++’s main distinguishing feature is polymorphism.

We started talking about it during the introductory lesson on C++, and illustrated it with an example that, by this point, should hold no more secrets for you:

/**
 *  @file src/cplusplus-template.cpp
 *  Example use of class templates.
 */

#include <iostream>
#include <ctime>
#include <cstring>
#include <list>

using namespace std;

/** Defines two new data types */
typedef time_t Date;
typedef enum _sex {
    male = 'm',
    female = 'f'
} Sex;

/** Defines a base class for handling animals */
class Animal {
private:
    string _breed;
    Sex    _sex;
public:
    /** Copy and parametric constructors */
    Animal() {}
    Animal(const char* breed, const Sex sex ) {
        _breed  = breed;
        _sex  = sex;
    }
    /** Pure virtual function: makes the class "abstract" */
    virtual const char* getSpecies() const {
        return "";
    }
    /** Interface functions */
    const char getSex() const {
        return (char)_sex;
    }
    const char* getBreed() const {
        return _breed.c_str();
    }
};

/** Stream output operator for the Animal class */
ostream& operator << (ostream& os, const Animal& animal) {
    os  << "Species:" << animal.getSpecies() << "\t"
        << "Breed:"  << animal.getBreed()  << "\t"
        << "Sex:"  << animal.getSex()
        << endl;
    return os;
}

/** Definition of the derived class Horse */
class Horse : public Animal {
public:
    /** Definition of the class's constructors */
    Horse() {}
    Horse(const char* breed, const Sex sex )
    : Animal(breed, sex ) {
    }
    /** Redefinition of the pure virtual function */
    const char* getSpecies() const {
        return "Horse";
    }
};

/** Definition of the derived class Donkey */
class Donkey : public Animal {
public:
    /** Definition of the class's constructor */
    Donkey() {}
    Donkey(const char* breed, const Sex sex )
    : Animal(breed, sex ) {
    }
    /** Redefinition of the pure virtual function */
    const char* getSpecies() const {
        return "Donkey";
    }
};

/** Definition of the Mating class */
class Mating {
private:
    Animal*  _male;
    Animal*  _female;
    Date     _day;
    string   _outcome;
    /**
    *   Private function for determining
    *   the outcome of the mating
    */
    void setOutcome() {
        if(strcmp(_male->getSpecies(),"Donkey") == 0) {
            if(strcmp(_female->getSpecies(),"Donkey") == 0) {
                _outcome = "donkey";
            } else {
                _outcome = "mule";
            }
        } else {
            if(strcmp(_female->getSpecies(),"Horse") == 0) {
                _outcome = "foal";
            } else {
                _outcome = "hinny";
            }
        }
    }
public:
    /** Class constructor */
    Mating(Animal* male, Animal* female) {
        _male = male;
        _female = female;
        time(&_day);
        setOutcome();
    }
    /** Output operator, "friend" of the Mating class */
    friend ostream& operator<<(ostream& os, const Mating& mating) {
        os << "DATE:    " << asctime(localtime(&mating._day))
           << "MALE: " << *mating._male
           << "FEMALE: " << *mating._female
           << "OUTCOME:   " << mating._outcome
           << endl;
           return os;
     };
};

int main()
{
    /**
     *  Creates the derived class objects
     *  and assigns them to base class pointers.
     */
    Animal* horse  = new Horse("Lipizzaner", male);
    Animal* mare = new Horse("Maremmano", female);
    Animal* donkey    = new Donkey("Amiata", male);
    Animal* jenny    = new Donkey("Sardinian", female);

    /** Creates a list using a template class */
    list<Mating> matings;

    /** Adds Mating class objects to the list */
    matings.push_back(Mating(horse, mare));
    matings.push_back(Mating (donkey, jenny));
    matings.push_back(Mating (donkey, mare));
    matings.push_back(Mating (horse, jenny));

    /** Displays the contents of the list */
    list<Mating>::iterator it;
    for (it=matings.begin(); it!=matings.end(); it++) {
        cout << *it << endl;
    }

    return 0;
}

Before going on, though, we need to clear up three terms related to polymorphism: overload, override, and redefinition. By the term overload of a function, we mean a function that has the same name as another one, but different parameters. A typical example of function overload is the different versions of a class’s constructor:

Cavallo() {}
Cavallo(const char* razza, const Sesso sesso )
: Animale(razza, sesso ) {
}

The two functions have the same name, and the compiler will pick one or the other depending on the parameters used. An overridden function is a function with a different definition from that of a virtual function in one of its base classes:

const char* getSpecie() const {
    return "Asino";
}

As we’ve seen, the compiler picks one or the other depending on the type of the object used in the call. If the base class’s function hadn’t been virtual, this would have simply been a redefinition:

class Persona {
public:
    void getClass(){
        cout << "Persona" << endl;
    }
};

When it handles these functions, the compiler doesn’t perform a dynamic type check based on the object’s type at runtime, but picks the function to call based on the type of the pointer or reference used, which, as you know, can cause problems:

Madre   * ptrM = new Madre;
Persona * ptrP = ptrM ;
ptrM->getClass() ;
ptrP->getClass() ;  // chiama la funzione di Persona - ERRORE

In C++, every operator corresponds to a function. The one for the += binary operator, for example, is:

<tipo>& operator +=  (<tipo>& a, <tipo>& b) ;

where a and b are the two objects involved in the operation, and <tipo> is the type of the variables involved in the operation.

int&    operator +=  (int&    a, int&    b) ;
float&  operator +=  (float&  a, float&  b) ;
double& operator +=  (double& a, double& b) ;

Since unary operators can be prefix or postfix, to let the compiler tell which is the correct function to use, the postfix operator’s function gets an extra, unused, second parameter:

void operator ++ (<tipo> a) ;           // versione prefissa
void operator ++ (<tipo> a, <tipo>) ;   // versione postfissa

Overloaded operator functions can be called directly. The two instructions below, once compiled, produce the same code and the same result. If you can find any reason to use the first syntax instead of the second, go right ahead:

a = b.operator + (c) ;  
a = b + c ;

The behavior of operators is predefined for all standard types, and can be redefined to also handle aggregate data types like structs or classes. C++’s standard library string class, for example, redefines, among other things, the behavior of the assignment operators += and + and of the stream output operator <<, so that operations on strings can use the same syntax used for other data types:

/** 
 * @file src/polimorfismo-operatori.cpp
 * Esempio di overload di un operatore.
 */

#include <iostream>
#include <string>

using namespace std;

int main ()
{
    string s1 ("Pip");
    string s2 ("po");
    const char* s3 = "Plut";

    /** 
     *  La classe string definisce tre overload 
     *  per l'operatore += .
     *
     *  string& operator+= (const string& str);
     *  string& operator+= (const char* s);
     *  string& operator+= (char c);
     */
    s1 += s2;
    s1 += s3;

    cout << (s1 + 'o') << endl;

    return 0;
}

You could get the same result with the append function:

string& append (const string& str)

but using a standard operator makes the code easier to read and to write, if only because you don’t have to remember what the function for joining two strings is called. The only operators that cannot be redefined by a class are:

  • the selection operator .;
  • the pointer-to-member resolution operator .*;
  • the scope resolution operator ::;
  • the conditional operator ? :;
  • the symbols # and ##, used by the preprocessor.

Except for a few exceptions we’ll see shortly, every C++ operator can be redefined either as a class’s member function or as a global function:

/** 
 * @file src/polimorfismo-in-out.cpp
 * Operatori come funzioni membro o globali.
 */

#include <iostream>
#include <string>

using namespace std;

struct A
{
    int _a;
    A(int a) : _a(a) {}
};

struct B
{
    int _b;
    B(int b) : _b(b) {};   
    /** Overload come funzione membro */
    int operator + (const A& a) {
        return _b + a._a;
    }
};

/** Overload come funzione globale */
int operator + (const A& a, const B& b) { return a._a + b._b; }

int main ()
{
    struct A a(3);
    struct B b(5);
    
    cout << (a + (b + a)) << endl;

    return 0;
}

When you redefine an operator’s behavior for a class, you need to take into account the visibility of the member data it has to use. If the operator, as is likely, needs to handle private or protected data, there are two options: either make use of the class’s interface functions, or declare the operator a friend of the class. In the initial example, both options are used: the stream output operator for the Animale class uses the class’s interface functions:

ostream& operator << (ostream& os, const Animale& animale) {
    os  << "Specie:" << animale.getSpecie() << "\t"
        << "Razza:"  << animale.getRazza()  << "\t"
        << "Sesso:"  << animale.getSesso()  
        << endl;
    return os;   
}

while the output operator for the Monta class is declared a friend of the class, and can therefore access the instance’s member data directly:

friend ostream& operator << (ostream& os, const Monta& copula) {
    os << "DATA: "    << asctime(localtime(&copula._giorno))
       << "MASCHIO: " << *copula._maschio
       << "FEMMINA: " << *copula._femmina;
       return os;   
};

The choice between the two options depends on the kind of program you need to write: if you’re after speed, choose the second one, which is more direct; otherwise choose the first, which will probably run slower, but won’t need rewriting if the class’s structure changes. It isn’t possible, though, to redefine as a class’s member function an operator function whose first parameter is a class you don’t control (such as, for example, the << operator function, whose first parameter is a reference to ostream), because in a member function this parameter would be replaced by the implicit this parameter, which has a different data type, causing a compile error. The operators =, (), [], and -> cannot be redefined as global functions, and must always be implemented as a class’s non-static member function. The other rules to remember, in these cases, are:

  • the unary assignment operator = is the only case of a member function that isn’t inherited by any child classes; if it isn’t redefined, it performs a member-by-member assignment of the attributes, and has the syntax:

      C& C::operator = (const C& origine) ;

  • the binary operator [] lets you implement vectors of a particular kind while keeping a standard syntax, and has the form:

      c.operator [] (n) ;

    where c is an object of class C and the index n can be of any data type;

  • to redefine the binary function-call operator (), you use the syntax:

      c.operator()(p) ;

    where c is again an object of class C, and p is a list, possibly empty, of parameters;

  • the unary class-member access operator -> is interpreted as:

      (C.operator -> ())->m ;

    and returns either an object or a pointer to an object of class C.

Redefining the new and delete operators, whose behavior is closely tied to the hardware, might not be a clever choice from the point of view of code portability; that said, if a class needs to manage memory in a particular way, it can do so, but must follow two rules:

  • the new operator must have its first argument of type size_t, and return a pointer to void;
  • the delete operator must be a void function with a first argument of type void*, and a second, optional, argument of type size_t.

In C, to convert an int into a double, you use the cast operators:

long int i = 5 ;
double d = (double) i ;

C++ accepts this syntax, just as it accepts using malloc instead of new, but its standard syntax, which vaguely resembles class constructors, requires the data to be converted to be passed as a parameter to a function with the same name as the type you want it converted into:

long int i = 5 ;
double d = double(i) ;

The C++ compiler is able to convert any primitive data type into another, but it has no way of knowing how to handle user-defined data types; we therefore have to instruct it, just as we did with constructors and operators, by defining coercion paths from and to primitive data types. The first case, that is, converting from a primitive type to a user-defined one, is the simplest: it essentially means defining, where one doesn’t already exist, a constructor for the new class that takes parameters of primitive type. When instead there’s no constructor to extend — that is, when the coercion goes from a user-defined type to a primitive data type, or one provided by a library whose source code you don’t have — you need to redefine the () conversion operator. Imagine you’ve created a new data type, Frazione, to handle rational numbers. To be able to use it in expressions containing primitive-type data, you would have to redefine every operator to accept mixed-type data, both as the first and as the second parameter:

Frazione operator + (int i, Frazione f) ;
Frazione operator - (int i, Frazione f) ;
Frazione operator + (double i, Frazione f) ;
Frazione operator - (double i, Frazione f) ;
...
Frazione operator + (Frazione f, int i) ;
Frazione operator - (Frazione f, int i) ;
Frazione operator + (Frazione f, double i) ;
Frazione operator - (Frazione f, double i) ;

You can save yourself this hassle by only redefining the operators’ behavior for the new class, and giving the compiler conversion paths from primitive types to the new data type, so that it can convert the data to the right type in the case of mixed expressions:

/** 
 * @file src/polimorfismo-cast.cpp
 * Gestione della conversione esplicita.
 */

#include <iostream>

using namespace std;

class Frazione
{
private:
    
    int _num ;
    int _den ;

public:
    
    /** Costruttore con parametri interi */
    Frazione(int n, int d = 1) 
    : _num(n), _den(d) {} 
    
    /** 
     * Costruttore con parametro a virgola mobile.
     * La definizione è piuttosto complessa, te la risparmio.  
     */
    Frazione(double d) ;

    /** Overload dell'operatore di cast a intero */
    operator int () { 
        return _num / _den ; 
    }

    /** Overload dell'operatore di cast a double */
    operator double() { 
        return (double) _num / (double) _den ; 
    }

    /** Overload degli operatori di somma e sottrazione */
    friend Frazione operator+ (Frazione f1, Frazione f2); 
    friend Frazione operator- (Frazione f1, Frazione f2); 

};

The last thing I need to tell you about, regarding polymorphism, is templates. In C++, templates are blueprints used to define multi-purpose functions or classes. If the same task can be carried out in a similar way on parameters of different types, instead of writing identical functions or classes for each parameter type, you can write a single template function or class and call it each time with the appropriate parameter type:

int    somma(int    a, int    b) { return a + b; }
float  somma(float  a, float  b) { return a + b; }
double somma(double a, double b) { return a + b; }

template <class T>
T somma(T a, T b) { return a + b; }

When the compiler finds a template in the code, whether it’s a class declaration or a function call, it replaces it with the corresponding code, the same way it does for precompiler macros, but, unlike what happens with macros, the type of the template’s parameters is subject to strict checking, just like the rest of the code. The format for declaring a template function is:

template <class identificatore> dichiarazione;
template <typename identificatore> dichiarazione;

There’s no difference between the first and second form: both class and typename produce the same effect. identificatore is a symbol identifying a given data type, or a user-defined class. For example, the syntax for a template function that returns the greater of two parameters would look something like this:

template<class T>
T maggiore (T x, T y) {
    return (x > y) ? x : y;
}

In this case, the type identifier is the letter T, which appears both between the angle brackets on the first line and in the parentheses on the second, but it can be any string. There can be more than one parameter:

template<class C1, class C2>
T funz (C1 x, C2 y) {
...
}

and they can have a default value:

template<class N = int>
T funz (N n) {
...
}

Calling template functions is similar to calling ordinary functions, with the addition of the type of the parameters to be handled:

cout << maggiore<int>   (  9,  12) << endl;    
cout << maggiore<double>(0.4, 1.2) << endl;    
cout << maggiore<char>  ('a', 'z') << endl;

The next example shows the difference between a precompiler macro and a template function: 

/** 
 * @file src/polimorfismo-macro-template.cpp
 * Funzioni template e macro precompilatore.
 */

#include <iostream>
#include <functional>

using namespace std;

/**
*  Definzione di una macro istruzione per il 
*  precompilatore: nessun controllo di tipo.
*/
#define MAGGIORE(a,b) ((a > b) ? a : b) 

/**
*  Definzione di una funzione template
*  che torna il maggiore fra due parametri.
*/
template<class T>
T maggiore (T x, T y) {
    return (x > y) ? x : y;
}

int main ()
{     
    int    a = 10;
    short  b = 0;   
    double d = 3.123456789;
    
    /** Utilizzo della macro */
    cout << MAGGIORE(9,12)     << endl;    
    cout << MAGGIORE(0.4, 0.7) << endl;    
    cout << MAGGIORE('a', 'z') << endl;    

    /** 
    *  La stessa funzione si può utilizzare 
    *  con tipi di dato diversi:
    */
    cout << maggiore<int>   (  9,  12) << endl;    
    cout << maggiore<double>(0.4, 1.2) << endl;    
    cout << maggiore<char>  ('a', 'z') << endl;    
   
    /** Errore: confronta un carattere con un double */
    cout << MAGGIORE('a', d) << endl;    

    /**
    *   Errore: il compilatore non sa quale
    *   dei due tipi di dato utilizzare.
    */
    cout << maggiore(a, b) << endl;    
    
    return 0;
}

The MAGGIORE macro and the maggiore template function perform the same operation: they compare the two parameters they receive as input, and return the greater of the two. The big difference between these two approaches is that, while the type of the template’s parameters is checked by the compiler, the macro is a plain substitution that performs no checks at all on the variables it uses. The instruction:

cout << MAGGIORE('a', b) << endl;

compares a character with a double and, without causing any compile problems, returns the value 97, corresponding to the ASCII code of the letter a. By contrast, the instruction:

int   a = 10;
short b = 0;
cout << maggiore(a, b) << endl;

causes a compile error, because the two parameters are of different types:

The declaration of a template class has the form:

template <class identificatore> dichiarazione;

The list of parameters between the <> symbols can contain one or more symbols for the data types the class handles. Using these classes is similar to using template functions:

/** 
 * @file src/polimorfismo-classe-template.cpp
 * Esempio di classe template.
 */

#include <iostream>

using namespace std;

/**
*   Definisce una classe che gestisce coppie 
*   di coordinate.
*/
template<class T>
class Coord {
private:
    /** Dati membro con tipo variabile */
    T _x, _y; 
public:
    /** Costruttore con tipo di parametri variabile */
    Coord(const T x, const T y) 
    : _x(x), _y(y) {        
    }
    friend ostream& operator << (ostream& o, const Coord& c) {
        o << c._x << ',' << c._y ;
        return o;
    }
};

int main ()
{         
    /** Istanza con coordinate geografiche */
    Coord<double> obelisco(41.903219, 12.458157);
    
    /** Istanza con coordinate schermo */

The code I showed you at the start of this lesson uses a template class:

list<Monta> monte;

The list class is one of the classes of C++’s Standard Template Library, a library of classes and functions that let you solve common programming problems, such as storing, sorting, or searching a set of data. The components of the STL are:

  • a library of containers, which let you store objects and data;
  • iterators, which let you move through a container’s contents;
  • a collection of algorithms, which let you perform sorting and searching operations on sets of data;
  • function-objects, or functors, which encapsulate a specific function.

The list class is an example of a container, and represents a list of elements stored in non-contiguous areas of memory. By contrast, the vector class implements a list of elements stored in a single area of memory, the same way C arrays work. All STL vectors have member functions that let you manage their elements; the push_back function, for example, adds an element to the end of the list:

monte.push_back(Monta(cavallo, giumenta));
monte.push_back(Monta (asino, asina));       
monte.push_back(Monta (asino, giumenta));     
monte.push_back(Monta (cavallo, asina));

Iterators are constructs that let you move through a container’s contents, picking out its elements. We used one in the instruction:

list<Monta>::iterator it;
for (it=monte.begin(); it!=monte.end(); it++) {
    cout << *it << endl;
}

The first instruction in the for loop assigns to the iterator it the first element of the list, returned by the member function monte.begin. The second instruction checks that the iterator is different from monte.end, which points to the end of the list. The third instruction increments the iterator by one position, and shows how redefining an operator for a class makes the code easier to read: even if you’ve never seen a template class before, you can immediately tell that instruction increments the value of it by one. The algorithms of the STL, defined in the <algorithm> header, are template functions that let you locate, copy, sort, merge, or remove data within a container.

/** 
 * @file src/polimorfismo-algoritmi.cpp
 * Esempio di utilizzo degli algoritmi della STL.
 */

#include <iostream>
#include <vector>
#include <algorithm>

using namespace std;

int main()
{
    /** Crea il vettore */
    vector<int> vect;

    /** Aggiunge dei valori al vettore */
    vect.push_back(10);
    vect.push_back(70);
    vect.push_back(21);
    vect.push_back(49);
    vect.push_back(35);

    /** Mostra il valore più alto e il più basso */
    cout << *min_element(vect.begin(), vect.end()) << endl;
    cout << *max_element(vect.begin(), vect.end()) << endl;
    
    /** Mostra i valori nell'ordine in cui sono stati inseriti */
    vector<int>::iterator i;
    for(i = vect.begin(); i != vect.end(); i++) {
        cout << *i << " ";
    }
    cout << endl;

    /** Ordina i valori dal più basso al più alto e li visualizza */
    int n = vect.size();
    sort(vect.begin(), vect.end());
    for (int i=0; i<n; i++) {
        cout << vect[i] << " ";       
    }
    cout << endl;

    /** Ordina i valori dal più alto al più basso e li visualizza */
    reverse(vect.begin(), vect.end());
    for (int i=0; i<n; i++) {
        cout << vect[i] << " ";

Function-classes, or functors, are classes that redefine the behavior of the () operator, and can therefore act as if they were functions:

/** 
 * @file src/polimorfismo-functor-stl.cpp
 * Esempio di function objects della STL.
 */

#include <iostream>
#include <functional>

using namespace std;

int main ()
{
    int a = 12;
    int b = 4;

    /** Dichiarazione di oggetti functor */
    plus<int>       p;
    minus<int>      m;
    multiplies<int> x;
    divides<int>    d;
    modulus<int>    o;

    /** Utilizzo degli oggetti come fossero delle funzioni */
    cout << "plus: "       << p(a,b) << endl;
    cout << "minus: "      << m(a,b) << endl;
    cout << "multiplies: " << x(a,b) << endl;
    cout << "divides: "    << d(a,b) << endl;
    cout << "modulus: "    << o(a,b) << endl;

    /** "esegue"" l'oggetto o con nuovi parametri */
    cout << "modulus: "    << o(a,5) << endl;

    return 0;
}

Used this way, functors don’t make much sense, but they can be (and are) very useful when using those STL functions that process every element of a container, such as the transform function:

/** 
 * @file src/polimorfismo-rot13.cpp
 * Trasformazione di una stringa con una funzione.
 */

#include <string>
#include <cctype>
#include <iostream>
#include <functional>

using namespace std;

/** 
*   Funzione che converte i caratteri di una stringa in rot13  
*   Il ROT13 è un algoritmo di cifratura piuttosto banale, 
*   perché incrementa di 13 il valore di ciascun carattere. 
*   Non è un algoritmo realmente sicuro, però, perché per 
*   decifrare il testo crittografato basta crittografarlo 
*   di nuovo.  
*/
unsigned char rot13(unsigned char c) 
{ 
    unsigned char rot = c;
    if (isalpha(c)) {
        rot = ((tolower(c) - 'a') < 14) ? c + 13 : c - 13;
    }
    return rot;
}

int main ()
{
    string pp("PippoPluto"); 

    /** Elabora la stringa con la funzione transform */
    transform(
               pp.begin()   // inizio del container da modificare
             , pp.end()     // fine del container da modificare
             , pp.begin()   // container di output
             , rot13        // funzione da applicare
             );

    cout << pp << endl; 

    /** Ripetendo l'operazione, il testo torna normale. */
    transform(
               pp.begin()   
             , pp.end()     
             , pp.begin()   
             , rot13        
             );

    cout << pp << endl; 
    
    return 0;
}

This is exactly where functors come in handy, because they can be initialized with one or more specific values and then used as unary functions:

/** 
 * @file src/polimorfismo-functor.cpp
 * Creazione di una classe functor.
 */

#include <string>
#include <cctype>
#include <iostream>
#include <functional>

using namespace std;

/** Dichiarazione della classe functor */
class Cifra
{
private:

    int _chiave;

public:
    
    /** 
    *   Il costruttore della classe ha come parametro
    *   il valore della chiave di cifratura
    */
    Cifra(int chiave) : _chiave(chiave) {  }
  
    /** Ridefinizione dell'operatore () */
    unsigned char operator () (unsigned char c) const {
        return c + _chiave;
    }
};

int main ()
{
    string pp("PippoPluto"); 

    /** 
    *   Richiama transform passando come parametro
    *   un'istanza del functor, inizializzata con 
    *   la chiave di cifratura.
    */
    transform(
               pp.begin()   
             , pp.end()     
             , pp.begin()   
             , Cifra(1)        
             );

    cout << pp << endl; 
    
    return 0;
}

For thousands of years, humans have tried to understand what the meaning of Existence is. The answers they’ve come up with vary depending on the historical period and the region where the prophet or philosopher lived, but they all share one peculiarity: they require their followers to accept unprovable postulates, such as the existence of one or more divinities, or of states of existence different from the one we know. Science too has tried to answer these same questions, but its inquiry has been limited to the practical aspects of the problem: it has produced interesting theories about the origin of the Universe and the events that led to our existence, but it has never taken a position on what our role in all this might be, with the consequences we discussed during the lesson on memory. Master Canaro, who couldn’t accept either the dogmas of traditional religions or the disconnect between man and the Universe produced by scientific hypotheses, asked himself a question:

Is it possible to explain Existence using only what we have direct experience of?

Most religions, to “work”, require between one and three additional dimensions, beyond the known ones; Science, for its superstrings, needs at least seven additional dimensions — twice as many as are needed for a non-spiritual Afterlife. Is there a simpler explanation? Being neither a philosopher nor a mystic, he approached the development of his doctrine as if it were a piece of software. First he analyzed the “system in production”, highlighting its main flaws; then he identified logical vulnerabilities in canonical religions and defined guidelines meant to prevent them; finally, he described the features of C’hi++, explaining how they could solve some of the problems he’d identified earlier. As he wrote in the Proposal, there are “bugs” we can consider common to all metaphysical systems:

  • dogmas, which are the foundations of doctrines, are easily attacked because they cannot be proven, only accepted on faith;
  • a religion may have difficulty modifying its own doctrine, even when it's evident that one of its dogmas is wrong;
  • challenging a dogma almost inevitably causes a split, and splits are likely to lead to conflicts.

and others, which we can consider common to human beings:

  • the tendency to defend one's own principles even by means that contradict those very principles;
  • the tendency to let one's hopes influence one's objectivity.

To fix, or at least mitigate, these problems, his metaphysics needed to:

  • limit the number of dogmas;
  • limit metaphysical elements and matters accepted on faith;
  • do not present itself as the One Unquestionable Truth, but as a certainly incomplete and perfectible approximation of the Truth;
  • recognize the contradictions of the doctrine and analyze them objectively, even if this leads to modifying the doctrine itself.

Master Canaro applied to the development of his non-metaphysical metaphysics the same approach he used when he had to build a piece of software. There are two different ways to design software: the first is to analyze every system that carries out similar tasks, take the best of each, and put it into the new system; alternatively, you can design the system from scratch, and only once its overall structure is defined, study the solutions adopted by others, folding them into your own program if you think they’re useful. The first approach is faster and safer, but tends to produce derivative results; the second approach is more complex, both in terms of analysis and implementation, but makes innovation easier, because the analyst’s imagination isn’t constrained by what they’ve already seen. Being a proponent of the second method, Master Canaro applied it to C’hi++ too, and, after some years of study, concluded not only that it’s possible to hypothesize a cosmogony that’s almost entirely free of metaphysical elements (not entirely free, because, as we’ll see later on, a minimal dose of transcendence is needed to guarantee the doctrine works properly), but that the precepts of this doctrine were compatible with many principles of canonical religions.

C’hi++ inherited some concepts from philosophies Master Canaro already knew, like the Gravity/Electricity dualism developed by Poe in Eureka, which had fascinated him for the way it turned a blind and inexplicable force like Gravity into the conscious intention, shared by everything that exists, to become One again. On the other hand, Master Canaro’s doctrine rejected some concepts common to many religions, such as the possibility of escaping the cycle of rebirth, or the presence of ad-personam punishments or rewards. Just as when you analyze how a piece of software works you don’t concern yourself with individual variables, but think about the overall flow of the system, so C’hi++ sees existence not in terms of interactions between individuals, but as the evolution of the flow of the One’s Energy within the three-dimensional matrix of the spazioni. For C’hi++ there are no souls, no infernal rivers, and whoever dies at sea won’t be welcomed by Rán, in her tavern at the bottom of the sea, but will simply be recycled, like RAM memory areas inside a computer. Our existences are incidental; the idea of punishing or rewarding them would make no sense, and would clash with the general principle that everything that exists is the manifestation of a single Entity. As I told you at the beginning of these lessons, it isn’t possible to go to Heaven or Hell alone: whatever happens in the Universe concerns all of us. This, however, doesn’t mean C’hi++ rejects every concept from the religions that came before it; quite the opposite. Between C’hi++ and canonical religions there’s the same difference as between a topographic map and a satellite image. That godless Dawkins once said:

One of the marks of foolish extravagance is an overly enthusiastic use of analogy.2

A curious statement, coming from a member of a sect that tries to describe everything that exists through mathematical analogies, and denies the existence of whatever it can’t convert.. Enthusiasm aside, maps and satellite images have several things in common with metaphysical disciplines. Maps and images too, like metaphysics, are forced to represent their subject at a reduced scale, and with only two dimensions instead of three (or four, if, besides depth, you also want to factor in time). For this reason, maps and images too have to represent their subject by means of analogies: topographic maps use contour lines and symbols; satellite images use pixels or tiny colored dots. In neither case is what we see actually what it represents; it’s our brain that decides to believe it is: in the case of the topographic map, because the legend lets us establish a correlation between meaning and symbol; in the case of the image, because our eye recognizes trees, the sea, or houses in those combinations of pixels or colored dots. Another analogy, a consequence of the two points above, is that it’s a mistake to confuse symbols with what they represent: the little dark squares on maps are not houses; the colored dots in the images are not a forest. Maps and images only make sense at a certain level of reading; if we go beyond it, if we try to get more information or more realism by looking closer, we get the opposite effect, because the symbols reveal themselves for what they are: colored dots or lines on a sheet of paper. This doesn’t mean that what they represent is false, though, only that we aren’t looking with the right pair of eyes, as Hunter Thompson would say. Master Canaro thought this was the reason some religions oppose the direct depiction of the Divinity: because it’s easy to end up confusing the symbol with what it represents. Going back to our original comparison, traditional religions are satellite images, while C’hi++ is a topographic map. While religious Creeds manage to reproduce — within the limits imposed by our condition — all the beauty of Creation, C’hi++ limits itself to giving a schematic description of it, poorer in content and in poetry, but easier to accept for those who don’t have the blessing of Faith. A satellite image has contemplative value: it’s beautiful to look at on your computer screen, or even to hang on a wall, like a painting, but if you’ve gotten lost in a forest or in the middle of the mountains, a topographic map, precisely because of its schematic nature, will more easily let you find your way home. C’hi++ doesn’t try to steal followers away from canonical religions. That would make no sense: it would be like trying to convince someone already married to the love of their life to enter a marriage of convenience instead — if you have Faith, you don’t need rational confirmations; they might please you, but you don’t need them. What C’hi++ can do, though, is give strength to those (many) people who still believe in everything no one believes in anymore, as Longanesi described them; that Band of Honest Men3 who, every day, do their duty as best they can even when it doesn’t pay off, even when everything and everyone around them seems to push them toward selfishness and indifference. It can help them not to give up, and can teach them that winning isn’t what matters — playing ever better is. Recognizing the mistakes you’ve made, learning from them, and trying not to repeat them, game after game, in a continuous search for the best you can do. If they behave this way, whatever their job is, even if it’s cleaning toilets, it will still be Art.
