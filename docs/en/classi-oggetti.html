Classes and objects

Ceci n'est pas une |

The ability to define new data types by means of classes is C++’s main distinguishing feature.

“Traditional” programming languages, like Cobol, Fortran, or Pascal, have a limited set of data types: integers, floating-point numbers, booleans, characters, and strings.. just what you need to manage a personal record or a bank account. C and Pascal also let you group this data into structs, enums, or arrays, but these are just containers, with no internal logic. Moreover, as you’ve seen, the data inside a struct is accessible to any component of the program, so if you change it, the code that uses it has to change too. Imagine defining a struct to manage the time of day, containing three integers, one for hours, one for minutes, and one for seconds:

struct Orario {
    int h;
    int m;
    int s;
};

To use this struct, you need to know its contents and the relationship between one value and another; in particular, you need to know (and remember):

  • that the variable h can only hold values from 0 to 23;
  • that the value of m can only hold values from 0 to 59;
  • that the value of s can only hold values from 0 to 59;
  • that if s exceeds 59, m must be incremented by 1;
  • that if m exceeds 59, h must be incremented by 1;

This is the opposite of the low coupling we talked about a while back, because it inextricably ties a function to the structure of the data it has to handle. To put it plainly: a function that updates the minutes would have to look something like:

void aggiornaMinuti(struct Orario &o, int minuti)
{
    /** Incrementa il numero dei minuti */
    o.m += minuti;

    /** Se necessario, incrementa le ore */
    if(o.m >= 60) {

        o.m -= 60;
        o.h += 1;

        /** Se necessario, passa al giorno dopo */
        if(o.h >= 24) {
            o.h -= 24;
        }
    }
}

If we ever decided to change the Orario struct, we’d have to remember to rewrite this function too, adapting it to the struct’s new features, at the cost of time and the risk of mistakes. Moreover, nothing would stop a careless programmer from writing a function that ignores the relationship between hours, minutes, and seconds entirely:

void incrementa_m(struct Orario &o, int minuti)
{
    o.m += minuti;
}

If we put these two functions into a program, we get:

/** 
 * @file classi-struttura-orario.cpp
 * Gestione dei dati membro di una struct.
 */
 
#include <iostream>
#include <iomanip> 

using namespace std;

/** Dichiarazione di una struttura per gestire un orario */
struct Orario {
    int h;
    int m;
    int s;
};

/** Funzione per l'incremento dei minuti */
void aggiornaMinuti(struct Orario &o, int minuti) 
{
    /** Incrementa il numero dei minuti */
    o.m += minuti;
    
    /** Se necessario, incrementa le ore */
    if(o.m >= 60) {

        o.m -= 60;
        o.h += 1;
        
        /** Se necessario, passa al giorno dopo */
        if(o.h >= 24) {
            o.h -= 24;
        }
    }
}

/*  Funzione cialtrona per l'incremento dei minuti */
void incrementa_m(struct Orario &o, int minuti) 
{
    o.m += minuti;
}

int main()
{    
    struct Orario ora;

    /** Definisce dei valori iniziali prossimi al cambio di data */
    ora.h = 23;
    ora.m = 45;
    ora.s = 00;

    /** Visualizza i valri iniziali */
    cout << setfill('0') << setw(2) << ora.h << ":" 
         << setfill('0') << setw(2) << ora.m << ":" 
         << setfill('0') << setw(2) << ora.s << endl;

    /** Richiama la funzione di aggiornamento */
    aggiornaMinuti(ora, 20);
    
    /** Visualizza i valori dopo l'aggiornamento */
    cout << setfill('0') << setw(2) << ora.h << ":" 
         << setfill('0') << setw(2) << ora.m << ":" 
         << setfill('0') << setw(2) << ora.s << endl;

    /** Reimposta i valori iniziali */
    ora.h = 23;
    ora.m = 45;
    ora.s = 00;

    /** Richiama la funzione cialtrona */
    incrementa_m(ora, 20);
    
    /** Visualizza i valori dopo l'aggiornamento */
    cout << setfill('0') << setw(2) << ora.h << ":" 
         << setfill('0') << setw(2) << ora.m << ":" 
         << setfill('0') << setw(2) << ora.s << endl;

    
    return 0;
}

Compiling and running this code, you get:

> ./src/out/esempio                                          
23:45:00
00:05:00
23:65:00

As you can see, the first function updated the data correctly, while the second produced an invalid value with no way for the program to catch it.

Variables inside a class are called the class’s member data or attributes; functions, on the other hand, are called member functions or methods. When you create a variable of class X, we say you’re instantiating an object of class X, or creating an instance of the class. A class’s member data and functions are directly accessible to the class’s own member functions, but to use them from functions outside the class, you have to use the selection operators . and ->. The first, called the direct selection operator, is used with instances of the class; the second, called the indirect selection operator, with pointers to them:

/** 
 * @file classi-punto.cpp
 * Accesso ai dati membro della classe.
 */
 
#include <iostream>
#include <iomanip> 

using namespace std;

class Punto
{
public:             
    
    /** Dichiara i dati membro della classe */
    int _x, _y;

    /** 
     *  Le funzioni interne alla classe accedono ai
     *  dati membro con la sintassi ordinaria.
     */  
    Punto(int x, int y) {
        _x = x;
        _y = y;
    }
} ;

int main(int argc, char** argv) 
{
    /** Crea un oggetto di classe Punto */
    Punto p(5,6);       

    /** Assegna l'istanza della classe al puntatore ptr */
    Punto *ptr = &p ;   
    
    /** 
     *  Le funzioni esterne alla classe accedono ai
     *  dati membro tramite gli operatori di selezione.
     */  
    p._x    = 3 ;       // assegna un valore tramite l'oggetto
    ptr->_y = 2 ;       // assegna un valore tramite il puntatore
    
    //...
}

The public label you see at the start of the class declaration is an access specifier, used to establish which members of the class are accessible to outside functions and which are reserved exclusively for the class itself.

the savage does not like to say his own name, or to be photographed, because through his name or his portrait he becomes accessible, and can therefore be harmed by anyone able to reach him through these means.1

This line of Lucien Lévy-Bruhl’s applies to C++ classes too. Both a class’s attributes and its methods can be protected from unwanted access or changes thanks to the access modifiers private, protected, and public. Methods or attributes declared private are accessible only to the class itself; those declared protected are accessible to the class and to any derived classes; those declared public are accessible to any part of the program. In the absence of specifiers, all of a class’s data and functions are considered:

  • private, in the case of a class;
  • public, in the case of a struct or union.

The visibility of a struct’s member data can be changed with access specifiers; the data of a union, on the other hand, can only be public. C++ lets you split a class’s declaration into as many sections as you like, in whatever order of private, public, protected you prefer, but code written this way is certainly harder to read than code where all the private members sit on one side and all the public ones on another. So, unless you have good reasons to do otherwise (and there could be some, in the case of particularly complex classes), try to group all the functions and member data with the same access attributes into just three sections: private, protected, and public:

class Persona
{
 private:
    ...
 protected:
    ...
 public:
    ...
};

This kind of ordering in the declaration, besides giving you more readable code, lets you, if you want, omit the initial private access specifier (it’s the default, remember?). Let’s put all this into practice by converting the Orario struct into a class:

class Orario {
private:
    int _h;
    int _m;
    int _s;
public:
    Orario() {
        _h = 0;
        _m = 0;
        _s = 0;
    }
};

The declaration begins with the class keyword, followed by the class’s name. The block of code in curly braces that makes up the body of the class contains the member data and functions, grouped by visibility. In this case we put the private member data first and the public ones after, but we could just as well have done the opposite. The _h, _m, and _s attributes appear after the private keyword, and will therefore be visible only to the class’s own functions. The Orario function appears after the public label, and will therefore be accessible from any part of the program. This function, which has the same name as the class, is called the constructor, and is called every time a variable of type Orario is created. Its purpose is to initialize the variables inside the class, in this case, setting all three values to 0. We’ll talk about it shortly. The last thing you need to notice, in the code above, is the presence of the ; character at the end of the class’s code block, just as with unions and structs.

When we declare a variable of a primitive type like int or double, the compiler automatically carries out a whole series of operations to allocate the memory space needed to hold it, and to initialize it. The compiler, however, doesn’t know how a variable of type Orario should be created and initialized, which is why the class has to define management functions explaining both how to create a new variable and how to destroy it, if needed. Management functions are of two kinds: constructors and destructors. Constructors have a few peculiarities that set them apart from other member functions:

  • they have the same name as the class;
  • they have no return type, because it’s implicit that they return a variable of the class they belong to.

A single class can have more than one constructor; the Orario class, for example, could have a constructor with no parameters, which initializes hours, minutes, and seconds to zero, and a constructor that instead lets you assign specific values to each attribute:

class Orario {
protected:
    int _h;
    int _m;
    int _s;
public:
    Orario() {
        _h = 0;
        _m = 0;
        _s = 0;
    }
    Orario(int h, int m, int s)
    : _h(h % 24), _m(m % 60), _s(s % 60) {
    }
};

The line:

: _h(h % 24), _m(m % 60), _s(s % 60)

is called an initialization list, and is equivalent to writing:

_h = h % 24;
_m = m % 60;
_s = s % 60;

Using the modulo operator % here is essential, to prevent incorrect values from being assigned to the variables. When you define a constructor, you can use either syntax, or even mix them, depending on the case. A more concise way of getting the same result with a single constructor is to use default values for the parameters:

Orario(int h = 0, int m = 0, int s = 0)
: _h(h % 24), _m(m % 60), _s(s % 60) {
}

Sometimes it can be useful to define a constructor that creates new variables of the class starting from existing variables, effectively performing a kind of cloning. This kind of function is called a copy constructor or initialization constructor, and requires, as an argument, a reference to a variable of the same class:

/**
 * Dichiarazione del costruttore di copia
 * all'interno della classe.
 * Possiamo copiare il valore delle variabili
 * così com'è perché è già stato verificato
 * dal costruttore della variabile o1.  
 */
Orario::Orario(const Orario& o)
: _h(o._h), _m(o._m), _s(o._s) {
}

/** Utilizzo */
Orario o2 = o1;

The copy constructor is a very important kind of constructor, since it presides over most of the initialization activity for objects of the class it belongs to; for this reason, if it isn’t defined by the user, it’s automatically generated by the compiler.

As you can probably guess, while a class’s constructor presides over the creation of new variables, the destructor handles their removal. It isn’t always necessary to define a destructor for a class. A variable of type Orario, which only holds three integers, probably won’t need a destructor, while a variable that makes use of dynamic memory almost certainly will. The reason becomes clearer if you look at it from the compiler’s point of view. To create a variable of type Orario, the compiler has to allocate space for:

3 * sizeof(int);

When the time comes to destroy the variable, all the compiler will do is free the 3 * sizeof(int) bytes following the object’s address; a behavior that’s correct in this case, but that could turn out disastrous with a class like this one:

class Buffer
{
private:
    char* _dati;
    int   _size;
 public:
    Buffer(int size)
    : _size(size) {
        _dati = new char[_size];
    }
    ...
};

In the absence of specific instructions, to destroy a variable of type Buffer, the compiler will free sizeof(char*) + sizeof(int) bytes after its memory address, but in doing so, it will only destroy the _size integer and the _dati char pointer, without freeing the memory area that pointer was pointing to. This, as you know, is a serious mistake, so the class needs a function that instructs the compiler otherwise. Like the constructor, a class’s destructor has no return type, but while a class can have several constructors, its destructor is always unique. It never has formal parameters, and its name is the same as the class it belongs to, preceded by a tilde ~:

class Buffer
{
private:
    char* _dati;
    int   _size;
 public:
    Buffer(int size)
    : _size(size) {
        _dati = new char[_size];
    }
    ~Buffer() {
        delete [] _dati
    }
};

Destructors can be called in two ways:

  • implicitly, by the program, every time an object goes out of scope, or, in the case of objects with global visibility, at the end of the main function;

  • explicitly, from your code, but in that case you’ll have to specify their full name, prefixing the class name and the resolution operator ::, as we’ll see shortly.

Be careful, though: if it’s a pointer that goes out of scope, the class’s destructor is not called automatically, so objects created dynamically with the new operator must always be destroyed by means of the delete operator.

Member functions must be declared inside the class declaration, and can be defined either inside it or outside of it. Defining them inside the class declaration is equivalent to declaring them inline. If instead you define them outside the class declaration, they need to be identified by adding the class name before the function name, followed by the resolution operator:

/** 
 * @file classi-classe-orario.cpp
 * Gestione dei dati membro di una classe.
 */
 
#include <iostream>
#include <iomanip> 

using namespace std;

/** Dichiarazione della classe */
class Orario {
private:
    int _h;
    int _m;
    int _s;
public:
    Orario(int h = 0, int m = 0, int s = 0);
};

/** Definizione del costruttore della classe */
Orario::Orario(int h, int m, int s) 
: _h(h % 24), _m(m % 60), _s(s % 60) 
{
}
    
/** La funzione main non fa parte della classe */
int main()
{    
    Orario ora;

    /** Questo codice darà errore */
    cout << setfill('0') << setw(2) << ora._h << ":" 
         << setfill('0') << setw(2) << ora._m << ":" 
         << setfill('0') << setw(2) << ora._s << endl;
    
    return 0;
}

C++ offers two ways to make a class’s attributes available to functions outside the class:

  • friend classes or functions;
  • interface functions.

The seemingly quicker way to access a class’s private data through an outside function or class is to declare them as friend. This grants the function or class full visibility over the protected data:

class Orario {
private:

    /** Dati membro privati */
    int _h;
    int _m;
    int _s;

public:

    /** Costruttore della classe */
    Orario(int h = 0, int m = 0, int s = 0) ;

    /** Costruttore di copia inline */
    Orario(const Orario& o) {
        _h = o._h;
        _m = o._m;
        _s = o._s;
    }

    /** Funzioni di lettura  */
    int getH() { return _h; }
    int getM() { return _m; }
    int getS() { return _s; }

    /** Funzioni di scrittura */
    int setH(int h) { return _h = (h % 24); }
    int setM(int m) { return _m = (m % 60); }
    int setS(int s) { return _s = (s % 60); }

    /** Dichiarazione di una funzione friend */
    friend int aggiornaMinuti(Orario& o, int m);
    friend class Orologio; 	

};

As I said, this solution is only seemingly quicker, because all the logic for managing the Orario class’s data would have to be replicated both in the aggiornaMinuti function and in the Orologio class. Moreover, if you later needed to make changes to the Orario class, the same changes would have to be carried over into the functions of the friend classes that use it. A safer and more efficient way of managing a class’s private attributes is to define public member functions that allow controlled access to the data you want to protect. In the case of the Orario class, you need six: one for reading and one for writing each of the three member data items:

/** Funzioni di lettura */
int getH() { return _h; }
int getM() { return _m; }
int getS() { return _s; }

/** Funzioni di scrittura */
int setH(int h) { return _h = (h % 24); }
int setM(int m) { return _m = (m % 60); }
int setS(int s) { return _s = (s % 60); }

Obviously, you can name these functions however you like, but using the prefixes get and set, followed by the name of the parameter they operate on, makes it easier for other programmers to use the class. It’s the same reason I add an underscore character before the name of a class’s member data, so they can be told apart from function parameters that share the same name:

return _h = (h % 24);

Using the initialization list, inside the constructor, lets you use parameters with the same name as the class’s member data:

class Punto
{
    int x, y;
public:             
    Punto(int x, int y)
    : x(x), y(y) {
    }
};

but the fact that something is possible doesn’t mean it’s a good choice, as I imagine you’ve learned over the course of your life. You aren’t obligated to declare interface functions as inline either; I did so here because they were extremely simple, but you should generally avoid adding function code inside the declaration of an already complex class, because it makes it harder to read. There are also those who think this is a mistake, because, while it makes life easier for whoever writes the code, it complicates things for whoever reads it, since it mixes what the class does with how it does it.2 I don’t entirely agree with this claim, because sometimes it’s more convenient and quicker to have the function code inside the class declaration, but since it’s a principle that puts the good of the many (those who use the code) ahead of the good of the one (the code’s author), I felt obliged to pass it on to you. Just as we did for the class’s constructor, we could unify the read and write functions, using a default parameter to determine the program’s behavior:

int ore(int h = -1) {
    return (h != -1) ? (_h = (h % 24)) : _h;
}

This syntax is the equivalent of:

int ore(int h = -1) {
    if(h != -1) {
        _h = (h % 24);
    }
    return _h;
}

Although less obvious, it’s more convenient because it lets you keep the code on a single line, and gives you a bit of practice with operators. This kind of function, though, has two drawbacks: it limits the values you can assign to the attribute, and it limits the granularity of the privileges you can grant to whoever uses the class. It limits the number of values you can assign to the attribute, because it excludes the value of the default parameter — which causes no problems in this case, since there’s no such thing as hour -1, but which could be a problem in the case of a string with a null default parameter. It limits the granularity of privileges on the functions, because it forces you to make the write functions for the member data public, which in certain cases might not be wise. I’d therefore recommend always writing two separate interface functions for reading and writing: at the time it’ll seem like a waste of effort, but, unless your program is particularly trivial, sooner or later you’ll realize it was the right choice.

Every variable of a given class holds its own copies of the member data, while a class’s member functions are shared by all of its instances. To let the program know which instance is calling a given method, the compiler adds a hidden parameter, called this, to every function call, pointing to the instance that requested the function. The this parameter, even though it’s never declared, can be used inside the body of member functions to refer to the current instance. For example, the Orario class’s copy constructor (like any other member function of the class) could be rewritten like this:

Orario(const Orario& o) {
    this->_h = o._h;
    this->_m = o._m;
    this->_s = o._s;
}

The only member functions that cannot make use of the this pointer are those declared static. A class can have both static attributes and static member functions. The peculiarity of these elements is that they aren’t tied to a specific variable, but are shared by all instances of the class; this means they behave slightly differently from non-static members:

  • to initialize them inside the declaration, they must be declared inline static,3 otherwise, they must be initialized elsewhere in the program, like any object with global visibility;

  • besides the normal selection operators, you can also access them by referring to the class itself.

Let me try to clarify things a bit with an example:

/** 
 * @file classi-static.cpp
 * Funzioni e dati membro statici di una classe.
 */
 
#include <iostream>

using namespace std;

/** Classe generica con membri statici */
class Contatore {
private:

    /** Definisce il dato statico */
    static int _nIstanze;     

public:
    
    /** Il costruttore incrementa il numero di istanze */
    Contatore() {
        _nIstanze++;
    }
    
    /** Funzione di interfaccia statica */
    static int nIstanze() {  
        return _nIstanze;
    }
};

/** Inizializza il membro statico  */ 
int Contatore::_nIstanze = 0;

int main()
{    
    /** Crea la prima istanza della classe */
    Contatore c1;

    /** Richiama la funzione statica dall'oggetto */
    cout << "Da istanza c1: " << c1.nIstanze() << endl;

    /** Crea altre due istanze della classe */
    Contatore c2, c3;
   
    /** 
     * Richiama la funzione statica dagli oggetti 
     * e direttamente dalla classe
     */
    cout << "Da istanza c2: "<< c2.nIstanze()         << endl;
    cout << "Da istanza c3: "<< c3.nIstanze()         << endl;
    cout << "Dalla classe : "<< Contatore::nIstanze() << endl;
    
    return 0;
}

If you compile and run this code, you’ll get:

> g++ src/cpp/classi-static.cpp -o src/out/esempio
> src/out/esempio                                 
Da istanza c1: 1
Da istanza c2: 3
Da istanza c3: 3
Dalla classe : 3

As you can see, every instance of the class shares the same value for the nIstanze member data, and the interface function, declared static, can be called even without referring to an instance. For this reason, if you use the this pointer inside a static class:

static int nIstanze() {  
    return this->_nIstanze;
}

you get the compile error:

> g++ src/cpp/classi-static.cpp -o src/out/esempio
src/cpp/classi-static.cpp:26:16: error: invalid use of 'this'
 outside of a non-static member function
        return this->_nIstanze;
               ^
1 error generated.

because, if the function were called directly from the class, this wouldn’t point to any object.

The last thing we need to cover is anonymous classes, a special kind of class that, as the name suggests (forgive the pun), has no name, and for that reason can have neither a constructor nor a destructor, and cannot be used either as a function parameter or as a return value. The only way to declare an object with an anonymous class is to add it to the class’s own declaration:

class
{
...
} obj;

This code declares both the class and its one and only instance, the global variable obj, at the same time. When Master Canaro tried to port the Universe to C++, he wanted to use an anonymous class for the Dio (God) variable, because, lacking a constructor and instantiated by its own class, that variable would have been visible throughout the code, but no part of the program could have generated another one:

/** 
 * @file classi-dio.cpp
 * Dichiarazione della classe Dio.
 */

#include <list>

// Classe astratta da utilizzare come base
// per tutti gli elementi del Creato.
class Creatura {
public:
   // Funzione virtuale pura di verifica.
   virtual bool isGood() = 0;
   
};

// Classi derivate per la gestione degli 
// elementi del Creato.
class Mare   : public Creatura {
    bool isGood() { return true; }; 
};
class Uomo   : public Creatura { 
    bool isGood(); 
};
class Donna  : public Creatura {};
class Padre  : public Uomo     {};
class Figlio : public Uomo     {};
class Popolo : public list<Creatura> {};

class  {
private:

    // Funzioni membro per la generazione degli elementi 
    // del Creato, accessibili solo alla classe. 
    Creatura* creaLuce();
    Creatura* creaStelle();
    Creatura* creaAcquaTerra();
    Creatura* creaPiante();
    Creatura* creaSoleLuna();
    Creatura* creaAnimali();

    // Funzioni per la generazione degli umani.
    Uomo&  creaUomo();
    Donna& creaDonna(Uomo& adamo);

    // Interruzione per il settimo giorno.
    void shabat();
    
    // Funzioni di interfaccia con gli umani.
    bool popEden(Uomo& adamp, Donna& eva);
    bool printComandamenti(ostream& tavole);
    bool checkFede(Padre& abramo, Figlio& isacco);
    bool splitAcque(Mare& marRosso, Popolo& ebrei);

protected:

/** 
 * @file classi-pesce.cpp
 * Definizione della classe Pesce.
 */
 
#include <iostream>
#include <iomanip> 

using namespace std;

/**
 *  Dichiarazione della classe: idea
 */
class Pesce {
private:

    /** Dati membro privati */
    string _specie;
    bool   _commestibile;
    
public:
    
    /** Costruttore della classe inline */
    Pesce(const char* specie, bool commestibile) 
    : _specie(specie), _commestibile(commestibile)
    {
    }
    
    /** Costruttore di copia inline */
    Pesce(const Pesce& p) {
        this->_specie       = p._specie;
        this->_commestibile = p._commestibile;
    }

    /** Distruttore inline */
    ~Pesce() { 
    }

    /** Funzioni di interfaccia */
    const char* getSpecie() { return _specie.c_str(); }
    bool isCommestibile()   { return _commestibile;   }

};


/**
 *  Istanza della classe: forma
 */
int main()
{

    cout << pesce.getSpecie() << ": "
         << (pesce.isCommestibile() ? "sì" : "no")
         << endl;
     
    return 0;
}

As we said when talking about the preprocessor, though, the word fish can mean different things depending on who’s using it, so the declaration/idea of the Pesce class will vary depending on what it needs to be used for. For example, knowing whether a fish is edible or not is crucial for a fisherman or an environmentalist, but might not matter to a marine biologist. Conversely, the type of skeleton or the respiratory system, relevant to a biologist, are entirely irrelevant to a fishmonger, who instead will certainly care about the price per kilo, the catch date, and how it’s been preserved. We love what will kill us (if all goes well)

We saw that Love is a force that is at once gravitational and entropic, because it unites individuals, but at the same time drives them to reproduce in different forms. In a sense, we can think of Love as the “constructor” of our instances, because it generates the conditions that push our parents to meet and mate, and above all pushes them to stay together after mating. Monogamy isn’t a constant — in fact, there are four different ways nature handles offspring-rearing, and John Maynard Smith catalogued them by the animal species that adopts each one:4

Duck the male abandons, the female raises
Stickleback the female abandons, the male raises
Midge both parents abandon
Gibbon both parents raise

We, for the most part, behave like gibbons, even though it’s a behavior that mainly benefits females. From a strictly statistical standpoint, a male would have better odds of passing on his DNA by fertilizing more mates. It’s the same reason why, in our field, copies of our data are created in different (and distant) server farms from the original one, so that if the main site catches fire or is struck by a meteorite, the data isn’t lost. Love, on the other hand, drives the webmaster to keep his data in a single server farm, tending to it and protecting it personally, to preserve its informational content. Each of us is the instance of a class that was called into being to carry out a certain task. We are generated, we carry out our task, and then, like the objects of a program, we must be removed, so as not to needlessly occupy the system’s resources. To make sure this happens, we need a destructor that ends our existence at the right moment. Love can serve this purpose too: just as it generated in our parents the interest necessary to bring about our birth, it generates in us an interest that brings about the conditions necessary for our death. This interest can apply to anything — a substance, an activity, a place, one or more people — and must be greater than the interest the individual has in their own existence. You can see clearly that this is a potentially risky aspect of our doctrine, because it could justify forms of self-indulgence toward harmful substances or activities, like drug or alcohol abuse. The death of Robert Capa, who survived the Normandy landings only to be killed, ten years later, by a landmine while photographing soldiers on the Red River delta, is an unambiguous example of this precept, but how can we know whether the death of Jimi Hendrix, Janis Joplin, or Jim Morrison was what was meant to happen, or was instead, on the contrary, the result of a tragically wrong choice? Above all, how can we prevent this precept from becoming an excuse for other wrong choices? On the other hand, things don’t always go as planned, and it can happen that, through a wrong choice or out of fear, an “instance” strays from its path, bringing its own end forward or delaying it. This is because our behavior is preordained, but not compulsory: we’re programmed to be good at some things and less good at others, but we’re free to choose what to do. Many have wondered whether our actions are predestined or whether free will exists; the answer is: yes, but at different times. Going back to the comparison with classes, predestination lies in the declaration, while free will belongs to the instances. The set of your attributes and methods is predefined: you can jump, but you cannot fly; you can swim, but you cannot stay underwater for long; your blood has a well-defined flow, and so does your food. On the other hand, what you decide to do is up to you: you can decide to stick to the limits you’ve been given, or try to exceed them, building yourself artificial wings to fly or breathing apparatus to go underwater. Be careful, though: this won’t make you either a bird or a fish. You can exceed the limits imposed by your class, but you cannot change it. It isn’t hard to reconcile free will with God’s omniscience and omnipotence; the answer is obvious, if you don’t draw a distinction between the artifact and its maker. With all due respect to Einstein, God really does play dice with the Universe; the mistake is thinking that the result of a dice roll comes from random factors. When you roll dice, the final result varies based on two kinds of factors, one easily predictable and one hard to predict. The possible combinations of the dice are easily predictable, once you know the number of faces and the values printed on them. The combination actually produced by a given roll is just as deterministic, but depends on much more complex factors, such as how long the dice were shaken, their position at the moment of the throw, or the angle of impact with the surface. Just as it’s impossible for a roll of the dice in Backgammon to produce a value higher than twelve, it’s impossible for a die, thrown with a certain energy in a certain direction, to follow a trajectory different from the one imposed on it by the Laws of physics. Just because we can’t predict it doesn’t mean it’s random. In the same way, when the primordial One “throws” its energy into the Universe, it is omniscient, because it knows every choice that has previously proven correct, and every one that has instead produced suffering; it is benevolent, because it hopes to achieve the best possible outcome, avoiding repeating mistakes already made; it is omnipotent, because it can potentially produce every possible permutation of existence arising from the interaction of energy and spazioni. And yet, it doesn’t know which of those permutations will actually occur, and it doesn’t even want to know. It hopes the new cycle of existence will be better than the previous ones, but it doesn’t wish for one particular event to happen or another not to happen, because wishing would make it vulnerable to the temptations of the Age-Old Dichotomy. A famous sailor6 once said:

When you’re racing and there’s no wind, don’t go looking for it.
Point the bow in the right direction and wait: the wind will come.

The One behaves in a similar way: faithful to the Taoist precept of Wu Wei,7 it sets the conditions necessary for events that have proven beneficial to happen again, but doesn’t force them to. It writes the system’s analysis, but leaves the programmers to write the code, even though it knows they will certainly make mistakes. It defines rules, but leaves its instances free to break them, because it knows that evolution is always the fruit of a mistake gone right, of something that shouldn’t have happened this way, and yet turned out better for it. Rules shouldn’t be a net that traps and immobilizes, but a net that catches and holds you up, just as the “binding” at the root of the word “religion” shouldn’t be a bond that shackles, but one that supports. The rules that define, and therefore limit, our existence are like a building with a reinforced-concrete structure and drywall partitions. The interior walls can be knocked down or altered, if needed, but the pillars and beams must stay where they are. Similarly, changing the rules can be beneficial, but it must only be allowed to those who know them well because a clumsy carpenter might, through error or stupidity, remove one of the load-bearing pillars, putting the soundness of the whole building at risk. For these reasons, changing the rules cannot be an activity sanctioned by those who promulgated them, even if they recognize its usefulness, but must be an apparently clandestine activity, carried out by expendable elements who can serve as scapegoats if something goes wrong. It was not out of naivety that the Most High granted Iblīs a stay of his punishment,8 and it was no accident that Master Canaro was added to the ranks of the fallen angels, after, a victim of the Age-Old Dichotomy, he tried to break the rules set by the Analyst. Just as the image of a lithographic print or a screen print exists both in the plate and in the copy, each of us has two levels of existence. One is ideal, similar to the class declaration, and defines what our role in existence is: what we can do, what we cannot do, and what we should do. The other level is our actual manifestation, arising from the interaction of the One’s energy with the spazioni. This level is comparable to an instance of a class, which enacts what was only potential in the declaration: the hearts that break, the mess we make, as the song says. Our ideal entity is single and constant, while our physical manifestation is changeable: as you know, the same object can be allocated in different areas of memory across successive runs of a program, just as the same print can be reproduced on different media. In this cycle of existence, Master Canaro and the dog Lele were a human and a dog running through the hills around Lake Bracciano, but in other existences they may be — or may have been — other people and other animals, in other places, or even on other planets. Our ideal level, in fact, doesn’t establish what we must be, but what our contribution to the economy of the Universe must be; the form we take, or the place where we manifest, are entirely incidental. For C’hi++, as for Hinduism,9 Descartes’s phrase “Cogito ergo sum” is meaningless, because what does the thinking is the instance, which is transient. What we really are, our essence, shows itself in what we do instinctively.
