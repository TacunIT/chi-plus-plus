Conditional instructions

Doing something intentional implies understanding that alternatives exist, and choosing one; and these are attributes of artificial intelligence

Conditional instructions are the most important element of code.

Every program has to know how to react correctly as the conditions it’s used under change; to do this, we use so-called conditional instructions, which let you define how the system behaves depending on whether a given condition turns out to be true or false. C++ has two kinds of conditional instruction: if-else sequences and the switch instruction. The general form of if-else instructions is as follows:

if ( <condizione> ) {
    // istruzioni da eseguire se la condizione è vera
} else {
    // istruzioni da eseguire se la condizione è falsa
}

If the conditional expression at the start of the code is true, the program will run the first block of instructions; if not, it will run the second block of instructions.

if ( a > 8 ) {
    cout << "maggiore" << endl;
} else {
    cout << "minore" << endl;
}

If the false condition requires no specific action, the second block of instructions can be omitted:

typedef Importo unsigned long;

Importo raddoppiaStipendio(Importo stipendioCorrente)
{
    if ( stipendioCorrente > 0 ) {
        stipendioCorrente *= 2;
    }    
    return stipendioCorrente;   
}

In the same way, curly braces can be omitted if the block of instructions they enclose consists of a single instruction:

if ( a > 8 )
    cout << "maggiore" << endl;
else
    cout << "minore" << endl;

Personally, I find this form inelegant, and think it makes the code less clear, which invites mistakes. I only use it in error-handling instructions, where the program’s flow breaks off abruptly, because the ungainly look of the instruction highlights the exception, making the code more self-explanatory.

if ( divisore == 0 )
    throw std::runtime_error("errore");

If there are more than two conditions to evaluate, you can chain several conditional instructions using the else if instruction, which lets you define an alternative condition to the first one and associate a block of code with it. Here too, you can close the sequence with an else instruction, defining a block of instructions to run if none of the expected conditions is met.

if ( <prima condizione> ) {
    /*
     * istruzioni da eseguire se
     * la prima condizione è vera
    */
} else if ( <seconda condizione> ) {
    /*
     * istruzioni da eseguire se
     * la prima condizione è vera
     */
} else {
    /*
     * istruzioni da eseguire se nessuna
     * delle due condizioni è vera
     */
}

if-else instructions affect the readability of the code; it’s something a good programmer must always keep in mind. C++ is a formatting-independent language, so the same instruction can be written in many different ways:

if ( <condizione> )
{
    ...
}
else
{
    ...
}

or, equally:

if ( <condizione> ) {
    ...
} else {
    ...
}

or even:

if ( <condizione> ) { ...} else { ... }

If the instructions are few and simple, one form is as good as another (matters of style aside, of course), but if the program’s flow is, as it usually is, more complex, you need to make sure the form of the instruction makes the code easier to write, read, and, if needed, fix. Imagine a piece of code that has to run a check at the start of processing and, depending on the outcome, either run a sequence of instructions or send an error message:

if ( <condizione> ){
    /*
     * righe di codice da
     * eseguire in caso la
     * condizione sia vera
     */
} else {
    /* gestione dell'errore */
}

If the instructions to run when the check succeeds are few and simple, this sequence won’t cause any problems, but if instead the instructions were numerous and complex, while reading the code you might reach the else instruction and no longer remember which condition it was tied to. In these cases, I prefer the form:

if ( <errore> ) {
    /* gestione dell'errore */
} else {
    /*
     * righe di codice da
     * eseguire in caso la
     * condizione sia vera
     */
}

Since error handling will never require more than a few lines of code, you’ll be able to grasp the whole flow of the program at a glance, regardless of how long the second block of instructions is. None of this, of course, is meant as either an invitation or a justification for writing complex instructions. Unless you need to limit function calls to guarantee high execution speed, it’s always better to break the program’s flow down into a series of distinct, specialized functions. You’ll make your program a bit slower (or, rather: a bit less fast), but the code will be much easier to read or modify. Now imagine a piece of code that requires many chained if conditions:

esito = 0;

if ( <condizione 1> ) {
    esito = 1;
} else if ( <condizione 2> ) {
    esito = 2;
} else if ( <condizione 3> ) {
    esito = 3;
} else {
    esito = 9;
}

return esito

This form, however correct and formally impeccable, could turn out to be hard to manage if the conditions to consider were very complex or numerous. The good programmer, in that case, may decide to break the (correct) rule that says not to put return instructions inside the code, and write the sequence like this:

esito = 0;

if ( <condizione 1> ) {
    return 1;
}
if ( <condizione 2> ) {
    return 2;
}
if ( <condizione 3> ) {
    return 3;
}

return 9

I’m not telling you this is the right way to write it, and you can see for yourself that the code is crude and inelegant, but there could be cases where this is the preferable form. For example, for a system that generates code automatically, it’s much simpler to handle isolated if instructions than chained if-else conditions. Think of a stored procedure that has to check the referential integrity of the parameters it receives:

CREATE FUNCTION utente_insert (
  _id_classe INTEGER
, _id_gruppo INTEGER
, _username  VARCHAR(255)
, _cognome   VARCHAR(80)
, _nome      VARCHAR(80)
)
RETURNS INTEGER DETERMINISTIC
BEGIN
    DECLARE _id    INTEGER DEFAULT -1;
    DECLARE _count INTEGER DEFAULT 0;

    SELECT count(*) INTO _count FROM classe WHERE (id = _id_classe);
    IF _count = 0 THEN
        RETURN -2;
    END IF;

    SELECT count(*) INTO _count FROM gruppo WHERE (id = _id_gruppo);
    IF _count = 0 THEN
        RETURN -3;
    END IF;

    IF (_username IS NULL) OR (_username = '') THEN
        RETURN -4;
    END IF;
    
    IF (_cognome IS NULL) OR (_cognome = '') THEN
        RETURN -5;
    END IF;

    IF (_nome IS NULL) OR (_nome = '') THEN
        RETURN -6;
    END IF;

    ...

If you write the code this way, you can add or remove a parameter (and its related checks) without altering the rest of the code, which wouldn’t be the case if you chained the if instructions together. You’ll lose a bit of execution speed, but the code will be much easier to write or modify. Be careful, though: putting simple if instructions in sequence is quite different from building a chain of else-if instructions, because, if you don’t halt processing with a return instruction in case of an error, the program will keep going, checking the following conditions, and the error in the first condition could have knock-on effects on the code that follows:

/** Qui comincia il male.. **/
if ( divisore == 0 ) {
    cout << "Errore: divisione per zero" << endl;
}

/** ..e il peggio lo segue **/
if ( (dividendo / divisore) > 1 ) {
    ...
}

Since there’s no return instruction, the code of the first check won’t stop the function from running, and it will fail when it tries to perform a division by zero.

The switch instruction lets you handle multiple cases, based on the evaluation of an expression:

switch(<espressione>)
{
    case <costante> : istruzioni... [break];
    case <costante> : istruzioni... [break];
    ...
    default: istruzioni...
}

The case and default keywords identify the values handled by the switch instruction. There can (and it’s useful for there to) be more than one case, but the constants associated with each of them must have different values. The default condition, by contrast, must be unique. Execution of the instruction begins at the case whose constant matches the value of the switch expression, and ends at the break keyword. If the expression has a value not covered by any case, the instruction runs the code associated with the default label:

#include <iostream>
#include <cstdlib>

#define POS_NESSUNO -1
#define POS_ERRORE   0
#define POS_MERCURIO 1
#define POS_VENERE   2
#define POS_TERRA    3
#define POS_MARTE    4
#define POS_GIOVE    5
#define POS_SATURNO  6
#define POS_URANO    7
#define POS_NETTUNO  8
#define POS_PLUTONE  9

using namespace std;

int main(int argc, char** argv)
{    
    int pianeta;
    
    /** 
     * Legge i parametri di input.
     * Se ce ne sono, prova a convertire il primo parametro 
     * in un intero. Se il parametro non è un intero, la 
     * funzione atoi torna 0. 
     */
    if(argc > 1) {
        pianeta = atoi(argv[1]);
    } else {
        pianeta = POS_NESSUNO;
    }
    
    /** Gestisce i casi possibili */
    switch( pianeta ) {
        case POS_ERRORE:   cout << "Valore non valido"; 
            break;
        case POS_MERCURIO: cout << "Mercurio";          
            break;
        case POS_VENERE:   cout << "Venere";            
            break;
        case POS_TERRA:    cout << "Terra";             
            break;
        case POS_MARTE:    cout << "Marte";             
            break;
        case POS_GIOVE:    cout << "Giove";             
            break;
        case POS_SATURNO:  cout << "Saturno";           
            break;
        case POS_URANO:    cout << "Urano";             
            break;          
        case POS_NETTUNO:  cout << "Nettuno";           
            break;          
        case POS_PLUTONE:  cout << "Plutone";           
            break;
        default:
            cout << "Inserire un valore da: " 
                 << POS_MERCURIO 
                 << " a " 
                 << POS_PLUTONE;                    
    }
    
    cout << endl;
    
    return 0;
}

Compiling and running this code, you’ll get:

% g++ src/cpp/istruzioni-condizionali-switch.cpp -o src/out/esempio
% src/out/esempio                                                  
Inserire un valore da: 1 a 9
% src/out/esempio 4
Marte
% src/out/esempio terra
Valore non valido

In essence: with a break at the end of each case, the switch instruction is a more elegant (and efficient) form of the if - else if instruction:

if(pianeta == POS_ERRORE) {
   cout << "Valore non valido";
} else if(pianeta == POS_MERCURIO) {
    cout << "Mercurio";
} else if(pianeta == POS_VENERE) {
    cout << "Venere";   
} else if ..

If you removed the breaks at the end of each case, the program’s output would be:

% src/out/esempio 4    
MarteGioveSaturnoUranoNettunoPlutoneInserire un valore da: 1 a 9

which makes no sense in this particular case, but which can be the right choice when two possible cases need to be handled the same way. One last thing: always remember that, to declare variables inside a case, you need to add curly braces; otherwise, you’ll get a compile error:

switch( x ) {
    case 1:
        int y = 9;      /** errore di compilazione */
        cout << x + y;
        break;
    case 2: {
        int y = 2;      /** corretto */
        cout << x + y ;
        break;
    }
    default:
        cout << "default" << endl;
}

When that happens, you have two options: either you make the choice that’s most convenient for you, or you make the choice that seems most beneficial to the greatest number of people for the longest possible time. Human beings are members of a whole, in creation of one essence and soul. If one member is afflicted with pain, other members uneasy will remain.1

Answering the question about the convicted man is harder. The purpose of your actions must always be improvement: whoever makes a mistake must understand that they’ve made an error, and not repeat it in other cycles of existence:

If the sinner truly repents, then it’s right that they be absolved, because, as Attar says:

A hundred Worlds of sin are dissolved by the light of a single repentance.7

But the repentance must be real: the sinner must despise their mistake, and choose to die rather than repeat it. Cutting off the hand of a thief, forcing them to bring food to their mouth with the same hand they use to clean themselves, is a rather drastic method, but an effective one for forcing someone to meditate on the senselessness of their past actions — especially in a place like the desert, where bidets are rarer than in France. The Lex Talionis can work for lesser crimes, because whoever suffers it has time to reflect on their mistakes, but in the case of a murder it’s not only contrary to the duty of benevolence we owe other sentient beings, it could also be counterproductive, because if the condemned doesn’t understand their mistake before dying, their criminal actions might be repeated in other cycles of the Universe. On the other hand, we also have a duty of benevolence toward the other inmates and prison guards, so we cannot allow the condemned to kill them. The ideal solution would be to put them in a position where they can’t harm others, and then let them meditate on their mistakes, but if that isn’t possible, how should we behave? If an individual acknowledges their own mistakes and learns from them, they contribute to the improvement of the Universe; if instead they repeat the same criminal act over and over, they clearly don’t understand, or don’t want to understand, their mistake. If they don’t understand, they aren’t sentient, and therefore cannot contribute to the improvement of the Universe. If they’re of no use to that improvement, we can treat them the same way as Nansen’s cat:

Nansen found the monks of the eastern and western halls arguing over a cat. He seized the cat and said: “If you can say a word of Zen, I will spare the cat; if not, I will kill it.” No one could answer, and Nansen cut the cat in two.8

Be careful, though: the death of the condemned, should it ever prove necessary, must not be seen as a revenge to be celebrated, but as an event as painful as it is unavoidable, to be mourned like an amputation. A Buddhist teacher, having heard that Master Canaro, in his writings, claimed there was no way to escape the cycle of rebirth, went to see him and, determined to prove him wrong, challenged him to a Dharma Combat to settle their respective positions. Master Canaro replied that he didn’t know what a Dharma Combat was; at which the Buddhist teacher explained that it was a dialectical exchange, meant to demonstrate one’s knowledge of the doctrine. Master Canaro then nodded and said: “All right, but before we begin, tell me whether you want to win or lose this exchange, so that I can oblige you.” Hearing those words, the Buddhist teacher realized his intentions were not pure: he didn’t want that challenge to arrive at the truth, only for the pleasure of winning. So he bowed, thanked Master Canaro for making him see that weakness in himself, and from that moment on became his disciple.
