Functions

Call me: IsNull

Functions are C++’s main distinguishing feature.

The purpose of functions is to make code easier to manage. It’s possible to write a program that only has the main function, but this only makes sense for very simple programs, like some of the examples we’ve seen so far. The flow of a non-trivial program, as Stroustrup would say, will always be split across several functions, because this makes the code easier to read, to fix, and to modify.

/**
 * @file data-types-size.cpp
 * Shows the size of C++'s main data types.
 */

#include <iostream>
#include <iomanip>
using namespace std;
int main()
{
    int w = 12;

    cout << setw(w) << "bool: "   << sizeof(bool)   << endl;
    cout << setw(w) << "char: "   << sizeof(char)   << endl;
    cout << setw(w) << "int: "    << sizeof(int)    << endl;
    cout << setw(w) << "float: "  << sizeof(float)  << endl;
    cout << setw(w) << "double: " << sizeof(double) << endl;

    return 0;
}

This is the code of an example we saw in the lesson on data types. If we decided to change how the program displays data to the user, we’d have to touch every single line of the program, which would be tedious and could introduce bugs. By contrast, if we isolate the display function, any changes or fixes only need to be applied in one place. The code of the following example is a perfectible illustration of this approach:

/**
 * @file data-types-limits.cpp
 * Shows the possible values for C++'s main data types.
 */
#include <iostream>
#include <iomanip>
#include <limits>
using namespace std;
/**
 * Template function that shows the size and the
 * minimum and maximum values for a data type.
 */
template<typename T>
void limits()
{
    cout << sizeof(T) << " bytes,"
         << setw(4)   << "da:"
         << setw(21)  << numeric_limits<T>::min()
         << setw(4)   << "a:"
         << setw(21)  << numeric_limits<T>::max()
         << endl;
}
int main()
{
    /** Variation of the size and values of the int type */
    cout << setw(16) << "int: ";
    limits<int>();
    cout << setw(16) << "unsigned int: ";
    limits<unsigned int>();
    cout << setw(16) << "short int: ";
    limits<short int>();
    cout << setw(16) << "unsigned short: ";
    limits<unsigned short>();
    cout << setw(16) << "long int: ";
    limits<long int>();
    cout << setw(16) << "unsigned long: ";
    limits<unsigned long>();

    return 0;
}

Whenever there are instructions in your code that repeat, consider the possibility of isolating them into a function. For example, in the main function of the program above there are repeated output functions, which makes the code harder to read, more laborious to modify, and increases the chance of mistakes, because you’re the one defining the string with the data type’s name, and you have to be careful to write the right value every time:

cout << setw(16) << "int: ";
...
cout << setw(16) << "unsigned int: ";
...
cout << setw(16) << "short int: ";
...
cout << setw(16) << "unsigned short: ";
...
cout << setw(16) << "long int: ";
...
cout << setw(16) << "unsigned long: ";

In this particular case, it was an unavoidable choice, because, being one of the first examples I gave you, I didn’t want to complicate things too much for you, but this isn’t the right way to write code. It’s wrong (or, at the very least, risky) to mix data processing with interface functions, especially in a case like this, where part of the output is already delegated to a specific function. Better to let the dimension function handle all the output, while the main function limits itself to defining the flow of processing, calling the output function for the data types we’re interested in:

/**
 * @file functions-limits-modified.cpp
 * Example on C++ limits with output in a single function.
 */

#include <iostream>
#include <iomanip>
#include <limits>
#include <typeinfo>
#include <cxxabi.h>

using namespace std;

/**
 * Displays the name, the size, and the minimum and
 * maximum values for the current data type.
 */
template<typename T>
void dimension()
{
    cout << setw(14)
         << abi::__cxa_demangle(typeid(T).name(), NULL, NULL, NULL)
         << ": "
         << sizeof(T) << " bytes,"
         << setw(6)   << "from:"
         << setw(21)  << numeric_limits<T>::min()
         << setw(4)   << "to:"
         << setw(21)  << numeric_limits<T>::max()
         << endl;
}

/**
 * The new code of the main function: more compact
 * and easier to read or modify.
 */
int main(int argc, char** argv)
{
    dimension<int>();
    dimension<short int>();
    dimension<unsigned short>();
    dimension<long int>();
    dimension<unsigned long>();
    return 0;
}

The instruction:

abi::__cxa_demangle(typeid(T).name(), NULL, NULL, NULL);

displays the data type of the current variable. For now, just accept it on faith; I’ll explain how the :: operator behaves when we talk about classes. The output of this code is identical to that of the previous example; if anything, it’s a bit better, because I reduced the width of the first column from 16 to 14 characters, and did so by modifying a single setw(14) instruction, instead of the six in the previous code:

% g++ src/cpp/functions-limits-modified.cpp -o src/out/example

unsigned short: 2 bytes, from:                    0 to:                65535

Functions are one of C++’s data types, and like all data types, they can be declared and defined. The declaration of a function establishes its return type and the parameters it requires as input:

float removeTax(long salary, float rate);

The definition of a function establishes its behavior:

float removeTax(long salary, float rate)
{
    float value;
    value = salary / ((100 + rate) / 100);
    return value;
}

As I told you when we talked about programming languages, generating an executable file happens in two stages: first the compiler converts the C++ code into an object file, then the linker turns the object files (there could be more than one) into a single executable. For this process to work, a function’s declaration must be present in every piece of code that uses it, so the compiler can check that it’s being used correctly; the definition, by contrast, must appear only once.

/**
 * @file functions-salary-main.cpp
 * Main function of the program with separate object files.
 */
#include <iostream>

/**
 * Declaration of the doubleSalary function.
 * I put it here for simplicity: usually, declarations
 * live in a separate include file.
 */
float doubleSalary(float salary);

/**
 * The main function calls the doubleSalary function,
 * which is defined in another source file.
 */
int main(int argc, char** argv)
{
    std::cout << doubleSalary(1500) << std::endl;
    return 0;
}


/**
 * @file functions-salary-func.cpp
 * Function of the program with separate object files.
 */

/**
 * Definition of the doubleSalary function.
 */
float doubleSalary(float salary)
{
    return salary * 2;
}

If we compile the two files above separately, adding the -c parameter, which tells the compiler to generate only the object file without invoking the linker to produce an executable:

% g++ -c src/cpp/functions-salary-main.cpp -o src/out/main.o
% g++ -c src/cpp/functions-salary-func.cpp -o src/out/func.o

and then generate an executable file using the two object files:

% g++ -o src/out/example src/out/main.o src/out/func.o

we’ll get three files, two object files and the executable file example, which will give the expected result:

% ls -1 src/out                                               
example
func.o
main.o

% src/out/example                                             
3000

But since I’m lazy, in the examples I’ll give you, I’ll always use a single command for compiling and linking the programs:

g++ src/cpp/functions-salary-func.cpp  \                                                       
    src/cpp/functions-salary-main.cpp  \
    -o src/out/example
% src/out/example

Earlier I told you to consider the possibility of isolating repeated instructions into a function, because creating a function isn’t always the right choice. Software, like many human endeavors, is the product of a series of trade-offs, and you’ll have to make different architectural choices depending on the kind of program you need to build. Good software, besides working correctly, should be fast, easy to modify, and require few system resources. Sometimes, though, you have to sacrifice one of these positive traits to boost another. For example, if you had to write extremely fast software, it might be better to have duplicated instructions rather than function calls, because calling a function inevitably causes some slowdown. But if you duplicate instructions, you not only increase the size of the program, you also make it harder to read and modify, which is bad. In these cases, some increase in code size is unavoidable, but inline functions let you keep the code readable and modifiable:

/**
 * @file functions-inline-1.cpp
 * Example of code with repeated instructions.
 */

#include <iostream>
#include <iomanip>
#include <fstream>

#define S_FILENAME  "src/out/example.txt"

using namespace std;

int main(int argc, char** argv)
{
    /**
     * Creates a variable of the ofstream class
     * to handle the output file  */
    ofstream doc;

    /** Opens the output file */
    doc.open (S_FILENAME);
    cout << "opened the file " << endl;

    /** Writes to the output file */
    doc << "Document text.\n";
    cout << "wrote to the file" << endl;

    /** Closes the output file */
    doc.close();
    cout << "closed the file" << endl;

    return 0;
}

Compiling and running the code above, you get:

% ./src/out/example

wrote to the file
closed the file

The output instructions in the example differ only in the text to display, and could easily be isolated into a standalone function.

/**
 * @file functions-inline-2.cpp
 * Example of code with a single output function.
 */

#include <iostream>
#include <fstream>

#define S_FILENAME  "src/out/example.txt"

using namespace std;

/** Single output function */
void log(const char* message)
{
    /** Displays the message on screen */
    cout << message << endl;
}

int main(int argc, char** argv)
{
    /** Creates a variable to handle the output file */
    ofstream doc;

    /** Opens the output file */
    doc.open (S_FILENAME);
    log("opened the file");

    /** Writes to the output file */
    doc << "Document text.\n";
    log("wrote to the file");

    /** Closes the output file */
    doc.close();
    log("closed the file");

    return 0;
}

The output of this program is identical to that of the previous example, but if you add the inline keyword before the return type of the log function:

inline void log(const char* message)
{
    /** Displays the message on screen */
    cout << message << endl;
}

and recompile the program, you’ll see that the size of the executable file has grown, because the compiler replaced every function call with a copy of the function’s own code. Even though the executable’s size has grown, the code is still easy to read and modify:

/**
 * @file functions-inline-3.cpp
 * Example of an inline function.
 */

#include <iostream>
#include <iomanip>
#include <fstream>

#define LOG_DEBUG    1
#define LOG_WARNING  2
#define LOG_ERROR    3

#define S_DEBUG      "debug"
#define S_WARNING    "warning"
#define S_ERROR      "error"

#define ERR_NO_FILE_NAME  -1
#define ERR_NO_FILE_OPEN  -2

using namespace std;

/** Inline output function */
inline void log(const char* message, int level)
{
    /** Sets the width of the first field */
    cerr << setw(8);

    /** Writes the message level */
    switch(level) {
        case LOG_DEBUG:   cerr << S_DEBUG  ; break;
        case LOG_WARNING: cerr << S_WARNING; break;
        default:          cerr << S_ERROR  ; break;
    }

    /** Writes the message text */
    cerr << ": " << message << endl;
}

int main(int argc, char** argv)
{
    /** Creates a variable to handle the output file */
    ofstream doc;

    /** Pointer for the output file name */
    const char* filename = NULL;

    /** If the output file name is missing, error */
    if(argc < 2) {
        log("specify the file path", LOG_ERROR);
        return ERR_NO_FILE_NAME;
    }

    /** Reads the output file name */
    filename = argv[1];

    /** Tries to open the output file */
    doc.open (argv[1]);

    /** If there was an error, reports it and exits */
    if(!doc.is_open()) {
        log("cannot open the file", LOG_ERROR);
        return ERR_NO_FILE_OPEN;
    }

    /** OK, the file is ready */
    log("opened the file", LOG_WARNING);

    /** Writes to the output file */
    doc << "Document text.\n";
    log("wrote to the file", LOG_DEBUG);

    /** Closes the output file */
    doc.close();
    log("closed the file", LOG_WARNING);

    return 0;
}

The existence of an inline function is somewhat reminiscent of a passage from the Samyutta Nikaya:

There is no distinction between a being, its function, and the time of its appearance. Beings appear out of non-existence, exist for an instant, and then cease to exist. Their existence, activity, and action are one and the same thing. Past and future are mere names.2

In general, though, the lifecycle of a function involves three distinct phases: declaration, definition, and its subsequent use:

/** Declaration */
float doubleSalary(float salary);

/** Definition */
float doubleSalary(float salary)
{
    return salary * 2;
}

/** Call */
float amount = doubleSalary(1500);

Only the main function is exempt from this rule, because it requires no declaration and is never called by other functions. The variable float salary, which appears in parentheses both in the declaration and in the definition of the function, is called the function’s parameter. The value 1500, assigned to the parameter in the function call, is called the argument. These variables are also called formal parameters, in the case of the declaration and the definition, and actual parameters, in the case of the call. When you call a function, you can pass it arguments in three ways: by value, by reference, by pointer:

/**
 * @file functions-arguments.cpp
 * Example of how function arguments are handled.
 */

#include <iostream>
#include <iomanip>
#include <fstream>

using namespace std;

/**
*   Call by value: whatever happens here to the
*   variables a and b has no effect on the x and y
*   variables of the calling function.
*/
void byValue(int a, int b)
{
    int tmp = a;
    a = b;
    b = tmp;
    return;
}

/**
*   Call by reference: whatever happens here to the
*   variables a and b is as if it were done to the
*   x and y variables of the calling function.
*/
void byReference(int &a, int &b)
{
    int tmp = a;
    a = b;
    b = tmp;
    return;
}

/**
*   Call by pointer: here too, whatever happens to
*   the variables a and b also happens to the x and y
*   variables of the calling function, but the code
*   is less clear.
*/
void byPointer(int *a, int *b)
{
    int tmp = *a;
    *a = *b;
    *b = tmp;
    return;
}

int main(int argc, char** argv)
{
    int  x = 11; int  y = 22;
    cout << setw(20) << "Initial value: x=" << x << ", y=" << y << endl;

    int* p = &x;
    int& r = x;

    cout << "p:" << p << ",r=" << r << endl;

    /**
    *   Passing by value: the argument of the function
    *   call is the value of the variables x and y:
    */
    byValue(x,y);
    cout << setw(20) << "byValue: x=" << x << ", y=" << y << endl;

    /**
    *   Passing by reference: the arguments of the function
    *   call are the variables x and y, but the function
    *   will use aliases of them, not their value:
    */
    byReference(x,y);
    cout << setw(20) << "byReference: x=" << x << ", y=" << y << endl;

    /**
    *   Passing by pointer: the argument of the call
    *   is explicitly the address of the variables x and y:
    */
    byPointer(&x,&y);
    cout << setw(20) << "byPointer: x=" << x << ", y=" << y << endl;

    return 0;
}

The output of this program is:

% g++ src/cpp/functions-arguments.cpp -o src/out/example
% src/out/example                                      
   Initial value: x=11, y=22
         byValue: x=11, y=22
    byReference: x=22, y=11
      byPointer: x=11, y=22

Passing a reference to a variable as a function argument is equivalent to passing the variable itself. This can be useful for functions that require large objects as input, or that need to directly modify the value of the variables passed to them as parameters, but it should be avoided in any other case, because it lets the program modify the value of a variable in a sneaky way, which can be very hard to spot when debugging. Another thing worth knowing, about function parameters, is that they can have default values:

/**
 * @file _man/src/functions-default-parameters.cpp
 * Example of a function with default parameters.
 */

#include <iostream>

using namespace std;

/** Function with default parameters  */
void printValue(int value = 0)
{
    /** Prints the parameter's value on screen */
    cout << "Value: " << value << endl;
}

int main(int argc, char** argv)
{
    /** Calls the function with an argument */
    printValue(123);

    /** Calls the function without arguments */
    printValue();

    return 0;
}

The output of this program will be:

% g++ src/cpp/functions-default-parameters.cpp -o src/out/example
% src/out/example
Value: 123
Value: 0

A parameter’s default value must be specified in the function’s declaration. Remember, though, that when you assign a default value to one of a function’s parameters, you must do the same for every parameter that follows it, if there are any:

void func1(float f, void * ptr = NULL);        // OK
void func2(double d, int b = 2, char c = 'c'); // OK
void func3(int i = 3, int n );                 // ERROR!

Default values are used when one or more of a function’s parameters is often assigned the same value. For example, if the log function we saw in the previous example were mostly called with the same value for the level parameter, we could assign it as a default:

inline void log(const char* message, int level = LOG_WARNING);

making the code faster and easier to write:

log("opened the file");

doc << "Document text.\n";
log("wrote to the file", LOG_DEBUG);

doc.close();
log("closed the file");

In one of our earliest conversations, I told you that the Good Programmer, while writing code, always asks whether there’s a more efficient way of doing what he’s doing. Today you’ve seen proof of that: we improved the first example in the section on inline functions by unifying message handling into a single function, and then improved it again by making that function inline. There’s a problem, though: the program’s output:

% ./src/out/example                                   
opened the file
wrote to the file
closed the file

only works if there’s a single file to handle, as in our case. But if there were two (or more) input or output files, it would be useful to know which file the message refers to. To solve the problem, we could increase the number of formal parameters of the log function:

void log(int level, const char* message, const char* file);

but the new parameter would be useless for calls like:

log(LOG_ERROR, "specify the file path");

Alternatively, we could build the message inside the calling function:

string s1 = "closed the file: ";
string s2 = filename;
string s3 = s1 + s2;
log(LOG_WARNING, s3.c_str());

but this would make the code heavier, more complicated, and slower; it would also shift back into the calling functions part of the output functionality we had happily isolated into the log function. The correct solution to this kind of problem is functions with variable parameters. A typical example of this kind of function is the C language’s printf function, whose first argument is used to determine the type and number of the arguments that follow it:

/** Declaration, in stdio.h */
int printf(const char *format, ...) ;

/** Usage */
printf("string: %s; integer: %d", "error code", -1);

To indicate variable arguments, in a function’s declaration, three dots are used after the fixed parameters:

void log(int level, int n_params, ...);

In the function’s definition, to handle the parameters, you need to create a reading loop using three macros defined in the stdarg.h file:

void va_start(va_list ap, lastfix);
type va_arg(va_list ap, type);
void va_end(va_list ap);

va_start initializes the ap variable list so it can receive the variable arguments; the lastfix parameter specifies which is the function’s last fixed parameter.

va_arg returns the next parameter in the ap list; the type parameter indicates the parameter’s data type.

va_end ends the processing of the parameters, and must always be called before the function returns.

/**
 * @file functions-variables.cpp
 * Example of a function with variable parameters.
 */

#include <iostream>
#include <iomanip>
#include <fstream>
#include <cstdarg>

#define LOG_DEBUG    1
#define LOG_WARNING  2
#define LOG_ERROR    3

#define S_DEBUG      "debug"
#define S_WARNING    "warning"
#define S_ERROR      "error"

#define ERR_NO_FILE_NAME  -1
#define ERR_NO_FILE_OPEN  -2

#define S_ERR_NO_FILE_NAME  "specify the file path"
#define S_ERR_NO_FILE_OPEN  "cannot open the file:"

using namespace std;

/** Output function with variable parameters */
void log(int level, int n_params, ...)
{

    /** Sets the width of the first field */
    cerr << setw(8);

    /** Writes the message level */
    switch(level) {
        case LOG_DEBUG:   cerr << S_DEBUG  ; break;
        case LOG_WARNING: cerr << S_WARNING; break;
        default:          cerr << S_ERROR  ; break;
    }

    /** Writes the separator character */
    cerr << " | ";

    /** Declares the variable for the parameter list */
    va_list param_list;

    /**
     * Initializes the parameter list and treats
     * every argument after n_params as variable.
     */
    va_start(param_list, n_params);

    /** Reads all the parameters in the list and prints them on screen */
    for(int p = 1; p <= n_params; p++) {
        cerr << va_arg(param_list, char*) ;
    }

    /** Closes the parameter list */
    va_end(param_list);

    /** Writes the message text */
    cerr << endl;

}

int main(int argc, char** argv)
{
    /** Creates a variable to handle the output file */
    ofstream doc;

    /** Pointer for the output file name */
    const char* filename = NULL;

    /** If the output file name is missing, error */
    if(argc < 2) {
        log(LOG_ERROR, 1, S_ERR_NO_FILE_NAME);
        return ERR_NO_FILE_NAME;
    }

    /** Reads the output file name */
    filename = argv[1];

    /** Tries to open the output file */
    doc.open (argv[1]);

    /** If there was an error, reports it and exits */
    if(!doc.is_open()) {
        log(LOG_ERROR, 2, S_ERR_NO_FILE_OPEN, filename);
        return ERR_NO_FILE_OPEN;
    }

    /** OK, the file is ready */
    log(LOG_WARNING, 2, "opened the file: ", filename);

    /** Writes to the output file */
    doc << "Document text.\n";
    log(LOG_DEBUG, 2, "wrote to the file: ", filename);

    /** Closes the output file */
    doc.close();
    log(LOG_WARNING, 2,  "closed the file: ", filename);

    return 0;
}

At this point, all that’s left is to tell you about functions that call themselves, that is, recursive functions:

/**
 * @file /src/functions-recursive.cpp
 * Example of a recursive function.
 */

#include <iostream>
#include <iomanip>

using namespace std;

/** Declaration of the recursive function */
long factorial(int n);

/** Main function before the definition */
int main(int argc, char** argv)
{
    int n = 9;
    cout << n << "! = " << factorial(n) << endl;
    return 0;
}

/** Definition of the recursive function */
long factorial(int n)
{
    long fact = 1;
    if(n > 1) {
        fact = n * factorial(n-1);
    }
    return fact;
}

Be careful, though: a function that calls itself, if not written well, can generate an in[de]finite recursion, like two mirrors facing each other. The perspective space inside the mirrors is inexhaustible, but a computer’s memory, however large, has its limits, and too many recursions can cause a stack overflow.3 For this reason, you should resort to recursion only when it’s truly necessary, preferring, whenever possible, iterative instructions and always applying mechanisms to control recursion depth, to prevent a stack overload. Another kind of recursion to be wary of is the kind sometimes used to lend credibility to a belief with no real foundation, by citing another, similar belief.

In a novel about the life of the Japanese swordsman Miyamoto Musashi, there’s a line that follows logically from the Roman proverb I told you about yesterday:

One must not interfere with the workings of the Universe, but first one must understand what the workings of the Universe are4

The problem with novels is that they often tell you what to do, but almost never explain how, or why. Be wary of teachers who only know how to teach, because everything they tell you they learned from books. A good teacher must excel at something, whether it’s fencing, archery, carpentry, or motorcycle maintenance. He may not be the best, but he must be among the best; only then will you know his ideas are sound. By contrast, teachers who have never left a school have never had to put their ideas to the test of reality. They teach you what they think is right, but what’s right or true within a school isn’t always equally true or right in the real world. It’s easy to be a saint inside a monastery, among people who share your culture and your own principles; the hard part is remaining a saint outside of it. The Buddha Shakyamuni preached Universal Loving-kindness because he lived in India twenty-six hundred years ago, but if he’d been stuck in city traffic after a day’s work, his ideas might have looked a bit more like Nietzsche’s. Fortunately, Musashi — the real one, the Samurai — was always very clear both about what’s right to do and about how to achieve it. In the Book of Five Rings, he gave his disciples nine rules for life:

  • Do not think dishonestly.
  • The Way is in training.
  • Become acquainted with every art.
  • Know the Ways of all professions.
  • Distinguish between gain and loss in worldly matters.
  • Develop intuitive judgment and understanding for everything.
  • Perceive those things which cannot be seen.
  • Pay attention even to trifles.
  • Do not do anything useless.

In the Dokkodo, written a week before his death, he was even more specific:

  • Accept everything just the way it is.
  • Do not seek pleasure for its own sake.
  • Do not, under any circumstances, depend on a partial feeling.
  • Think lightly of yourself and deeply of the world.
  • Be detached from desire your whole life long.
  • Do not regret what you have done.
  • Never be jealous.
  • Never let yourself be saddened by a separation.
  • Resentment and complaint are appropriate neither for oneself nor others.
  • Do not let yourself be guided by the feeling of lust or love.
  • In all things have no preferences.
  • Be indifferent to where you live.
  • Do not pursue the taste of good food.
  • Do not hold on to possessions you no longer need.
  • Do not act following customary beliefs.
  • Do not collect weapons or practice with weapons beyond what is useful.
  • Do not fear death.
  • Do not seek to possess either goods or fiefs for your old age.
  • Respect Buddha and the Gods without counting on their help.
  • You may abandon your own body but you must preserve your honour.
  • Never stray from the Way.

These rules, which let Musashi reach the venerable age of sixty,5 after emerging victorious from just as many duels against all the finest swordsmen of his time, can help you understand how the Universe works, and what to do so as not to disturb it. The practice of the arts, whether intellectual or martial, and the knowledge of trades, combined with attention to everything around you, will broaden your store of experience and let you tell the pros and cons of any situation apart. This will free you from the demon of envy, because you’ll learn that every condition, even the seemingly idyllic ones, has its downsides. The next step will be to free yourself from desire and attachment to things: just as the log function needs to know which are its fixed parameters and which are variable, you’ll have to learn to tell your desires apart from your needs, in order to escape the influence of the Age-Old Dichotomy. The Way, with a capital “W”, is a lot like a way with a lowercase “w”. Along a way, you come across road signs, which show you which direction to go or what speed to keep, and billboards, which point out nearby attractions and invite you to stray from your path to go visit them. In the same way, along the Way, you’ll find needs, which will guide you toward your destination, and desires, which will dis-tract you from your path and at-tract you toward alternative destinations. If you act according to your needs, you’ll always know you’re moving in the right direction, even when you’re forced to slow down or take roads you don’t enjoy. If instead you act according to your desires, you’ll wander here and there, like “a donkey without reins”,6 and by the time you finally get back on the right road, you might no longer have time to reach your destination.

I told you about Musashi’s rules not because they’re the only ones available, but because they’re intensely personal. They draw on the principles of other disciplines, like Buddhism, Bushido, or Tao, but they’re something else, as Eddie Cochran would say. You’ll have to do the same: learn everything you can, from good and bad teachers alike, and then define your own rules for life, which may end up the same as, similar to, or entirely different from the ones you were taught. The Way, like Truth, is a model, whom each of us portrays from our own point of view, trying to guess at the body hidden behind the folds of the drapery. Portraying her the same way someone else has would be wrong, because your point of view isn’t the same as theirs, but looking at her from several points of view can help you understand her shape better. Musashi says to pay attention even to insignificant things, Nan-in and Tenno develop their Every-Minute Zen, Wittgenstein, in his Diaries, writes:

Only one thing is necessary: to be able to observe everything that happens to you. Concentrate! God help me!

It’s clear they’re all painting the same picture, even if each does so in his own style. This is why Musashi prescribes knowing the Way of other trades: because there’s always something to learn from someone who does their job well. This, for example, is the parallel he draws between the strategist and the carpenter:

To build a house, materials must be carefully selected. For the outer pillars, straight and knot-free timber should be chosen, while for the inner ones, straight timber with minor defects may be used. For thresholds, lintels, fittings, and sliding doors, the finest-looking wood should be used, even if not too sturdy, and so on. For structural parts, visual appearance matters less than robustness. Lower-grade timber with many knots is used instead for scaffolding and, in the end, is burned.7

What’s true for the carpenter, with a shift in perspective, is also true for the strategist, and could be true for you too, if you ever find yourself managing a project or a team. Even as a programmer, though, you’ll have to keep constantly up to date on techniques and programming languages, and you’ll need to understand how those who work on databases or systems do their job. You’ll need to be able to spot malfunctions even when they have no visible effect on the graphical interface, and you’ll only manage that if you pay attention to even the smallest detail. The names change, because four centuries have passed, but the problems remain the same, and so do the solutions. After all, art, science, philosophy, and religion are all attempts to answer the same questions about our existence. Since the object of study is the same, it’s only natural that, sometimes, the answers resemble each other, in the same way that the algorithm for calculating a number’s factorial is more or less the same, regardless of the programming language.
